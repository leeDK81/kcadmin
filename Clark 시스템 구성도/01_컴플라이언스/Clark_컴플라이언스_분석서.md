## 2. 마이데이터 사업자 의무

### 2-1. 본인신용정보관리업 허가 요건 (신용정보법 제22조의9)

마이데이터 사업자(본인신용정보관리업자)로서 준수해야 할 핵심 의무:

**허가 유지 의무**
- 자기자본 요건 유지 (5억 원 이상)
- 물적 설비 (시스템, 보안) 기준 유지
- 개인신용정보 관리 기준 준수
- 금융위원회 보고/감독 수인 의무

**데이터 전송 요구권 행사 절차 (신용정보법 제33조의2)**
- 정보주체(사용자)의 명시적 전송 요구 필요
- 전송 요구 시 포함해야 할 사항:
  - 전송을 요구하는 개인신용정보의 종류
  - 전송을 요청받는 기관 (수신기관 = Clark)
  - 전송 주기 및 기간
- 전송 요구 철회 방법 안내 의무

**수집·이용 제한 원칙 (신용정보법 제32조)**
- 수집 목적 범위 내 이용 원칙
- 목적 외 이용 시 별도 동의 필요
- 제3자 제공 시: 정보주체 동의 + 제공 목적·항목·기간 명시

### 2-2. 마이데이터 API 연동 요건

**정보 제공자 API 연동 (금융보안원 표준)**
- 마이데이터 표준 API 사용 의무 (자체 스크래핑 방식 금지 — 단, 일부 예외 적용 중)
- API 상호 인증: mTLS (Mutual TLS) 적용 필수
- API 접근 토큰: OAuth 2.0 기반 사용자 동의 토큰
- 데이터 전송 시 암호화 필수 (TLS 1.2 이상, 권고: TLS 1.3)

**데이터 보관 제한**
- 원칙: 정보주체 동의 기간 동안만 보관
- 동의 철회 시 즉시 삭제 (단, 법정 보존 기간 예외)
- 마이데이터 원본과 가공 데이터 분리 관리

### 2-3. ⚠️ 핵심 주의 사항 — 마이데이터 + 외부 콘텐츠 결합

```
[위반 리스크]
마이데이터로 수집한 보험/금융 정보를
KC카드 등 외부 콘텐츠와 결합하여 AI 분석/추천에 활용하는 경우
→ 신용정보법 제32조 목적 외 이용에 해당할 수 있음
→ 정보주체 동의 없는 제3자 정보 결합으로 위반 가능성
```

**대응 방향**: 마이데이터 데이터와 KC카드 서비스는 논리적·물리적으로 분리 운영 (5항 상세 분석)

## 3. 개인정보보호법 적용 기준

### 3-1. 수집 항목별 처리 근거

| 개인정보 항목 | 수집 목적 | 법적 근거 | 보존 기간 |
|-------------|---------|---------|---------|
| 이름, 연락처, 이메일 | 회원 식별, 서비스 제공 | 동의 (법 제15조 제1항 제1호) | 탈퇴 후 30일 |
| 소셜 로그인 식별자 | 소셜 로그인 연동 | 동의 | 탈퇴 시 즉시 삭제 |
| 본인인증 결과 (CI/DI) | 중복 가입 방지, 본인 확인 | 동의 + 법령 의무 | 5년 |
| 보험 계약 정보 | 마이데이터 서비스 제공 | 동의 (전송 요구권) | 동의 기간 + 3개월 |
| 건강 데이터 | 건강 정보 서비스 제공 | **별도 동의 (민감정보)** | 서비스 이용 기간 |
| AI 질문/답변 이력 | 서비스 개선, 이력 조회 | 동의 | 1년 |
| 접속 로그, 행동 데이터 | 보안, 서비스 개선 | 정당한 이익 또는 동의 | 6개월 |

### 3-2. 민감정보 별도 동의 요건 (법 제23조)

건강 데이터는 **민감정보**로 분류되어 별도 동의 절차가 필수:
- 일반 개인정보 동의와 분리된 독립 동의 화면
- 동의 거부 시에도 기본 서비스(마이데이터) 이용 가능 명시
- 수집 항목, 이용 목적, 보유 기간, 제3자 제공 여부 명확히 기재

### 3-3. 개인정보처리방침 필수 기재 사항
- 처리 항목, 목적, 보유 기간
- 제3자 제공 현황 (프롬에이지, 쿠콘 — 위탁 또는 제공으로 분류)
- 개인정보 처리 위탁 현황
- 정보주체 권리 행사 방법 (열람, 정정, 삭제, 처리 정지)
- 개인정보보호책임자 (CPO) 연락처
- 자동화된 의사결정 (AI) 활용 사실 고지

## 4. 건강 데이터 규제

### 4-1. 프롬에이지 / 쿠콘 API 연동 시 규제 검토

**건강 데이터 종류별 규제 적용**
| 데이터 유형 | 규제 | 특이 사항 |
|-----------|------|---------|
| 건강검진 결과 | 개인정보보호법 (민감정보) | 별도 동의 필수 |
| 병원 진료 기록 | 의료법 + 개인정보보호법 | 정보주체 직접 동의만 허용 |
| 처방 정보 | 의료법 | 의료기관 외 제공 제한 |
| 웨어러블 건강 데이터 | 개인정보보호법 (민감정보) | 별도 동의 필수 |
| 보험사 건강 관련 지급 이력 | 신용정보법 + 보험업법 | 마이데이터 범위 내 |

**⚠️ 의료법 주의 사항**
- 프롬에이지/쿠콘이 병원 진료 기록을 제공하는 경우, 해당 API의 법적 적합성 확인 필요
- 진료 기록 원본 제공이 아닌, 정보주체가 직접 내보내기한 데이터인지 확인
- Clark이 건강 데이터를 기반으로 의학적 분석/진단성 정보를 제공하는 경우 의료법 위반 가능

**권고**: 프롬에이지/쿠콘과의 계약 시 데이터 출처 및 법적 적합성 보증 조항 포함

### 4-2. 건강 데이터 AI 활용 제한
- 건강 데이터 기반 AI 분석 결과를 진단/치료 목적으로 제공 금지 (의료법)
- 보험료 산정 등 불이익적 활용 금지 조항 필요
- 정보 제공 목적: "건강 관리 참고 정보" 수준으로 한정, 면책 고지 필수

## 5. AI LLM + KC카드 서비스 리스크 분석

### 5-1. 서비스 구조별 리스크 분류

#### 시나리오 A: 마이데이터 + KC카드 + AI 통합 (Same Server, 원래 구상)
```
[위험도: 매우 높음 — 현재 구조로 서비스 불가]

사용자 마이데이터 (보험/금융) + KC카드 콘텐츠
        ↓ 결합
      AI LLM 분석/답변
        ↓
    사용자에게 제공

문제점:
① 신용정보법 위반 — 마이데이터를 목적 외(KC카드 추천)로 활용
② 금융소비자보호법 — AI 답변이 금융 상품 권유에 해당할 경우 미등록 투자/보험 권유
③ 보험업법 — AI가 보험 상품 비교/추천 시 보험대리점 자격 필요
④ 혁신금융위 검수 — AI + 마이데이터 결합 심사 기준 미충족 가능성
```

#### 시나리오 B: 서비스 물리적 분리 (권고 구조)
```
[위험도: 중간 — 구조 분리 후 각 영역 별도 검토]

[마이데이터 서비스 영역]          [AI/KC카드 서비스 영역]
  보험 마이데이터 조회    ←분리→   KC카드 기반 AI 답변
  건강 데이터 연동                  (마이데이터 미활용)
  데이터 전송 요구권 행사

→ 마이데이터 영역: 기존 마이데이터 사업자 규제 준수
→ AI 영역: KC카드 콘텐츠(FAQ/상품 정보) 기반 정보 제공 서비스 (별도 규제 검토)
```

#### 시나리오 C: AI는 일반 정보 제공만 (최소 리스크 구조)
```
[위험도: 낮음]

KC카드 콘텐츠 기반 FAQ/안내 서비스 수준
→ 투자/보험 권유 아님 명시
→ 개인화 분석 없음 (사용자별 맞춤 추천 제외)
→ 일반 정보 제공 서비스로 분류 가능
```

### 5-2. AI 답변의 금융업 해당 여부 판단 기준

**보험업법 — 보험모집 해당 여부**
| AI 행위 | 해당 여부 |
|--------|---------|
| KC카드 보험 상품 일반 정보 제공 | 비해당 (광고 수준) |
| 사용자 상황 기반 특정 보험 추천 | **해당 가능** (보험모집 행위) |
| 마이데이터 기반 보험 필요성 분석 | **해당 가능** (보험설계 행위) |
| 보험료 비교 안내 | 비교공시 기준 준수 시 비해당 |

**금융소비자보호법 — 금융상품 권유 해당 여부**
| AI 행위 | 해당 여부 |
|--------|---------|
| KC카드 상품 일반 설명 | 비해당 |
| 특정 카드 상품 개인 추천 | **해당 가능** (투자권유 유사) |
| 수익률/혜택 비교 후 추천 | **해당 가능** |

### 5-3. ⭐ 권고: AI 서비스 설계 원칙

```
① 일반 정보 제공 범위로 한정
   KC카드 FAQ, 상품 설명, 혜택 안내 수준
   → 개인 맞춤 추천 기능 Phase 1 제외

② 마이데이터와 AI 서비스 데이터 격리
   AI 서버는 마이데이터 DB 접근 불가
   → 사용자가 직접 입력한 정보만 AI 활용 가능

③ 면책 고지 UI 의무화
   "이 답변은 일반적인 정보 제공 목적이며, 금융 조언·보험 권유가 아닙니다.
    금융 의사결정 전 전문가와 상담하세요."

④ AI 답변 이력 관리
   감사(Audit) 목적 로그 보존
   문제 발생 시 답변 내용 증빙 가능한 구조
```

## 6. 혁신금융서비스 지정 제도 검토

### 6-1. 혁신금융서비스란?
금융위원회가 신기술 기반 금융 서비스에 대해 기존 규제 적용을 일시 면제하고 테스트 기간을 부여하는 제도 (금융혁신지원특별법)

**지정 기간**: 최초 2년, 최대 4년 연장 가능  
**지정 후 의무**: 테스트베드 운영 결과 보고, 금융위 모니터링 수인

### 6-2. Clark 서비스의 혁신금융 신청 검토 필요 영역

| 서비스 | 기존 규제 충돌 | 혁신금융 신청 검토 |
|-------|------------|----------------|
| 마이데이터 보험 연동 | 없음 (기존 마이데이터 허가 범위) | 불필요 |
| 건강 데이터 + 보험 연계 분석 | 개인정보 민감정보 결합 이슈 | **검토 필요** |
| AI + KC카드 + 마이데이터 결합 | 신용정보법 목적 외 이용 충돌 | **신청 대상 가능성 높음** |
| AI 보험 추천/비교 | 보험업법 모집 행위 충돌 | **신청 필요** |

### 6-3. 혁신금융서비스 신청 절차
```
① 서비스 기획 확정
② 충돌 법규 특정 (법무 법인 검토 권고)
③ 금융위원회 혁신금융 신청서 제출
   - 서비스 개요, 혁신성, 규제 특례 필요 사유, 소비자 보호 방안
④ 혁신금융심사위원회 심의 (통상 2~3개월)
⑤ 지정 후 테스트베드 운영 (조건부 서비스 가능)
```

**⚠️ 주의**: 혁신금융 신청 전까지는 해당 기능 서비스 불가 — Phase 분리 필수

## 7. 전자금융거래법 / 금융소비자보호법

### 7-1. 전자금융거래법
- Clark이 직접 자금 이동/결제를 처리하지 않는 경우 전자금융업 등록 불필요
- 본인인증 서비스 활용 시: 공인된 본인확인기관 서비스 이용 (PASS, 카카오 등)
- 앱 보안 요건: 금융보안원 전자금융 보안 기준 준수 권고

### 7-2. 금융소비자보호법 적용
- **적합성 원칙**: 금융상품 권유 시 소비자의 재무 상황, 위험 성향 파악 의무
- **적정성 원칙**: 소비자가 요청해도 부적합한 상품 판매 제한
- **설명 의무**: 금융상품 주요 내용, 위험, 수수료 설명
- **불공정 영업 금지**: 소비자에 불리한 계약 체결 강요 금지

**Clark AI 서비스 적용**:  
AI가 금융상품 추천 기능을 탑재할 경우 → 금융소비자보호법상 금융상품판매업자 등록 필요 가능성  
→ Phase 1에서는 정보 제공 수준으로 한정 권고

## 10. 동의 체계 설계 가이드

### 10-1. 동의 레이어 구조

```
[레이어 1] 서비스 이용 기본 동의 (필수)
  ├── 서비스 이용약관 동의
  ├── 개인정보 처리방침 동의
  └── 만 14세 이상 확인

[레이어 2] 마이데이터 서비스 동의 (마이데이터 이용 시 필수)
  ├── 마이데이터 서비스 이용약관
  ├── 개인신용정보 전송 요구 동의
  │   ├── 제공 기관별 선택
  │   ├── 전송 항목별 선택
  │   └── 전송 주기/기간 설정
  └── 마이데이터 정보 분석 이용 동의

[레이어 3] 건강 데이터 동의 (민감정보, 별도 화면 필수)
  ├── 건강 데이터 수집 동의 (민감정보)
  ├── 건강 데이터 분석 이용 동의
  └── 건강 데이터 제3자 제공 동의 (해당 시)

[레이어 4] 부가 서비스 동의 (선택)
  ├── 마케팅 정보 수신 동의
  ├── 개인화 서비스 동의
  └── 서비스 개선을 위한 데이터 활용 동의
```

### 10-2. 동의 화면 필수 원칙
- 필수/선택 항목 명확히 구분 (색상, 레이블로 시각적 분리)
- 일괄 동의 버튼 제공 시 개별 선택도 병기 (법원 판례 기준)
- 전문 보기 링크 제공 (핵심 요약 + 전문 분리)
- 동의 일시, 동의 버전 이력 DB 보존 (분쟁 시 증빙)
- 동의 철회 경로: 앱 내 설정 메뉴에서 즉시 가능

## 11. 최종 확정 구조 (2026-05-14 오전 세션)

### 11-1. 확정된 회사 구조

| 구분 | 회사 | 역할 | 법적 자격 |
|------|------|------|---------|
| A사 (모회사) | Clark App 운영사 | 마이데이터 수집, 앱 서비스 | 보험 마이데이터 사업자 (본인신용정보관리업 허가) |
| B사 (자회사) | KC카드 LLM 운영사 | 보험 보장 분석, AI 답변 | GA사 (법인보험대리점) |

### 11-2. 구조 검토 결과

**B사가 GA사라는 사실로 해소된 리스크**
- ✅ 보험업법 위반 → GA사는 보험모집 행위 허가 주체 → LLM 보험 분석 적법
- ✅ 금융소비자보호법 → GA사는 기존 금융상품판매업 준수 주체
- ✅ A사 혁신금융 검수에서 AI 서비스 제외 가능 (A는 데이터 전달만 담당)

**남은 조건 (충족 필요)**
- A→B 전송: "업무 위탁" 구조로 포지셔닝 권고 (위탁 계약 + 개인정보처리방침 공개)
- 건강 데이터(민감정보): B사 명시한 별도 동의 화면 필수
- CI 미전송: 세션 토큰 방식으로 개인 식별 최소화
- B사의 데이터 목적 외 이용 금지 조항 위탁 계약서 포함 필수

### 11-3. 수정된 리스크 매트릭스 (A/B 분리 구조 적용)

| 기능 | 이전 판정 | 수정 판정 | 변경 사유 |
|------|---------|---------|---------|
| KC카드 기반 보험 분석 AI | ⛔ Phase 1 불가 | ✅ 가능 (B사 운영) | B=GA사, 보험업법상 적법 주체 |
