## 1. 핵심 쟁점 요약

B사(GA사)의 KC카드 기반 AI 엔진이 해외 LLM API를 호출할 경우
프롬프트에 포함된 보험 내역·건강 정보가 해외 서버로 전송되어
**개인정보 국외 이전** 규제 적용 대상이 될 수 있음.

```
[문제 구조]
B사 서버(서울) → 해외 LLM API → 미국/유럽 서버
                                  ↑
                        보험내역 + 건강정보 전송
                        = 개인정보 국외 이전 해당 가능
```

## 2. LLM 인프라 방안 3가지

### 방안 1 — AWS Bedrock (서울 리전) ★ 최우선 권고

**개념**
> AWS 서울 리전(한국 데이터센터) 안에서 운영되는 AI LLM 호출 서비스.  
> B사가 기존에 AWS를 사용하고 있으면 별도 인프라 구축 없이 활성화만으로 사용 가능.

**구조**
```
B사 서버(AWS 서울) → AWS Bedrock(AWS 서울) → 결과 반환
                           ↑
                   데이터가 한국 서버 내에서만 이동
                   = 국외 이전 없음
```

**사용 가능 모델**
| 모델 | 제공사 | 특징 |
|------|------|------|
| Claude 3.5 Sonnet | Anthropic | 현재 최고 성능, 한국어 우수 |
| Llama 3.1 | Meta | 오픈소스 기반, 파인튜닝 가능 |
| Titan | Amazon | AWS 네이티브, 안정적 |

**장점**
- B사 기존 AWS 인프라와 바로 통합
- AWS 콘솔에서 활성화만으로 즉시 사용 가능
- 엔터프라이즈 옵션: 입력 데이터 AI 학습 사용 완전 차단
- 국외 이전 없음 → 규제 가장 깨끗한 구조
- API 호출 과금 → 초기 구축 비용 없음

**단점**
- 서울 리전에서 지원되는 모델 목록 지속 확인 필요
- Bedrock 서비스 별도 활성화 및 모델 접근 신청 필요

---

### 방안 2 — Azure OpenAI (한국 중부 리전)

**개념**
> Microsoft Azure 한국 중부(Korea Central) 데이터센터에서  
> OpenAI의 GPT-4o 등 모델을 직접 호출하는 서비스.  
> 데이터가 한국 서버에서 처리되어 국외 이전 없음.

**구조**
```
B사 서버(AWS 서울) → Azure OpenAI(Azure 한국) → 결과 반환
                              ↑
                    데이터가 한국 서버 내에서만 이동
                    = 국외 이전 없음
```

**장점**
- GPT-4o 등 OpenAI 최신 모델 사용 가능
- 엔터프라이즈 계약: 데이터 학습 사용 금지 보장
- 금융권 컴플라이언스 대응 레퍼런스 다수

**단점**
- B사가 AWS 외에 Azure 추가 계약 필요
- AWS ↔ Azure 크로스 클라우드 통신 구성 추가 필요
- 방안 1 대비 아키텍처 복잡도 증가

---

### 방안 3 — 네이버 HyperCLOVA X (국내)

**개념**
> 네이버가 운영하는 국내 LLM 서비스.  
> 한국 서버에서 완전 운영, 금융·보험 도메인 대응 경험 보유.

**구조**
```
B사 서버(AWS 서울) → HyperCLOVA X API(네이버 국내) → 결과 반환
                              ↑
                    국내 서버, 국외 이전 없음
                    국내 금융 규제 대응 경험
```

**장점**
- 규제 관점에서 가장 안전한 구조
- 한국어 특화 모델
- 네이버 금융 서비스 레퍼런스 보유

**단점**
- 글로벌 LLM 대비 성능 검토 필요
- KC카드 보험 도메인 특화 파인튜닝 별도 작업 필요

## 3. 최종 권고

```
1순위: AWS Bedrock (서울 리전)
  → B사 기존 AWS 인프라 활용, 즉시 적용 가능
  → 국외 이전 없음, 규제 안전
  → 방안 중 가장 현실적

2순위: HyperCLOVA X
  → Bedrock 서울 리전 모델이 요구 성능 미달 시 대안

3순위: Azure OpenAI (한국 중부)
  → OpenAI 모델이 반드시 필요한 경우에만 검토
```

## 4. KC카드 기반 AI 엔진 최종 구조

```
[KC카드 데이터]
보험 상품 DB, 설계 데이터, 분석 기준
        ↓ 벡터화 (임베딩)
[벡터 DB — AWS 서울]
pgvector (RDS) 또는 OpenSearch Serverless
        ↓
[사용자 분석 데이터 입력]
나이·성별·보험가입내역·건강정보 (A사에서 수신)
        ↓
[AWS Bedrock — 서울 리전]
KC카드 벡터 DB 검색 + 분석 데이터 조합
→ 보험 보장 분석 결과 생성 (RAG 방식)
        ↓
[결과 반환 — A사 Clark으로 전송]
텍스트 + 이미지 (원본 분석 데이터 즉시 파기)
```

## 5. CI 관리 정책 (확정)

### 정책 원칙
- CI(연계정보) 단방향 암호화 값은 **A사 Clark 내부에서만 관리**
- **B사 포함 어떠한 외부에도 전송하지 않음**

### 목적 및 보관 방식
| 항목 | 내용 |
|------|------|
| 보관 목적 | 탈퇴 후 재가입 시 동일인 확인 (중복 가입 방지) |
| 보관 주체 | A사 회원 DB (외부 미공유) |
| 암호화 방식 | 단방향 해시 (원본 복원 불가) |
| 보관 기간 | 탈퇴 후 1년 (개인정보처리방침 명시) |
| 외부 전송 | 절대 금지 (B사·GA사·원수사 모두 해당) |
| 활용 범위 | 재가입 시 동일인 여부 확인만 — 그 외 용도 사용 금지 |

### 데이터 흐름에서의 CI 처리
```
사용자 본인인증 완료
        ↓
CI 단방향 해시 생성 → A사 회원 DB 저장
        ↓
B사로 전송되는 데이터: 나이·성별·보험내역·건강정보
CI는 포함하지 않음 → 세션 토큰으로 대체
        ↓
탈퇴 시: CI 해시값만 보관 (1년), 나머지 개인정보 삭제
재가입 시: 새 CI 해시와 기존 해시 대조 → 동일인 확인
```

### 개인정보처리방침 명시 사항
```
"서비스 탈퇴 후에도 재가입 방지 및 부정 이용 방지 목적으로
 본인인증 연계정보(CI)의 단방향 암호화 값을 탈퇴일로부터
 1년간 보관합니다. 해당 기간 경과 후 즉시 파기합니다."
```
