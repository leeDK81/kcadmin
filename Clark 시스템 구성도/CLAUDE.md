# Clark 프로젝트 — CLAUDE.md

## 프로젝트 개요

**Clark**는 마이데이터 기반 보험 통합 관리 앱이다.

### 확정된 서비스 범위

| 기능 | 운영 주체 |
|------|---------|
| 소셜 간편 로그인 / 본인인증 | 아이지넷 (NCP) |
| 보험 마이데이터 조회 | 아이지넷 (NCP) |
| 건강 데이터 연동 (프롬에이지/쿠콘) | 아이지넷 (NCP) |
| KC카드 기반 보장 현황 분석 AI | **더파트너스** (AWS Bedrock 서울 리전) |
| 상담 신청 시 원수사/GA에 고객 DB 전달 | 아이지넷 (NCP) |

> **서비스 범위 외 (해당 없음)**: 상품 추천, 보험료 예측, 보험 비교

### 회사 구조

- **아이지넷** (모회사): 보험 마이데이터 사업자, Clark App 운영, NCP 인프라
- **더파트너스** (자회사): GA사 (법인보험대리점), KC카드 기반 AI 분석 엔진, AWS Bedrock 서울 리전

### 핵심 구조 원칙

- 아이지넷 → 더파트너스 데이터 전달 = **업무 위탁** (제3자 제공 아님)
- CI는 아이지넷 내부 DB에서만 단방향 해시로 보관, 외부 전송 절대 금지
- 더파트너스 AI 분석 후 임시 데이터 즉시 파기
- 통신: mTLS + AES-256 암호화

1. **컴플라이언스 우선**: 아키텍처·기능 설계 전 반드시 규제 요건 확인

## 에이전트 구성 및 역할

### 1. 총괄 PO (`agents/po-orchestrator.md`)
- 전체 방향 결정, 에이전트 간 조율, 최종 의사결정
- **호출 시점**: 방향 설정, 에이전트 간 충돌 조율

### 2. 컴플라이언스 (`agents/compliance.md`)
- 마이데이터 규제, 개인정보보호법, 보험업법, 금소법
- 법적 판정 완료 상태 유지 — 새 기능 추가 시만 재검토
- **호출 시점**: 새 기능 추가, 외부 API 연동 시

### 3. 인프라/아키텍처 (`agents/infra-architecture.md`)
- NCP (아이지넷) / AWS Bedrock 서울 리전 (더파트너스) 구조 설계
- **호출 시점**: 서버 구조, 인프라 설계

### 4. API 연동 (`agents/api-integration.md`)
- 마이데이터 표준 API, 프롬에이지/쿠콘 API, 아이지넷↔더파트너스 인터페이스
- **호출 시점**: 외부 서비스 연동 설계, API 명세 작성

### 5. 보안 (`agents/security.md`)
- OAuth 2.0, JWT, 소셜 로그인, mTLS, AES-256, CI 관리
- **호출 시점**: 인증 체계, 암호화, 보안 검토

### 6. 데이터 설계 (`agents/data-architecture.md`)
- DB 스키마, 개인정보 처리 항목, 동의 이력, 파기 정책
- **호출 시점**: DB 설계, 데이터 흐름 설계

### 7. UX/화면 설계 (`agents/ux-design.md`)
- 동의 화면, 보험 진단 결과 화면, 상담 신청 흐름
- **호출 시점**: 화면 설계, 사용자 흐름 정의
