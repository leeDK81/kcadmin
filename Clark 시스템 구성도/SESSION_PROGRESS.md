**리서치 결론** (insane-research 5개 에이전트, 법령원문·판례·최근제재사례 기반 — 상세는 `01_컴플라이언스/AI엔진_인프라배치_실질통제_리스크분석서.md` 참조):

- **핵심 리스크(높음)**: 이 변경은 "국내설치의무"·"망분리규정"·"재위탁공개의무" 같은 개별조문 위반은 아닐 가능성이 높음(리스크 낮음~중간). 진짜 문제는 **"업무위탁이 유효하려면 수탁자가 실질적으로 그 업무를 수행해야 한다"는 대법원 판례 법리(2016도13263 — "실질적 관리·감독", "실질적 이용주체가 누구인지")**. RAG DB·KC Admin이 아이지넷 인프라에 있으면 "더파트너스가 실질적으로 분석을 수행한다"는 2026-05-14 판정의 대전제 자체가 흔들림.
- **뒷받침 최신 사례**: 카카오페이-알리페이-애플 사건(2026.6.11 법원 확정) — 계약서 문언 아닌 실질적 이익귀속·통제 주체 기준으로 판단, PIPC 승소. 보람상조 사건(2026.5) — 계약상 적법한 위탁도 실질적 감독 부실이면 제재(과징금 5.4억+과태료). 두 사례 모두 최근 규제·사법당국이 형식보다 실질을 본다는 경향을 보여줌.
- **의외의 발견**: 전자금융감독규정 제14조의2(국내설치의무)는 조번호 정정 필요(제8항→제7항, 최신 2025-4호 기준) + **마이데이터 전용사업자(아이지넷)에게는 애초에 적용 안 될 가능성 높음**(전자금융업 겸영시에만 적용). ★아이지넷의 전자금융업 겸영 여부 확인 필요.

- **권고 대안**: (1) RAG DB(검색·질의 처리)만이라도 AWS(더파트너스)에 유지 — 2026-07-14 확정 원칙 재확인. (2) 불가피시 AWS 쪽이 실질적 처리행위(프롬프트구성·판단로직)를 수행하도록 설계+감사기록 확보. (3) PIPC 유권해석/금감원 비조치의견서 사전확보 권고.
- **산출물**: `01_컴플라이언스/AI엔진_인프라배치_실질통제_리스크분석서.md` (법무·보안 담당자용 신규 독립문서, 공식출처 인용). 리서치 원본: `RESEARCH/AI엔진_인프라배치_실질통제리스크_20260715/`

**미해결 질문**: "국내 리전만 사용한다"는 사용자 결정이 기술적으로 실현 가능한가?

2026-07-14 세션에서 사용자가 AI 인프라를 (기존 확정안이던 AWS Bedrock 1순위 대신) **Azure OpenAI**로 반복 언급했고, 마지막에 "리전은 국내 리전만 사용할 꺼야"라고 확정 지시함. 그런데 직전 리서치(`01_컴플라이언스/Clark_AI엔진_리전및법무리서치.html` 1번 섹션)에서:
- **AWS Bedrock 서울리전**: 최신 Claude 모델은 "Global" 크로스리전 라우팅만 지원, In-Region(순수 단일리전) 옵션 없음 — 단, 구세대 모델(Claude 3.x)이나 Amazon 자체 모델(Titan/Nova)이 In-Region으로 되는지는 **미확인**.
- **Azure OpenAI 한국중부**: 진짜 단일리전("Standard/Regional") 배포는 **임베딩 모델 1개만 지원** — GPT-4o 등 챗/생성형 모델은 전혀 지원 안 되는 것으로 1차 조사됨(재확인 필요).

즉 "국내 리전만" + "Azure OpenAI로 챗 응답"이 **동시에 성립 안 할 수 있음**. 이걸 검증하는 리서치 에이전트를 막 실행하려던 시점에 사용자가 작업을 중단시키고 "내일 이어서" 요청 — **재확인 리서치가 완료되지 않은 채로 세션 종료됨.**

**다음 세션에서 바로 할 일**:

2. **KC Admin 거버넌스 원칙 확정**:
   - **저작(작성/수정)**: 아이지넷 직원·외부 리서처 등 누가 해도 무방(보험모집행위 아님, 카피라이팅에 가까움)
   - **발행 승인(게시)**: 반드시 더파트너스(GA사) 소속 보험전문 담당자 전속 — 승인 없이 RAG DB 미반영
   - **최종 산출물(LLM 실제 출력문) 검증도 GA사 필수** — 카드 승인만으로는 부족(LLM 재구성 과정의 왜곡 가능성 때문)
   - **인프라(서버 운영)는 아이지넷 위탁 가능**하나 콘텐츠 승인 권한과는 분리(RBAC)
   - **저작 원본 DB(NCP 가능) ≠ RAG 서빙 DB(반드시 AWS/더파트너스)** — 이 둘을 같은 것으로 혼동하지 말 것
3. **"프록시/중계" 표현 관련 경고**: KC AI엔진을 "프록시 서버"로만 정의하면(RAG/프롬프트 구성 로직 없이 순수 패스스루), "GA사가 보험분석을 실질통제한다"는 기존 법적판정(2026-05-14)의 전제가 흔들릴 수 있음 — KC AI엔진은 반드시 RAG 검색·프롬프트 구성·결과 가공을 자체 로직으로 수행하는 능동적 처리자여야 함. **이 부분이 실제로 순수 패스스루로 구현되는 건지는 미확인 — 다음 세션에서 재확인 필요.**
4. **컴플라이언스 법무리서치 완료**: insane-research 스킬로 8개 쟁점(신용정보법 목적외이용, 가명정보/익명정보, 위탁·재위탁, 보험업법 모집행위, 혁신금융서비스 선례, 전자금융감독규정 국내설치의무, 망분리, AWS/Azure 데이터정책) 1차 공식출처 기반 조사 완료. 산출물: `01_컴플라이언스/Clark_AI엔진_리전및법무리서치.html`.
   - **신규 발견 — 조문번호 오류**: 기존 `Clark_컴플라이언스_분석서.md`의 "신용정보법 제32조 목적외이용"은 오기 — 정확히는 **제33조**(제32조는 제3자 제공 동의). 정정 필요.
