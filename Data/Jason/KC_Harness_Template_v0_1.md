# 문서 작업 하네스 템플릿 v0.1

**출처:** KC 기획서 v1.5 → v1.6.1 검증 루프 (2026.04)
**용도:** 기술 스펙·BRD·PRD 등 복잡한 인지 작업의 품질 보증 프로세스
**작성자:** Jason (아이지넷 플랫폼사업본부)
**기반 경험:** 5라운드 검증 사이클에서 축적된 실천 지식

## 사용 안내

본 템플릿은 두 가지 방식으로 활용 가능하다:

1. **세션 시작 프롬프트** — 새 대화 시작 시 첨부하여 하네스 루프의 맥락 사전 로딩
2. **라운드별 호출 도구** — 특정 라운드(검증·메타 검증·hotfix)에서 해당 Part만 발췌하여 에이전트에 전달

구체 시나리오는 본 문서 말미의 "사용 시나리오" 섹션 참조.

## PART A — 역할 분리 구조

### A-1. 기본 역할 2종

**작성 에이전트 (Writer)**
- 문서의 콘텐츠·구조·설계를 담당
- 편집 권한 보유 (파일 수정, 섹션 신설, 표 변경 등)
- 자가 검증은 수행하되 자가 보고 수치를 최종 권위로 삼지 않음

**검증 에이전트 (Verifier)**
- 문서의 내부 논리·cross-reference·일관성을 독립적으로 검증
- 편집 권한 없음. 지적만 함
- 작성 에이전트의 자가 보고를 받지 않고 독립 측정으로 확인

### A-2. 선택 역할 1종 (필요 시 활성화)

**메타 검증자 (Meta-Verifier)**
- 검증 에이전트의 검증 품질을 한 단계 위에서 검토
- 특히 작성 에이전트가 제시한 옵션 선택·결정 구조에 묵시적 가정이 깔렸는지 확인
- 활성화 조건:
  - 작성 에이전트가 옵션 (a)/(b) 등을 제시했을 때
  - 중요한 정책 결정 국면일 때
  - 작성 에이전트가 자가 진단 후 방어 장치를 스스로 설계했을 때

### A-3. 역할 분리의 최소 조건

같은 LLM을 쓰더라도 **다른 컨텍스트 창에서 호출**해야 한다. 같은 대화 내에서 "이제 검증해봐"로 전환하면 작성 중 형성된 framing이 검증에도 유지되어 분리 효과가 반감된다.

## PART B — 5단계 재발 방지 장치 (Writer용)

### B-1. Verify-then-write (참조 박기 전 대상 확인)

**언제 적용:** cross-reference, 섹션 참조, 약어 사용, 외부 문서 인용을 **처음 쓸 때**

**절차:**
1. 참조하려는 대상(§16 D-XX, §3-8-7 항목 N 등)을 view tool로 직접 조회
2. 조회 결과의 내용이 쓰려는 참조의 의도와 일치하는지 확인
3. 일치하면 참조 작성, 불일치하면 대안 참조 재탐색

**방어 대상 오류 유형:** 창작형 cross-reference (agent가 "§16 D-09가 당연히 X를 가리킬 것"으로 상상해서 박는 참조)

### B-2. 수정 후 즉시 grep 재검증

**언제 적용:** 각 편집 직후

**절차:**
1. 수정한 항목의 "옛 표현"을 grep으로 검색
2. 잔여 건수가 0 또는 예상치와 일치하는지 확인
3. 불일치 시 잔여 위치 즉시 추가 수정

**방어 대상 오류 유형:** "일괄 처리했다"는 자가 보고의 부정확성

### B-3. 수정 후 cross-ref 대상 view 재확인

**언제 적용:** B-1과 별개로, 수정 **완료 시점**에

**절차:**
1. 쓴 참조의 대상을 다시 view로 조회
2. 대상 내용이 참조 문맥과 여전히 일치하는지 재확인
3. 특히 대상 자체가 다른 편집으로 수정됐을 가능성 점검

**방어 대상 오류 유형:** 편집 중 대상이 다른 편집의 영향으로 바뀌어 사후 참조가 깨지는 경우

### B-4. 최종 Cross-reference Batch Sweep

**언제 적용:** 모든 hotfix 완료 후, 최종 보고 **직전**

**절차:**
```bash
# 모든 참조 유형을 한 번에 추출
grep -oE "§[0-9]+(-[0-9]+){0,3}" doc.md | sort | uniq -c
grep -oE "D-[0-9]+" doc.md | sort | uniq -c
grep -oE "T0[1-9]" doc.md | sort | uniq -c

# 각 참조 ID가 실제 정의 테이블에 있는지 대조
# 참조 수 vs 정의 수 mismatch 검증
```

**방어 대상 오류 유형:** 개별 편집 시 누적된 leak. B-1~B-3이 local 검증이라면 B-4는 global 검증

### B-5. 측정값 보고 강제

**언제 적용:** 최종 보고 작성 시

**금지 표현:**
- "N건 처리 완료"
- "모두 수정함"
- "일괄 반영"

**필수 형식 (항목별):**
```
[#N 항목명]
  - 수정 전 line X: "옛 표현"
  - 수정 후 line Y: "새 표현"
  - grep 검증: "옛 표현" 잔여 Z건 (기대 0)
  - cross-ref 검증: 대상 view 결과 일치/불일치
```

**방어 대상 오류 유형:** 자가 인지 편향 (자기 작업을 과대평가). 측정값으로 고정되면 주관적 "완료" 판단이 불가능

## PART C — 검증 에이전트용 체크리스트

### C-1. Leak 유형별 탐지 패턴

| Leak 유형 | 탐지 방법 | 우선순위 |
|---|---|---|
| 잘못된 cross-reference | 참조 대상을 실제 view하여 내용 대조 | Critical |
| 존재하지 않는 참조 (unresolved) | 참조 ID를 실재 정의 테이블과 대조 | Critical |
| 정책 선언 vs 본문 불일치 | 선언문의 주장과 본문 실태를 별도 카운트로 대조 | Critical |
| 명칭·개수 불일치 | 같은 개념의 다중 표기를 sort/uniq 병렬 비교 | High |
| 이전 버전 안전장치 누락 | 이전 버전 원문 내용과 현 버전 본문 비교 | High |
| 논리 순환·역설 | 정의 간 선후 관계 분석 | High |
| 예시의 의미 부조화 | 예시의 입력·출력 의미 관계 검증 | Medium |
| 시간 추정 낙관 | 작업 범위의 hidden dependency 추정 | Medium |

### C-2. 메타 검증 패턴 (Meta-Verifier 활성화 시)

**MV-1. False binary 의심 기본값**
- 작성 에이전트가 (a)/(b) 옵션을 제시하면 기본값으로 *"(c) 또는 (d)가 있을 수 있다"* 를 의심
- 확인 질문: *"(a)와 (b)는 같은 결인가? 대상들의 성격이 동일한가?"*

**MV-2. 묵시적 가정 노출**
- 작성 에이전트의 결정 이면에 깔린 전제를 표면화
- 확인 질문: *"이 결정은 어떤 가정 위에서만 유효한가? 그 가정이 흔들리면?"*

**MV-3. 자가 진단 함의 확장**
- 작성 에이전트가 자기 실수 원인을 진단했을 때, 그 진단이 시사하는 **잔여 위험**까지 평가
- 확인 질문: *"이 원인이 다른 곳에도 작용했다면, 아직 발견 안 된 유사 leak은?"*

**MV-4. 시간 추정 현실 보정**
- 작성 에이전트가 제시한 작업 시간을 독립적으로 재추정
- 보정 기준: (a) 숨은 dependency (b) 재검증 사이클 가능성 (c) 외부 응답 대기 시간

**MV-5. 옵션 추천의 이면 가정 확인**
- 작성 에이전트가 특정 옵션을 추천할 때, 그 추천이 묵시적으로 깔고 있는 상위 결정을 확인

### C-3. 독립 측정 원칙

**절대 하지 말 것:**
- 작성 에이전트의 "N건 처리" 보고를 그대로 받아 검증 결과에 인용
- "agent가 그렇게 말했으니 맞을 것"이라는 가정
- 대상 내용을 view하지 않고 참조의 정확성 평가

**반드시 할 것:**
- 모든 수치는 grep·wc·sort/uniq로 독립 측정
- 모든 cross-reference는 참조 대상을 별도로 view
- 보고서에 `agent 보고` vs `독립 검증` 두 컬럼으로 병렬 제시

## PART D — 작업 방법론 패턴

### D-1. Scope Discipline

**원칙:** hotfix에는 hotfix만. 구조 재편은 별도 사이클.

**판단 기준:**
- 이번 작업의 본래 범위에 포함된 것만 처리
- 작업 중 발견된 추가 이슈는 *"v1.X.2 또는 다음 사이클"* 로 이월
- 예외: 이번 작업과 불가분하게 묶인 이슈만 병합

**방어 대상:** scope crew. 이번 사이클 무한 확장

### D-2. 옵션 제시 전 성격 차이 분석

**원칙:** 옵션을 나열하기 전에 대상들의 성격이 동일한지 먼저 확인

**절차:**
1. 비교 대상(예: §3-8-7과 §16)을 나열
2. 각 대상의 성격을 한 줄로 요약 (절차적 checklist? 추적성 기록? 법적 자문?)
3. 성격이 다르면, 통합이 아니라 **분리 + 명시적 차별화**가 옵션
4. 성격이 같으면, 통합·분리·하이브리드 옵션 중 선택

**방어 대상:** 옵션 framing의 묵시적 대칭 가정

### D-3. Deprecate 패키지 구성

**원칙:** 문서/산출물을 deprecate할 때 단순 "종료 선언"으로 끝내지 않고 다음 4요소를 함께 처리

1. **흡수 대상 식별** — deprecated 문서에 살아있는 항목(미해결 이슈, 진행 중 협의 등)이 무엇인지
2. **흡수 위치 결정** — 각 항목을 어느 후속 문서의 어느 섹션으로 옮길지
3. **참조 문서 명시** — 새 독자가 어느 문서를 봐야 할지 헤더에 명시
4. **외부 통보** — 해당 문서를 이미 받은 부서·담당자에게 deprecate 사실과 후속 참조처 안내

**방어 대상:** 유령 문서. 형식상 deprecate되고 실질적으로 살아있는 상태

### D-4. 시퀀싱 리스크 점검

**원칙:** 작업 순서 A → B → C에서 각 단계가 다른 단계의 출력에 의존하지 않는지 확인

**체크 질문:**
- 외부 응답 대기 시간이 숨어 있는가?
- 후속 단계가 선행 단계의 결과 없이 시작 가능한가?
- 병렬 가능한 단계를 순차로 잡지는 않았는가?

### D-5. 클린 재제작 vs 누적형 정책 결정

**원칙:** 버전 관리 방식을 **사이클 시작 시점에** 명시적으로 결정

**클린 재제작:**
- 본문은 현재 시점 설계 상태만 서술
- 변경 이력은 Document History 박스에만 누적
- 직전 버전 파일은 별도 백업 디렉토리 보존
- 매 버전 작성 비용 증가, 대신 가독성 유지

**누적형:**
- 본문에 "v1.3 신규", "v1.4 추가" 같은 버전 마커 유지
- 변경 이력이 본문 안에 살아 있음
- 작성 비용 낮음, 대신 누적되면 가독성 악화

**방어 대상:** 정책 혼용 (선언은 클린이지만 본문은 누적)

## PART E — 자가 패턴 인식 기록 템플릿

### E-1. 세션별 메모 형식

```markdown
# [문서명] v[X.Y.Z] 작업 메모 — 자가 패턴 인식

## 이번 사이클에서 발생한 자가 오류 패턴

### 패턴 1: [패턴 이름]
- 구체 사례: [어디서 어떻게 발생했는지]
- 원인 가설: [왜 발생했는지]
- 재발 방지 장치: [다음 사이클에 적용할 장치]

### 패턴 2: ...

## 다음 사이클 인수인계 체크리스트
- [ ] [항목 1]
- [ ] [항목 2]
- [ ] [항목 3]

## 본 메모의 위치
이 파일은 본문 외 운영 메모이며, 다음 사이클에서 참조용.
Document History에는 포함되지 않음.
```

### E-2. KC 루프에서 식별된 핵심 패턴 (다음 사이클 참고용)

**P-01. 옵션 나열 전 성격 차이 분석 누락**
- 사례: H1 (정책 vs 운영 수치), H2 (절차적 checklist vs 추적성 기록)
- 원인: 두 대상의 성격이 같다는 묵시적 가정 위에서 (a)/(b) 옵션 생성
- 방지: 옵션 나열 전 *"대상들의 성격이 동일한가?"* 질문 우선

**P-02. 일괄 처리 자가 보고의 부정확성**
- 사례: v1.6 "v1.x 마커 12건 일괄 제거" → 실제 4건 본문 잔존
- 원인: Python/sed 스크립트 결과를 최종 권위로 수용, 독립 grep 재검증 생략
- 방지: 매 수정 후 grep 재검증 + 측정값 보고 강제

**P-03. 창작형 cross-reference**
- 사례: v1.6 §14-5-3 D-09 오참조 (agent가 "D-09가 당연히 X일 것"이라 가정)
- 원인: write-then-verify 순서로는 원리적으로 막히지 않음
- 방지: Verify-then-write 순서 적용 (대상 먼저 view)

**P-04. 외부 source에서 끌려온 표현**
- 사례: v1.5 §11-7-3 "동년배 상위 20%" (Clark Prototype v2 시각화에서 유입)
- 원인: 다른 문서·산출물에서 자연스럽게 느껴진 표현을 그대로 사용
- 방지: Source leak sweep (완료 후 외부 source와 독립 cross-check)

**P-05. 시간 추정 낙관**
- 사례: "옵션 A 30분", "옵션 B 1~2시간" → 실제 4~5시간
- 원인: 숨은 dependency와 재검증 사이클을 추정에서 제외
- 방지: 추정치를 1.5~2배로 보정 + 재검증 사이클 포함

## PART F — 사이클 운영 매뉴얼

### F-1. 표준 사이클 구조

```
[Round 1: 초안 작성]
  ↓ (Writer 작업 + 자가 검증 5단계)
  ↓
[Round 2: 독립 검증]
  ↓ (Verifier의 C-1 leak 탐지 + C-3 독립 측정)
  ↓
[Round 2.5: 메타 검증 — 선택]
  ↓ (Meta-Verifier의 MV-1~MV-5 적용)
  ↓
[Round 3: Hotfix 작성]
  ↓ (Writer의 B-1~B-5 5단계 방지 장치 적용)
  ↓
[Round 4: 최종 검증]
  ↓ (Verifier의 독립 batch sweep)
  ↓
[사이클 종결]
  - 측정 가능한 종결 조건 충족 확인
  - 자가 패턴 인식 메모 작성 (E-1)
  - 인수인계 패키지 구성
```

### F-2. 사이클 종결 조건 (Gatekeeping)

다음 **모두** 충족 시에만 종결:

- [ ] Cross-reference 미스매치 0건 (독립 batch sweep 결과)
- [ ] 정책 선언 vs 본문 실태 불일치 0건
- [ ] 명칭·개수 불일치 0건
- [ ] 자가 보고 수치와 독립 측정값 일치
- [ ] 자가 패턴 인식 메모 작성 완료
- [ ] 다음 사이클 인수인계 체크리스트 작성 완료

종결 조건 중 하나라도 미충족이면 추가 라운드 진행.
"이 정도면 됐다"는 주관적 판단 금지.

### F-3. 사이클 종결 후 artifact

- 최종 문서 파일
- Document History 블록 (변경 내역 구조화)
- 작업 메모 (자가 패턴 인식)
- 인수인계 메시지 (다음 단계 담당자용)

## 사용 시나리오

### 시나리오 1: 세션 시작 시 첨부 (Onboarding 방식)

**언제:** 새 대화를 시작하여 새로운 문서 작업 사이클을 돌리려 할 때

**방법:**
1. 새 대화 시작
2. 본 파일을 첨부하면서 다음과 같이 입력:
   > "본 템플릿에 따라 [문서명] 작업을 시작한다.
   > 이번 사이클에서 나는 [Writer / Verifier / Meta-Verifier] 역할을 맡을
   > 에이전트가 필요하다. PART [A/B/C/D]를 중점 적용해달라."
3. 에이전트가 템플릿 전체 맥락을 사전 로딩한 상태로 작업 시작

**장점:**
- 매 라운드마다 재설명 불필요
- 에이전트가 표준 어휘(Writer/Verifier, B-1~B-5, P-01~P-05 등)를 공유한 상태로 시작
- 자가 패턴 인식이 다음 사이클에 자연스럽게 이월

**단점:**
- 전체 템플릿이 컨텍스트를 크게 차지 (약 6~8K 토큰)
- 짧은 작업에는 과잉

**적합한 경우:**
- 여러 라운드를 돌 것이 예상되는 큰 문서 작업
- 작성 + 검증을 한 세션에서 번갈아 수행하는 경우
- 패턴 학습을 에이전트에게 맡기고 싶을 때

### 시나리오 2: 라운드별 발췌 호출 (Tool 방식)

**언제:** 특정 라운드에서 특정 체크리스트만 돌리고 싶을 때

**방법:**
1. 대화 중 필요한 Part만 복사해서 붙여넣기
2. 예시:
   - Writer가 수정 완료 후 검증 요청 → **PART B (5단계 방지 장치)** + **PART C (검증 체크리스트)** 만 발췌
   - 작성 에이전트가 옵션 (a)/(b) 제시 → **PART A 메타 검증자** + **PART C-2 (MV-1~MV-5)** 만 발췌
   - 사이클 종결 직전 → **PART F-2 (Gatekeeping 체크리스트)** 만 발췌

**장점:**
- 필요한 맥락만 주입하여 효율적
- 작업 단계가 불명확한 ad-hoc 상황에도 유연하게 대응
- 에이전트의 컨텍스트 윈도우 절약

**단점:**
- 매번 어느 Part가 필요한지 Jason이 판단해야 함
- 자가 패턴 인식(PART E)이 자동으로 이월되지 않음
- Writer·Verifier·Meta-Verifier 역할 분리 효과가 약화될 수 있음

**적합한 경우:**
- 이미 진행 중인 작업에 특정 검증만 추가 투입
- 짧은 문서 또는 소규모 수정
- 어떤 장치가 이번 작업에 필요한지 Jason이 명확히 아는 경우

### 시나리오 3: 하이브리드 (권장)

**실제로는 두 방식을 섞어 쓰는 게 실무적:**

1. **사이클 시작 시점 (시나리오 1):**
   - 새 대화 시작 시 본 템플릿 전체 첨부
   - 에이전트가 역할·어휘·5단계 장치를 사전 이해

2. **사이클 중간 (시나리오 2):**
   - 작업이 길어져 컨텍스트가 차면, 필요한 Part만 재발췌하여 리마인드
   - 특히 Round 2.5 메타 검증 국면에서 PART C-2 발췌 호출

3. **사이클 종결 (시나리오 1 + 2):**
   - PART E-1 형식으로 자가 패턴 메모 작성
   - 새로 발견된 패턴을 P-06 이상으로 추가
   - 다음 사이클 시작 시 업데이트된 템플릿 파일 첨부

## PART G — 이 템플릿 자체의 한계와 개선 방향

### G-1. 현재 한계

**수동 오케스트레이션 단계:** 검증·측정이 사람(Jason)의 호출로 작동. 자동화되지 않음

**도구 추상화 부족:** grep/wc/awk 기반 검증이 매번 ad-hoc으로 작성됨. 재사용 가능한 스크립트로 패키징되지 않음

**메타-하네스 부재:** Verifier 자체의 검증 품질을 누가 검증하는지 체계화되지 않음

**도메인 특화 부족:** 이 템플릿은 기술 스펙/BRD 문서에 최적화. 다른 종류의 산출물(코드, 발표 자료, 계약서 등)에는 수정이 필요

### G-2. 다음 버전(v0.2)에서 고려할 개선

1. **B-4 batch sweep을 재사용 가능한 스크립트로 패키징**
   ```bash
   # 예: harness-sweep.sh doc.md
   # → cross-ref mismatch, 정책 일관성, 명칭 불일치 자동 리포트
   ```

2. **E-2 핵심 패턴 목록을 세션 시작 시 자동 로드**
   - Writer가 작업 시작 시 이 목록을 먼저 보도록 워크플로우에 포함

3. **F-2 종결 조건을 체크리스트 자동 검증으로 구현**

4. **도메인별 fork 작성**
   - 기술 스펙용 (현재)
   - 코드 리뷰용
   - 외부 커뮤니케이션(메일/메시지)용

### G-3. Dogfooding 테스트

이 템플릿 자체도 하네스 검증 대상.
실제 사용해보고 어느 부분이 작동하지 않는지 피드백 받아 v0.2로 개정.
