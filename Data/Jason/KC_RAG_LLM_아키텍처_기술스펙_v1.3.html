①

Concept 카드

메타 정의 계층 — 보험 개념·용어 정의
Public Phase 1 대상 27종

Concept 카드는 KC 체계의 최상위 계층으로, 보험 및 보장 구조에 대한 기본 개념과 용어를 정의합니다. AIO 백서의 도입부이자 Clark AI의 AI 진단 철학의 근거가 되며, 사용자의 질문 의도를 AI가 정확히 이해하는 데 핵심적인 역할을 합니다.

예를 들어, "보험료"를 "보장을 유지하기 위해 지불해야 하는 비용으로, 월납보험료라고도 하며 일상적으로는 보험비라고도 불린다"라고 정의하면, Clark AI는 사용자가 "내 보험비가 얼마야?"라고 물어도 "월납보험료"로 의미를 정확히 매칭할 수 있습니다.

이원 구조: Standalone / Linked
Standalone (독립형)
Risk-type에 연결되지 않는 독립 개념 카드. 일반 대화 응답, 용어 설명, AIO 콘텐츠, 사내 교육에서 독립적으로 참조. 메인 체이닝에 참여하지 않음
Linked (연결형)
1개 이상의 Risk-type에 연결(1:N 또는 N:M). AI 진단 체이닝의 첫 번째 계층으로서 사용자 질문 의도를 파악한 후 해당 Risk-type으로 진입하는 역할 수행
핵심 필드
한 줄 정의
25자 이내. 카드의 핵심 개념을 한 문장으로 압축
synonyms (동의어)
배열 필드. Clark AI가 사용자 발화를 정확히 매칭하는 데 직접 활용. 예: ["보험비", "월납보험료", "보험 납입금"]
왜 중요한가
3가지 핵심 이유. AIO 콘텐츠와 Clark 진단 근거로 활용
대표 오해
2가지. 사용자가 흔히 가지는 잘못된 인식을 정의하여 Clark 응답 품질 향상

① 사용자 질문이 입력되면, vector search에서 Concept 카드(synonyms 결합 임베딩)가 검색됩니다. ② Linked Concept은 연결된 Risk-type으로 체이닝을 시작합니다. ③ Standalone Concept은 체이닝 없이 독립적으로 LLM 컨텍스트에 제공되어 일반 대화 응답에 활용됩니다.

②

Risk-type 카드

보장 구조 정의 계층 — 리스크 유형 분류
Public 초기 8종

Risk-type 카드는 보험을 상품이 아닌 "구조"로 정의하고, 사용자의 보장 상태에서 발생할 수 있는 위험 유형을 분류합니다. Clark AI가 사용자의 보험 데이터를 AI 진단한 후, 해당 사용자가 어떤 위험 유형에 해당하는지를 판단하는 기준이 됩니다.

각 Risk-type에는 리드 스코어가 배정되어, 보험 설계사가 선호하는 위험 유형에 가점을 부여하는 배정 최적화에도 활용됩니다. Risk-type은 담당자가 KC Admin을 통해 추가할 수 있습니다.

초기 8종 위험 유형
위험 유형설명
무보장형실손보험을 포함한 기초적인 보장이 없는 경우
암투병생활비 부족형암진단금 부족으로 암 발병 시 치료 지속이 어려워지는 경우
심혈관질환보장 협소형협심증 보장은 있으나 허혈성 심장질환 보장이 없는 등 좁은 질병코드 상의 보장만 있는 경우
뇌혈관질환보장 협소형뇌출혈 보장은 있으나 뇌졸중 보장이 없는 등 좁은 질병코드 상의 보장만 있는 경우
노후돌봄보장 공백형간병/치매 보험 등 장수리스크 보장이 없는 경우
납입여력부족 우려형갱신형 보험료가 과다하여 장기적으로 보장 유지가 어려운 경우
비용 대비 보장 비효율형동일 보장이 여러 상품에서 중복 가입되거나 과다 보장을 받고 있는 경우
생애주기 적합도 저조형부양가족이 있는 주소득원임에도 조기 사망 리스크 대비와 노후 자금 마련이 모두 부재한 경우
Clark 런타임에서의 작동

① Linked Concept에서 체이닝으로 진입합니다. ② Clark AI는 사용자의 MyData 보장 정보를 Risk-type 판정 기준과 대조하여, 어떤 위험 유형에 해당하는지 진단합니다. ③ 판정된 Risk-type은 연결된 Rule 카드로 체이닝을 이어가며, 동시에 리드 스코어 산출(aGCR)에 직접 입력됩니다. ④ Risk-type은 FC 전달 시 리드 메타데이터로 포함되어 설계사에게 "이 고객의 핵심 리스크"를 전달합니다.

③

Rule 카드

리스크 평가 계층 — 진단 규칙, 조건 로직
Baseline

Rule 카드는 "보닥만의 판단 기준"을 구축하는 카드로, Clark AI의 AI 진단 핵심 로직입니다. 각 Risk-type에 대해 "어떤 조건일 때 해당 위험 유형으로 판단하는가"의 구체적인 기준값을 정의합니다. 담당자가 KC Admin에서 마이데이터 규격 및 질병코드를 혼합하여 기준을 설정합니다.

핵심 필드
대상 Risk-type
이 Rule이 적용되는 위험 유형. 1:N 연결 가능
판단 조건
마이데이터 필드(coverage_code, coverage_amt 등 7종) + Amplitude 유저 속성(13종) 혼합 조건, AND 결합
기준값
조건의 임계값. KC Admin에서 코드 변경 없이 수정 가능
연결 Evidence
1개 이상 연결 필수. 미연결 시 Draft 상태 유지, Clark AI 참조 불가
공개범위: Baseline

판단 철학("보장 공백이 일정 기준 이상일 때 해당 위험 유형으로 진단한다")은 Public으로 공개. 구체적 기준값(암진단금 5천만 원, 갱신형 비율 70% 등)과 참조 데이터 규격 필드명은 Internal로 비공개.

납입여력부족 우려형갱신형 담보 비율 70% 이상 → "납입 여력" 문제 경고

Clark 런타임에서의 작동

① Risk-type에서 체이닝으로 Rule 카드를 수집합니다. ② Rule의 조건을 사용자 MyData + Amplitude 속성과 실시간 대조하여 충족 여부를 판정합니다 (evaluate-rules 엔진). ③ 충족된 Rule의 경고 메시지가 Clark AI의 응답 생성에 반영됩니다. ④ Rule이 참조하는 Evidence 카드도 함께 수집되어 진단 근거를 보강합니다.

Evidence 연결 필수: Evidence가 1개 이상 연결되지 않은 Rule은 Draft 상태로 유지되며, Clark/Lois가 참조할 수 없습니다. 근거 없는 판단 기준이 고객에게 노출되는 것을 원천 차단합니다.
④

Evidence 카드

Rule 설정의 근거 계층 — 근거 데이터
Internal 4유형

Evidence 카드는 Rule 카드와 별도의 독립 카드로, Rule을 설정할 때 참조하는 데이터 근거의 출처와 활용 방법을 정의합니다. Clark AI가 사용자에게 특정 Risk-type에 해당하는 이유를 설명할 때도 참조됩니다.

4유형 분류
유형정의예시
마이데이터 기반보험 마이데이터 API에서 조회한 담보코드·금액·갱신 상태 등의 데이터를 근거로 활용A4200(암진단) 담보 금액 3,000만 원 미만 고객 비율
보닥 통계 기반보닥 서버 내 마이데이터 통계(동일 연령 평균 비교 등)를 근거로 활용40대 남성 평균 암 진단금 대비 현재 보유 금액 비교
질병코드 기반KCD 질병분류코드와 ICIS 담보코드 간 보장 범위 매핑을 근거로 활용뇌출혈(I60-I62) vs 뇌졸중(I60-I64) 질병코드 범위 차이
프롬에이지 기반Mediage 프롬에이지 API에서 수신한 건강검진 데이터 기반 질병·암·사망 위험도를 근거로 활용. SCI 학술지 논문 7편 이상으로 검증심혈관 질병위험도 동년배 상위 20%, 대사나이 실제 나이 대비 +5세
프롬에이지 기반 Evidence 민감정보 주의: 건강검진 데이터(민감정보)를 근거로 하므로, Evidence 카드 내부에서만 참조되고 Amplitude 등 해외 서버에는 원본 수치를 전송하지 않습니다. Clark AI 프롬프트에는 추상화된 구간 등급(high/medium/low)만 포함됩니다. Phase 1.5에서 연동 예정.
Clark 런타임에서의 작동

① Rule에서 체이닝으로 Evidence 카드를 수집합니다. ② Evidence의 데이터 근거가 LLM 프롬프트 컨텍스트에 포함되어, Clark AI가 "왜 이 Risk-type으로 판정했는지"를 사용자에게 설명할 때 참조합니다. ③ 프롬에이지 기반 Evidence는 Fromage 연동 고객에게만 제공되는 정밀 진단 근거입니다 (이원 흐름).

⑤

Playbook 카드

개입 우선순위 계층 — 리드 전환·스코어링·배정
Internal 3레이어

Playbook 카드는 Clark AI가 인간 설계사의 개입 시점을 판단하여 리드를 전환·배정하는 방법을 정의합니다. 메인 체이닝(Concept→Risk-type→Rule→Evidence→Policy)과 병렬로 동작하며, 리드 전환 시그널이 감지될 때 참조됩니다.

3개 레이어
레이어 1: 리드 전환
"이제 어떻게 해야 해?", "보험 추천해줘" 등의 발화가 감지되면 보닥플래너에게 리드를 전환합니다. 발화 패턴별 우선순위(P0~P3)와 intent_weight(0.15~0.95)를 정의합니다. KC Admin에서 키워드 추가/수정 가능 (코드 변경 불필요).
레이어 2: 리드 스코어링
Risk-type 스코어 + 상담 희망 유형 스코어를 합산하여 총 리드 스코어를 산출합니다. aGCR 공식의 각 변수를 어디서 가져오고 어떻게 계산하는지 정의합니다.
레이어 3: 배정 최적화
AllocationScore = aGCR × PlannerFitScore로 최종 배정 점수를 산출하고, 배정 규칙 분기(무료/유상/고위험), 미수락 시 재배정 로직, aGCR 갱신 트리거를 정의합니다.
Clark 런타임에서의 작동

① Clark AI가 Risk-type을 진단한 후, 사용자의 매 발화에서 Playbook의 전환 키워드를 병렬 매칭합니다. ② 전환 시그널 감지 + 사용자 상담 신청 + 제3자 정보 제공 동의 완료 시, Playbook의 스코어링 레이어가 aGCR v0을 산출합니다. ③ 배정 최적화 레이어가 AllocationScore를 계산하여 FC API로 리드를 전달합니다. ④ Playbook은 체이닝에 포함되지 않으므로, 체이닝 로직에 Playbook을 포함하지 않습니다.

⑥

Policy 카드

최종 출력 통제 계층 — 규제 준수·출력 통제
Internal 준법감시인 승인 필수

Policy 카드는 Clark AI의 최종 출력을 규제 범위 내에서 통제하는 안전장치입니다. Clark AI가 생성한 답변이 금융소비자보호법, 보험업법, 기타 보험산업 내 AI 관련 규제 사항에 저촉되지 않는지를 검증하는 레이어입니다.

컴포넌트 간 호출 시퀀스

Clark 런타임의 컴포넌트 간 호출 순서입니다. 기술연구소가 어떤 서비스를 만들고, 어떤 서비스를 호출해야 하는지를 보여줍니다.

1

사용자 질문 입력

Clark 웹뷰 — 인증 완료 상태에서 보험 관련 질문 입력

Clark 웹뷰는 bodoc 앱과 별도의 독립 웹뷰로 배포됩니다. 사용자는 아래 인증 단계를 완료한 상태에서 대화를 시작합니다.

인증 플로우

비인증 대화 시작 → 카카오 로그인 → 본인인증(CI) → MyData 연동

공유 인프라

bodoc의 기존 MyData + 카카오 로그인 인프라 공유. 추가 인증 서버 불필요

사용 가능 데이터

MyData 보장 정보 (담보코드, 보장금액), 기본 인적사항, Amplitude 유저 속성 13종

배포 형태

2

질문 임베딩 → pgvector 검색

사용자 질문을 벡터로 변환하고, pgvector에서 유사한 KC 카드 top-k개를 검색
임베딩 모델

Solar Embedding 또는 별도 선정. 카드 입력 시 사용한 동일 모델로 질문도 임베딩

검색 방식

cosine similarity 기반 top-k (권장 k=5~10). pgvector의 ivfflat 또는 hnsw 인덱스 사용

-- pgvector 검색 쿼리 예시 SELECT card_id, card_type, title, content, 1 - (embedding <=> query_embedding) AS similarity FROM kc_cards WHERE status = 'APPROVED' ORDER BY embedding <=> query_embedding LIMIT 10;
KC vs 일반 RAG 차이: 일반 RAG는 문서를 임의 크기로 자르기 때문에 의미 경계가 모호합니다. KC는 카드 단위 = 청크 단위이므로 별도의 chunking 전략이 필요 없고, 카드 경계가 곧 의미 경계입니다.
3

카드 체이닝 (multi-hop)

검색된 카드의 링크 구조를 따라 Concept → Risk-type → Rule → Evidence → Policy 연쇄 수집

이 단계가 KC 아키텍처의 핵심 차별점입니다. 단순 vector search만으로는 기존 RAG와 차이가 없지만, 카드 간 참조 관계를 따라 관련 카드를 연쇄적으로 수집함으로써 LLM이 동일 의미 경계(Meaning Layer) 내에서만 응답을 생성하도록 통제합니다.

체이닝 방향

Concept → Risk-type → Rule → Evidence → Policy (단방향). Standalone Concept은 체이닝 미참여

Phase별 구현

Phase 1: SQL JOIN (카드 테이블 FK 기반)
Phase 2: edge table (from/to/relation_type)

-- Phase 1: SQL JOIN 기반 체이닝 예시 -- step 1: vector search 결과에서 linked concept 찾기 SELECT rt.* FROM kc_cards rt JOIN card_links cl ON cl.to_card_id = rt.card_id WHERE cl.from_card_id IN (:vector_search_results) AND rt.card_type = 'Risk-type' AND rt.status = 'APPROVED'; -- step 2: Risk-type에서 Rule로 SELECT r.* FROM kc_cards r JOIN card_links cl ON cl.to_card_id = r.card_id WHERE cl.from_card_id IN (:risk_type_ids) AND r.card_type = 'Rule'; -- step 3: Rule에서 Evidence로 (Evidence 1개 이상 연결 필수) -- step 4: Policy 카드 수집 (전역 적용)
Phase 2 전환 시: edge table (from/to/relation_type 스키마)로 전환하면 SQL JOIN 대신 그래프 순회 로직으로 변경됩니다. 향후 Graph DB 전환 대비 설계이므로, Phase 1에서도 card_links 테이블 스키마를 edge table 호환으로 설계하는 것을 권장합니다.
4

프롬프트 컨텍스트 조립

수집된 카드셋 + 사용자 MyData + Policy 규칙을 LLM 프롬프트 구조로 조립
System prompt

Clark 페르소나 (스마트한 조력자, 존댓말, 공감 중심) + Policy 카드 내용 (법적 준수 9개 항목, 출력 통제 규칙)

User context

체이닝으로 수집된 카드셋 (Concept~Evidence) + MyData 보장 정보 (담보코드, 보장금액) + Amplitude 유저 속성

민감정보 제한

건강검진 데이터(Fromage)는 국내 서버에서만 처리. LLM 컨텍스트에는 추상화된 구간 등급(high/medium/low)만 포함

컨텍스트 크기 관리

카드 수가 많을 경우 similarity 점수 + 카드 타입 우선순위로 pruning. Policy 카드는 항상 포함

리드 전환 · aGCR · FC 연동
Clark 대화에서 리드 전환 시그널이 감지되면, aGCR v0 스코어를 산출하고 FC API로 리드를 전달하는 프로세스입니다.

aGCR v0 스코어링 (리드 단계)

aGCR = (P_retain × LTV × Margin × P_convert) / Cost_lead
adjusted Gross Contract Revenue — "이 계약은 지금 팔아도 되는 계약인가?"에 대한 숫자 답
v0 (리드 단계) 제약사항: 상품(Product) 미확정 상태이므로 P_retain, LTV, Margin은 Product 노드를 직접 참조할 수 없습니다. 세그먼트 평균값 또는 Risk-type별 추정치를 사용하고, v1 (PROPOSED 시점)에서 실제 Product 기반으로 갱신합니다. v0 vs v1 차이율 30% 이상 시 담당자에게 알림이 발송됩니다.

FC 이원 구조

대상전달 데이터추적 단위
더파트너스 (내부 GA)리드 + aGCR + 추천 설계사 리스트 + Playbook 메타Planner Agent
제휴 GA (외부)리드 + aGCR + 배정 가이드 (설계사 리스트 제외)Channel Agent
2027 "1200% 룰" 대비: 규제 변경으로 Margin이 near-constant화되면 aGCR에서 P_retain × LTV × P_convert가 핵심 변수가 됩니다. Margin 변수를 분리 설계하여 향후 상수 처리 전환이 용이하도록 합니다.

Concept 카드 동의어(synonyms) 활용

Concept 카드의 synonyms 필드는 Clark AI가 사용자의 질문 의도를 정확히 매칭하는 핵심 메커니즘입니다. 예: "보험료" 카드의 synonyms = ["보험비", "월납보험료", "월 보험료", "보험 납입금"]. 사용자가 "내 보험비가 얼마야?"라고 물어도 "월납보험료"로 정확히 매칭됩니다.

활용 단계방식구현
임베딩 시카드 텍스트(title + content)에 synonyms를 결합하여 임베딩 생성. 동의어가 임베딩 벡터에 반영됨embedding_input = f"{title} {content} 동의어: {', '.join(synonyms)}"
검색 시사용자 질문의 vector search 결과에 동의어가 포함된 카드가 높은 유사도로 반환추가 구현 불필요 (임베딩에 이미 반영)
프롬프트 조립 시LLM 컨텍스트에 카드와 함께 synonyms 목록도 전달하여 응답 시 용어 매칭 정확도 향상카드 JSON에 synonyms 필드 포함

card_links 테이블 스키마 (Phase 2 edge table 호환)

CREATE TABLE card_links ( id SERIAL PRIMARY KEY, from_card_id INTEGER NOT NULL REFERENCES kc_cards(id), to_card_id INTEGER NOT NULL REFERENCES kc_cards(id), relation_type VARCHAR(50) NOT NULL, created_at TIMESTAMP DEFAULT NOW(), UNIQUE(from_card_id, to_card_id, relation_type) );

relation_type enum 정의

relation_typefrom → to설명
CONCEPT_TO_RISKTYPELinked Concept → Risk-type개념이 리스크 유형을 정의
RISKTYPE_TO_RULERisk-type → Rule리스크 유형의 판단 규칙
RULE_TO_EVIDENCERule → Evidence규칙의 근거 데이터 (1개 이상 필수)

RISKTYPE_TO_PLAYBOOKRisk-type → Playbook리스크 유형별 전환/배정 전략
CASE_FROM_RISKTYPECase → Risk-type성공 케이스의 원인 리스크 유형
Phase 2 전환: 이 스키마를 유지하면 Phase 2에서 edge table로 전환할 때 relation_type에 속성(weight, confidence 등)만 추가하면 됩니다. Graph DB 전환 시에도 동일 스키마를 edge로 매핑할 수 있습니다.
