01

논의 배경과 목적

KC(Knowledge Core) 온톨로지를 기반으로 보험 GA 시장의 디지털 트윈을 구현하고, 리드 스코어링과 배정 최적화를 실현하기 위한 아키텍처를 설계하는 과정에서, 다음의 핵심 질문들이 순차적으로 제기되었다.

디지털 트윈에서 "Agent"란 무엇인가, AI와 어떻게 상호작용하는가, 데이터는 어디에 저장하는가, 외부 CRM(First Connect)과 어떻게 연동하는가, 그리고 내부 조직(더파트너스)과 외부 조직(제휴 GA)에서 구조가 어떻게 달라지는가.

본 보고서는 이 질문들에 대한 논의 과정과 결론을 체계적으로 정리하여, 전체 시스템의 설계 방향을 공유하기 위해 작성되었다.

02

디지털 트윈의 개념과 GA 보험시장 적용

디지털 트윈이란

디지털 트윈은 현실 세계의 시스템을 소프트웨어 모델로 복제하고, 그 모델 위에서 시뮬레이션을 통해 미래를 예측하는 기술이다. 제조업에서는 물리적 설비의 센서 데이터를 기반으로 작동하지만, GA 보험시장에서는 "사람의 의사결정"이 핵심 변수이므로 확률적 모델로 설계해야 한다.

GA 보험시장에서의 의미

시장 참여자들(고객·설계사·상품·보험사·GA 채널)의 현재 상태와 상호작용을 소프트웨어 모델로 복제하고, "만약 이 리드를 이 설계사에게 배정하면 어떻게 될까?"를 시뮬레이션하는 것이 GA 보험시장 디지털 트윈의 핵심이다.

구현 7단계 프로세스

디지털 트윈 구현은 다음 순서를 따른다.

구성요소 식별(시장에 누가/무엇이 있는가) → 상태 정의(각 요소를 어떤 변수로 기술하는가) → 상호작용 모델링(요소들이 만나면 무슨 일이 벌어지는가) → 시간 축 설계(동기화 주기와 시뮬레이션 범위) → 데이터 파이프라인(수집·갱신·피드백) → 의사결정 연결(시뮬레이션 결과의 활용) → 검증과 보정(트윈의 현실 반영도 확인).

이 중 1~3이 설계, 4~5가 구현, 6~7이 운용이며, 7의 결과가 다시 2~3을 수정하는 순환 구조다.

핵심 판단: Phase 2에서는 시뮬레이션 레벨 2(배치 최적화)까지만 목표로 잡고, 레벨 3(시장 동태 시뮬레이션)은 Case 500건 이상 축적 후(Phase 3)에 도입한다.
03

Agent와 AI의 관계 정의

Agent는 AI가 아니다

디지털 트윈에서 Agent란 Agent-Based Modeling(ABM)의 개념으로, "자기 자신의 상태를 가지고 있고, 규칙에 따라 행동하며, 다른 Agent와 상호작용하는 독립적 데이터 객체"를 뜻한다. Customer Agent는 실제 고객 한 명의 보장 공백 점수, 의도 신호 등을 숫자로 표현한 객체이며, Planner Agent는 설계사 한 명의 응답 속도, 체결율, 업무량을 표현한 객체다.

AI와 Agent의 3가지 상호작용 패턴

패턴 1 — AI가 Agent의 상태를 생성·갱신한다. Clark이 고객과 대화를 분석해서 Customer Agent의 coverage_gap_score, intent_signal 등의 속성에 값을 기록한다. AI가 없으면 Agent는 빈 껍데기다.

패턴 2 — AI가 Agent 간 상호작용의 결과를 결정한다. 스코어링 엔진이 Customer Agent와 Planner Agent의 상태를 동시에 읽고 AllocationScore를 산출한다. 이때 AI는 Agent에 대해 read-only 접근만 하고, 상태 변경은 별도의 이벤트 핸들러가 담당해야 한다.

패턴 3 — Agent의 상태 변화가 AI를 트리거한다. Customer Agent의 intent_signal이 임계값을 넘으면 리드 전환 이벤트가 발생하고, 스코어링 AI가 호출된다.

설계 원칙 3가지: ① AI와 Agent의 책임을 섞지 않는다 (Agent는 상태 저장, AI는 판단). ② AI의 판단 근거를 항상 추적 가능하게 만든다. ③ AI 없이도 Agent 그래프가 마지막 스냅샷으로 유효하도록 설계한다.
04

3대 저장소 아키텍처

KC Admin, Agent 상태 저장소, 서버 DB는 각각 다른 성격의 데이터를 담으며, AI 엔진이 이 세 저장소를 연결하는 구조다.

그림 1. KC Admin · Agent 상태 · 서버 DB · AI 엔진 관계도
저장소 저장 대상 비유 특성
KC Admin 도메인 규칙, 기준값, 카드 사전(Dictionary) 수십~수백 장, 주/월 단위 변경, 사람이 검수
Agent 상태 개별 인스턴스의 현재 상태 환자 차트(Medical Record) 수천~수만 건, 분~시간 단위 변경, 시스템 자동 갱신
서버 DB 원시 트랜잭션 데이터 업무 장부(Ledger) 사건의 원본 기록, 해석 없이 저장
핵심 결론: KC Admin에 Agent를 관리하면 안 된다. UI 목적, 데이터 볼륨, 권한 체계가 근본적으로 다르기 때문이다. Agent 모니터링은 별도 도구(초기: 기존 어드민에 탭 추가)로 분리한다.
05

Graph RAG와 온톨로지의 관계

Graph RAG는 "그래프 DB를 쓰는 RAG"가 아니라, "연결된 지식 체계(온톨로지) 위에서 관계 탐색을 통해 AI에게 맥락을 제공하는 시스템"이다.

일반 RAG vs Graph RAG

구분 일반 RAG Graph RAG
검색 방식 유사도 검색 (similarity search) 관계 탐색 (graph traversal)
검색 결과 독립적인 텍스트 조각의 목록 연결된 경로 (path)
관계 추론 AI가 조각 사이의 관계를 스스로 추론 관계가 이미 그래프에 정의됨
선행 요건 문서 chunking + 벡터 변환 온톨로지 설계 + 그래프 구축

구현 순서는 온톨로지 설계(어떤 개념이 존재하고 어떻게 연결되는가) → 그래프 DB에 스키마 구현(노드/엣지 타입 정의, 데이터 적재) → Graph RAG 엔진 구축(질문을 그래프 탐색 쿼리로 변환하고 결과를 AI에게 전달)이다.

bodoc 맥락: KC Admin의 카드 체계(CONCEPT, Risk Type, Rule, Evidence, Playbook)가 온톨로지의 역할을 하고, DIAGNOSED_BY 같은 엣지가 지식 노드와 인스턴스 노드를 연결하는 구조다.
06

First Connect CRM 연동 구조 (B-2 모델)

배정 구조의 선택지

AI 엔진과 First Connect 사이의 역할 분배에 대해 세 가지 선택지를 검토했다.

선택지 설명 판단
A AI 엔진이 배정 결과까지 결정, FC는 실행만 설계사 데이터 필요 → 제약 존재
B-1 FC 자체 배정 로직, 규칙이 두 곳에 분산 규칙 동기화 문제 → 부적합
B-2 (채택) AI가 리드 스코어 + 배정 가이드를 FC에 전달, FC는 가이드 범위 안에서 실행 반위임 구조, 규칙 단일 원천 유지
B-2의 핵심: 배정의 "결정"은 AI 엔진 + KC Admin이, "실행"은 First Connect가 담당한다. 규칙의 원본은 KC Admin에만 있고, First Connect에는 복잡한 배정 규칙이 불필요하다.
07

더파트너스 vs 제휴 GA 이원 구조

First Connect는 내부용(더파트너스 = 보닥플래너)과 외부용(제휴 GA)으로 사용자가 나뉜다. 두 채널은 설계사 데이터 접근 수준이 근본적으로 다르며, 이 차이가 전체 구조를 양갈래로 나눈다.

그림 2. 더파트너스 / 제휴 GA 이원 서비스 플로우 (B-2 모델, 추적 제약 반영)

내부 vs 외부: 3가지 결정적 차이

구분 더파트너스 (내부) 제휴 GA (외부)
AI 엔진 출력물 리드 + aGCR + 추천 설계사 리스트 [1, 2, 3순위] 리드 + aGCR + 배정 가이드 (추천 없음)
피드백 데이터 수락/통화 시도(안심번호)/체결(후행 매칭) 배정 여부/체결 여부 (GA 보고 의존)
Agent 구축 단위 설계사 단위 (Planner Agent) GA 단위 (Channel Agent)
배정 최적화 시점 Phase 1부터 부분 정밀 배정 Phase 1은 리드 품질 최적화만, Case 축적 후 정밀화
전략적 의미: 더파트너스가 aGCR 시스템의 실험장이자 증명 무대가 된다. 내부에서 "AI 기반 배정이 전환율을 X% 높인다"를 실증하고, 그 데이터를 근거로 제휴 GA에 설계사 정보 공유를 제안할 수 있다.
08

데이터 추적 현실과 Planner Agent 구축 전략

추적의 현실적 제약

안심번호 트래킹을 통해 보닥플래너의 통화 시도까지는 추적이 되지만, 개인 핸드폰을 통한 고객 상담 및 제안 과정은 추적이 불가능하다. 계약 체결 정보는 앰플리튜드를 통해 고객 정보 기반으로 후행 매칭하는 구조다.

Planner Agent 변수별 현실성

변수 더파트너스 제휴 GA
response_speed (응답 속도) 즉시 가능 안심번호 첫 통화 시도 시간 불가 설계사 단위 미파악
closing_rate (체결율) 후행 가능 앰플리튜드 후행 매칭 (수일~수주 시차) GA 단위만 GA 보고 의존
matching_accuracy 축적 후 closing 축적 → Risk Type별 분해 불가
capacity (활성 리드 수) proxy 14일 경과 = 종료 추정 불가
현실적 결론: 더파트너스에서도 Phase 1에서는 response + closing(후행)만으로 동작하는 경량 PlannerFitScore로 시작한다. 제휴 GA에서는 설계사 단위가 아닌 GA 단위의 Channel Agent로 전환율을 축적한다. 상담 과정 추적 보강(FC에 상담 완료 버튼 추가 등)은 Phase 2 과제다.
09

Phase별 로드맵과 선행 과제

Phase 1 — 즉시 실행

AI 엔진이 리드 스코어(aGCR) + 배정 가이드를 산출하여 First Connect에 전송하는 파이프라인을 구축한다. 더파트너스 경로에서는 추천 설계사 리스트까지 포함하고, 제휴 GA 경로에서는 배정 가이드만 포함한다.

동시에 First Connect에서 배정 결과(수락/통화 시도/체결)를 콜백으로 받아 서버 DB에 축적하기 시작한다. 이것이 Planner Agent의 씨앗 데이터가 된다.

KC Admin에 Playbook/Rule 카드로 Agent 관련 규칙(임계값, 가중치, 배정 로직)을 입력한다.

Phase 2 — 데이터 축적 후

축적된 성과 데이터로 설계사별 프로필(Planner Agent)을 구축한다. 안심번호 통화 시도 데이터와 앰플리튜드 후행 매칭 데이터를 결합하여 response_speed, closing_rate를 산출한다.

Agent 모니터링 대시보드를 기존 어드민에 탭으로 추가한다. Graph DB 도입을 위한 Federated 구조 설계를 시작한다.

제휴 GA와의 계약에 설계사 정보 공유 조건 협상을 병행한다 — 더파트너스에서의 전환율 향상 실증 데이터를 협상 근거로 활용한다.

Phase 3 — 장기

Graph DB 전면 도입 및 Graph RAG 엔진 구축. 디지털 트윈 레벨 3(시장 동태 시뮬레이션) 도입. Case 피드백 루프의 자동 보정 파이프라인 구축.

First Connect에 요구되는 최소 기능 (선정 체크리스트)

기능설명
