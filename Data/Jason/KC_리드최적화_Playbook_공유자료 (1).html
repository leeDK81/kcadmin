01

한마디로 뭘 하려는 건가?

지금은 리드가 생기면 감이나 순서대로 설계사에게 배정합니다. 이걸 데이터와 AI가 자동으로 최적화하게 만들겠다는 거예요.

핵심 질문: "이 계약은 지금 팔아도 되는 계약인가?" 그리고 "이 설계사가 이 리드에 가장 적합한가?"
이 두 질문에 숫자로 답하는 시스템을 만드는 것이 목표입니다.

그런데 이걸 제대로 하려면 밑바닥부터 차근차근 쌓아야 합니다. 그래서 4층짜리 건물을 짓는 겁니다.

03

aGCR — 리드의 가치를 숫자로 표현

adjusted Gross Contract Revenue. "이 계약은 지금 팔아도 되는 계약인가?"에 대한 숫자 답.

aGCR = (유지확률 × LTV × 마진 × 전환확률) / 리드비용
aGCR이 높다 = "오래 유지할 것 같고, 보험료도 많이 내고, 수수료율도 좋고,
상담 의향도 높은데, 데려오는 비용은 적었다"

리드 단계에서의 현실성 검증 결과

리드 단계에서는 아직 가입할 상품이 정해지지 않았습니다. 따라서 유지확률, LTV, 마진은 상품을 직접 참조할 수 없고, 추정 → 확정 2단계 계산 구조로 설계했습니다.

변수리드 단계 추정 방법확정 시점상태
전환확률 Clark 발화 의도(0.4) + 앰플리튜드 engagement(0.35) + Risk Type 긴급도(0.25) 리드 생성 시 확정 즉시 가능
유지확률 마이데이터 갱신형 담보 비율 기반 추정 + Risk Type별 전환 패턴 통계 PROPOSED 시 상품 확정 proxy 시작
LTV 현재 월납 보험료 총액(마이데이터) × Risk Type별 리모델링 변동률 PROPOSED 시 상품 확정 proxy 시작
마진 Risk Type별 과거 체결 상품 평균 수수료율 (2027년 1200% 룰 이후 준상수) PROPOSED 시 보험사 확정 proxy 시작
리드비용 광고 채널별 CPA(설치 기준) ÷ 설치→리드 전환율 리드 생성 시 확정 즉시 가능

aGCR만으로는 부족합니다. 아무리 좋은 리드라도 엉뚱한 설계사에게 배정하면 체결이 안 되니까요. 그래서 PlannerFitScore를 곱합니다.

최종 배정 점수 = aGCR × PlannerFitScore
PlannerFitScore = (응답속도 × 0.3 + 매칭도 × 0.4 + 체결율 × 0.3) × 여력

이 시스템의 핵심 — 코드 없이 AI를 통제한다

엔진이 한번 만들어지면, 그 이후에는 KC Admin의 카드 수정만으로 Clark과 배정 엔진의 동작이 바뀝니다.

KC Admin에서 수치/컬럼 조정만으로 통제 가능

O
기준값 변경 — "암진단금 부족 기준을 5천만 원에서 7천만 원으로"
Rule 카드 기준값 수정 → 검수 승인 → Clark 진단 결과 즉시 반영
O
가중치 튜닝 — "matching 가중치 0.4 → 0.5, response 0.3 → 0.2"
Playbook 카드 수정 → 검수자 승인 → 배정 순위 자동 변경. 코드 배포 불필요.
O
전환 키워드 추가 — "보험료 줄일 수 있어?"를 P2 발화로 등록
Playbook 카드 키워드 추가 → Clark이 해당 발화 감지 시 전환
O
규제 대응 — "2027년 1200% 룰 시행"
Policy 카드 추가 + Playbook에서 Margin 고정값 처리. 코드 변경 없음.
O

실제 운용 시나리오

이 시스템이 동작하면, 일상적인 비즈니스 판단이 이렇게 바뀝니다.

시나리오 1: "암보험 리드 체결율이 떨어지고 있다"
1단계 → Case 카드에서 암투병생활비 부족형 실패 케이스 분석
2단계 → 원인 파악: 전환 시점이 너무 빨라 고객 준비 부족
3단계 → Rule 카드 수정: 암투병생활비 부족형의 전환 임계값 0.75 → 0.85 상향
4단계 → 검수자 승인
5단계 → Clark이 암 관련 상담에서 좀 더 충분한 정보를 제공한 후 전환
코드 변경: 없음. 기술연구소 일정과 무관하게 즉시 실행 가능.
시나리오 2: "특정 설계사가 리드를 받아놓고 연락을 안 한다"
1단계 → Evidence에서 해당 설계사 response_speed 확인: 평균 240분
2단계 → Rule 카드 자격 요건 확인: 180분 초과 → 미충족
3단계 → 시스템이 자동으로 해당 설계사를 배정 대상에서 제외
더 엄격하게 하고 싶으면? Rule 카드에서 180분 → 120분으로 수정 + 검수 승인
코드 변경: 없음.
시나리오 3: "2027년 1200% 룰 시행"
1단계 → Policy 카드에 1200% 룰 규제 추가, 준법감시인 승인
2단계 → Playbook 스코어링 카드에서 Margin 변수를 고정값으로 수정
3단계 → aGCR에서 Margin 변별력 소멸 → P_retain × LTV × P_convert가 핵심이 됨
코드 변경: 없음. 규제 변화에 카드 수정만으로 대응.

전략적 의미

LLM은 교체 가능하지만, 온톨로지는 교체 불가능한 플랫폼 자산이다.

Clark의 LLM을 GPT에서 Claude로 바꿔도, KC 카드에 정의된 기준값과 규칙은 그대로 동작합니다. 판단의 뇌(온톨로지)와 말하는 입(LLM)이 분리되어 있으니까요. Playbook 카드에 담긴 aGCR 공식, Risk Type별 스코어, 배정 규칙, 전환 키워드 — 이것들이 아이지넷만의 고유 자산이 됩니다.
데이터 인프라 과제 2건 (기술연구소 협의 필요)

1. Clark 진단 결과와 체결 데이터 연결 파이프라인 구축 → matching 변수 활성화 전제조건
2. 앰플리튜드 택소노미 이벤트 추가 정의 → engagement 변수 정밀도 향상
