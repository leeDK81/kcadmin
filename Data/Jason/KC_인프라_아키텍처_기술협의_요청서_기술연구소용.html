1. 프로젝트 배경과 대표 방침

Knowledge Core(KC)는 보닥의 보험 진단 지식을 체계화하여 AI가 활용할 수 있는 구조화된 지식 체계(온톨로지)로 구축하는 프로젝트입니다.

대표 방침 (즉시 적용):
① 보닥 앱 및 플래너웹 개발 업데이트 중지 → KC 기반 AI 아키텍처 설계에 집중
② 기술연구소 전체 인력을 KC 프로젝트에 투입
③ 기존 제품은 운영 유지(장애 대응, 긴급 버그 수정)만 진행

2. Phase별 인프라 로드맵

핵심 원칙은 "기존 인프라 활용 → 점진적 확장 → 필요 시 전환"입니다. 단계마다 인프라 리스크를 최소화하면서 점진적으로 확장합니다.

3-2. 왜 처음부터 Graph DB를 안 쓰는가?

기준edge table (PostgreSQL)Graph DB (Neo4j 등)
도입 비용₩0 (기존 DB에 테이블 추가)새 인프라 도입 + 운영 학습
2~3단계 탐색충분히 빠름과잉
5단계 이상 탐색느리고 쿼리 복잡자연스럽고 빠름
KC 카드 규모수백 장 → 충분수천 장 이상일 때 진가 발휘
운영 부담기존 DBA가 관리별도 전문성 필요

3-3. Graph DB 전환 판단 기준

Phase 2 후기~Phase 3에서 다음 세 조건 중 하나라도 충족되면 전환을 검토합니다:

① KC 카드 수가 수천 장 이상으로 증가
② 탐색 깊이가 5단계 이상으로 복잡해짐
③ edge table 기반 쿼리의 응답 시간이 임계값을 초과

3-4. 전환을 위한 edge table 설계 원칙

요청: edge table 설계 시 다음을 준수해주시기 바랍니다:
① Agent 간 관계를 별도 edge table로 분리 (FK로 직접 연결하지 않음)
② from_card_id, to_card_id, relation_type 구조 유지
③ 관계 유형을 문자열(enum)로 관리

이 구조를 유지하면, Graph DB로 마이그레이션 시 노드/엣지 변환이 단순해집니다.

4. 3대 저장소 분리 원칙

KC 아키텍처에서 데이터는 세 가지 저장소에 분리되어야 합니다. 혼재하면 UI 목적 충돌, 데이터 볼륨/갱신 주기 불일치, 권한 체계 혼선이 발생합니다.

3대 저장소 역할 분리
KC Admin에 Agent 데이터를 넣으면 안 되는 이유 3가지:
① UI 목적이 충돌 — 신중한 편집(KC Admin) vs 대량 모니터링(Agent)
② 데이터 볼륨/갱신 주기가 다름 — 수백 장을 천천히 vs 수만 건을 빠르게
③ 권한 체계가 다름 — 편집 권한(전문가) vs 모니터링 권한(운영자)

5. Clark/Lois 독립 웹뷰 아키텍처 개요

Clark/Lois는 보닥 앱과 별도의 독립 웹뷰 기반 AI 보험 진단 서비스입니다.

보닥 인프라 공유 항목

공유
마이데이터 인프라
별도 사업자 등록 불필요. 보닥의 마이데이터 API를 Clark/Lois 백엔드에서 호출.
공유
카카오 로그인
카카오 알림톡 연동 포함. CI 확보 후 마이데이터 조회에 활용.

고객 흐름

진입(인증 없이 대화) → 카카오 로그인(1차 인증) → 본인인증(2차, CI 확보) → 마이데이터 연동 → KC 기반 AI 진단 → 리드 전환 → FC 전달

폴백: 본인인증 불가 시 보험 증권 스크린샷 입력으로 대체

기술 고려사항

① 모든 대화 로그 저장 필수 — Phase 2 Agent 상태 갱신 데이터로 활용
② 파트너사 임베디드 웹뷰 사양 정의 필요 (SDK 또는 iframe)
③ 리드 하루 3만 건 이상 — 카드 매핑 자동화 파이프라인 필요

6. Rule-Evidence 연결 필수화 — 백엔드 구현 요건 v1.2.1 신규

KC Admin UI 설계서 v1.2.1에서 Rule 카드의 Evidence 연결이 필수화되었습니다. 이를 백엔드에서 구현해야 합니다.

구현 요건:
① Rule 카드 저장 시 Evidence 연결 여부를 검증하는 API 레벨 validation
② Evidence 미연결 Rule 카드의 상태 전이 차단: Draft → IN_REVIEW 전이 시 Evidence 연결 필수 체크
③ 검수 대기 API에서 Rule 카드 조회 시 Evidence 연결 건수를 함께 반환
④ Evidence 미연결 Rule 카드에 대한 승인 API 호출 차단 (400 응답)

7. First Connect 연동 최소 API 스펙

배정의 "결정"은 AI 엔진 + KC Admin이, "실행"은 First Connect가 담당하는 B-2 모델(반위임 구조)입니다. 규칙의 원본은 KC Admin에만 있고, First Connect에는 복잡한 배정 규칙이 불필요합니다.

기능설명
리드 수신 APIAI 엔진이 보내는 리드 + 메타데이터(aGCR, Risk Type, 추천 리스트 또는 배정 가이드)를 받는 엔드포인트
우선순위 기반 순차 배정더파트너스: 1순위 → 타임아웃 → 2순위 · 제휴 GA: 배정 가이드 참고 자체 매칭
설계사 워크스페이스리드 확인, 고객 정보 열람, Rule/Playbook 정보 조회, 컨택트 기록
결과 콜백 API수락/미수락/통화시도/체결 이벤트를 bodoc 서버로 전송 (피드백 루프 필수)

8. KC Admin 프로토타입 현황과 Phase 1 개발 범위

8-1. 프로토타입 현황

이미 있는 것: 바이브 코딩으로 만든 KC Admin 프로토타입 (고객경험팀 테스트 중)
포함 기능: 카드 CRUD, 카드 간 관계 설정, 승인 워크플로우(Draft→Active), 역할별 권한 관리

Phase 1은 "프로토타입을 프로덕션 수준으로 재구축"하는 것입니다. 스펙이 명확하므로 개발 기간이 단축됩니다.

10. 기술 협의 요청 질문 목록

Phase 1~1.5 실행을 위해 다음 항목에 대한 기술연구소의 검토와 의견이 필요합니다:

1
현재 PostgreSQL에서 카드 간 관계(엣지)를 저장하고 탐색하는 것이 가능한가? Phase 2에서 edge table 추가 시 기존 스키마에 영향은?
2
Clark/Lois가 KC 카드를 참조할 때 예상 쿼리 복잡도와 응답 시간은? (2~3단계 관계 탐색 기준)
3
Phase 2에서 Agent 상태 테이블을 추가할 때 기존 DB의 부하 영향은? (갱신 빈도: 분 단위)
4
향후 Graph DB 마이그레이션을 전제로 할 때, 지금 테이블 설계에서 지켜야 할 조건은?
5
Clark/Lois 웹뷰에서 보닥 마이데이터 인프라를 공유할 때 기술적 제약은?
6
기존 보장 분석 엔진의 판정 결과에 KC 카드 ID를 태깅하는 로직 추가가 가능한가? (하루 3만 건 처리 기준)
