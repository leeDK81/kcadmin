1. 프로젝트 배경과 목적

KC란 무엇인가?

Knowledge Core(KC)는 보닥의 보험 진단 지식을 체계화하여, AI가 활용할 수 있는 구조화된 지식 체계(온톨로지)로 구축하는 프로젝트입니다.

왜 필요한가?

현재 상태
보닥 앱이 "암보험 보장이 부족합니다"라고 알려줌

하지만 왜 부족한지,
설계사가 어떻게 접근해야 하는지는
설계사 개인 경험에 의존
→
KC 도입 후
AI가 "암 진단비가 3천만원 미만이라 공백입니다"(Rule)
"국가암등록통계에 따르면..."(Evidence)
"30대 남성에게는 이렇게 접근하세요"(Playbook)
까지 완전한 맥락을 제공
대표 방침: 보닥 앱/플래너웹 개발 업데이트를 중지하고, KC 기반 AI 아키텍처 설계에 집중합니다. 기술연구소 전체 인력을 KC 프로젝트에 투입합니다.

각 Phase는 "기능"이 아니라 "검증하려는 가설"을 기준으로 구분됩니다. 각 Phase의 끝에서 가설이 입증되어야 다음 Phase로 진행합니다.

3. Phase 1 — KC Admin 카드 체계

3-1. KC Admin에서 관리하는 카드 6+1종

KC의 지식은 카드(Card) 단위로 관리됩니다. 각 카드는 독립적인 지식 단위이며, 다른 카드와 관계를 맺고 있습니다.

Playbook
진단 시나리오
예: "30대 암보험 보장공백 진단" — 특정 상황에서 어떤 순서로 진단하고 상담할지를 정의
Rule
판정 규칙
예: "암 진단비 3천만원 미만 = 보장공백" — 특정 조건을 판정하는 기준
Evidence
근거 자료
예: "국가암등록통계 2024" — Rule의 판정 기준을 뒷받침하는 객관적 데이터
CONCEPT
개념 정의
예: "암 진단비", "보장공백" — 도메인 내 핵심 용어의 정확한 정의
Policy
정책/규제
보험업법, 금융소비자보호법 등 행동의 경계 조건
Case
실제 사례
과거 진단 및 상담 사례에서 추출한 패턴

3-2. 카드 간 관계가 왜 중요한가

카드 하나가 단독으로 존재하면 의미가 없습니다. 카드와 카드 사이의 "참조한다 / 의거한다 / 근거로 삼는다"라는 연결이 있어야 AI가 완전한 맥락을 구성할 수 있습니다.

카드 간 관계 구조 예시

이 관계 구조가 바로 온톨로지이며, Phase 2에서 AI(Clark/Lois)가 카드 관계를 따라 탐색해서 진단 맥락을 구성하는 Graph RAG의 기반이 됩니다.

3-3. Rule-Evidence 연결 원칙 KC Admin UI 설계서 v1.2.1 반영

핵심 원칙: Evidence가 연결되지 않은 Rule은 "Draft" 상태로 유지되며, Clark/Lois가 참조할 수 없습니다. Evidence가 연결되고 준법감시인 검수가 완료된 Rule만 "Active" 상태로 전환되어 AI가 활용할 수 있습니다.

UI 반영 (v1.2.1):
· 카드 에디터(03번)에서 Rule 카드의 "연결 Evidence 카드" 필드가 필수(*)로 변경
· Evidence 미연결 시 "검수 요청" 버튼 비활성화 + 경고 메시지 표시
· 검수 대기 목록(05번)에서 Evidence 미연결 Rule 카드는 승인 버튼 비활성화

이를 통해 Phase 1 초기에 카드를 빠르게 입력하면서도, 고객에게 노출되는 지식의 품질을 보장합니다.

3-4. 기존 보장 분석 엔진과의 관계

중요: 보닥 앱 내 기존 유지/조정/해지 판정 엔진은 현행 유지합니다. KC의 Rule/Playbook은 기존 엔진을 대체하는 것이 아니라, Clark/Lois라는 별도 채널에서 활용되는 독립적인 지식 체계입니다. 이를 통해 보닥 앱 경유 리드 vs Clark/Lois 경유 리드의 전환율을 비교할 수 있습니다.

4. Phase 1.5 — Clark/Lois 독립 웹뷰

4-1. 제품 개요

Clark/Lois는 보닥 앱과 별도의 독립 웹뷰 기반 AI 보험 진단 서비스입니다. 고객과의 대화를 통해 보장 공백을 진단하고, 상담 의향이 확인되면 리드로 전환하여 First Connect에 전달합니다.

4-2. 배포 형태

독립 앱
PWA 또는 네이티브 앱으로 출시하여 자체 채널로 고객 유입
+
파트너 임베디드
헬스케어/핀테크 앱 내 웹뷰로 삽입하여 파트너 채널에서 고객 접점 확보

4-3. 고객 흐름 (7단계)

1
진입 → 바로 대화 시작
인증 없이 즉시 대화를 시작합니다. 진입 장벽을 최소화하여 이탈을 방지합니다.
2
사전 질문 카드로 관심 영역 파악
"어떤 보험이 궁금하세요?" 등의 질문 카드로 맥락을 형성합니다.
3
카카오 로그인 (1차 인증)
계정 연동 + 이름/연락처 확보. 보닥의 기존 카카오 로그인 인프라를 공유합니다.
4
본인인증 (2차: CI 확보) → 마이데이터 연동
보닥의 마이데이터 인프라를 활용하여 가입 보험을 자동 조회합니다. 불가 시 증권 스크린샷으로 폴백합니다.
5
KC 기반 AI 진단
Rule + Evidence를 참조하여 보장 공백을 설명합니다. "왜 부족한지"와 "근거"를 함께 제시합니다.
6
상담 의향 확인 → 리드 전환
카카오에서 이미 연락처가 확보된 경우 추가 입력 없이 전환 가능합니다.
7
FC로 리드 전달
리드 + aGCR 스코어 + Risk Type + Playbook 메타데이터를 First Connect에 전달합니다.

4-4. 기술적 전제

공유 인프라: 보닥의 마이데이터 인프라(별도 사업자 등록 불필요), 카카오 로그인 인프라(알림톡 연동 포함)를 공유합니다.
데이터 수집: 모든 대화 로그를 저장합니다. Phase 2에서 Agent 상태 갱신, 상호작용 모델링 데이터로 활용됩니다.
자동화: 리드 하루 3만 건 이상 — KC 카드 매핑은 시스템 자동화 필수입니다.

6. Phase 1/1.5 타임라인과 인력 배정

6-1. KC Admin 프로토타입 현황

이미 있는 것: 바이브 코딩으로 만든 KC Admin 프로토타입이 존재하며 고객경험팀이 테스트 중입니다. 카드 CRUD, 카드 간 관계 설정, 승인 워크플로우(Draft→Active), 역할별 권한 관리가 포함되어 있습니다.

Phase 1의 개발 범위: 프로토타입을 프로덕션 수준으로 재구축하는 것입니다. 처음부터 만드는 것이 아니므로 스펙이 명확하고 개발 기간이 단축됩니다.

6-2. Phase 1 구현 대상 (KC Admin UI 설계서 v1.2.1 기반)

KC Admin UI 설계서의 13개 화면 중, Phase 1에서는 핵심 8개 화면을 구현합니다:

화면기능복잡도
