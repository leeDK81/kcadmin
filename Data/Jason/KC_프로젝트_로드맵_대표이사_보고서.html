현재 하루 3만 건 이상의 리드가 생산되고 있지만, 리드에 "왜 이 보장이 부족한지"와 "어떻게 상담해야 하는지"의 맥락이 없습니다. KC는 이 격차를 AI가 활용할 수 있는 구조화된 지식으로 채웁니다.

3. 전략: 보닥 유지 + Clark/Lois 독립 출시

보닥 앱의 기존 보장 분석 엔진은 그대로 유지하고, KC 기반의 AI 진단 서비스를 Clark/Lois라는 별도 채널로 출시합니다.

이 전략의 이점 3가지

이점설명
정치적 리스크 제로기존 유지/조정/해지 엔진의 주도권 문제를 회피합니다. 보닥은 보닥대로, Clark/Lois는 별도로 운영.
전환율 A/B 비교보닥 경유 리드 vs Clark/Lois 경유 리드의 전환율을 오염 없이 비교할 수 있습니다. KC의 가치를 데이터로 증명.
파트너 확장 즉시 가능Clark/Lois가 독립 웹뷰이므로, 헬스케어/핀테크 파트너 앱에 웹뷰로 즉시 임베디드 가능합니다.

Clark/Lois 배포 형태

독립 앱 (PWA 또는 네이티브) + 파트너사 임베디드 웹뷰 — 보닥의 마이데이터 인프라와 카카오 로그인 인프라를 공유하므로 별도 사업자 등록이 불필요합니다.

고객 흐름

인증 없이 대화 시작 → 카카오 로그인(1차) → 본인인증(2차, CI) → 마이데이터 연동으로 가입 보험 자동 조회 → KC 기반 AI 진단 → 상담 의향 확인 → 리드 전환 → FC 전달

4. Phase 구분과 로드맵

각 Phase는 "검증하려는 가설"을 기준으로 구분됩니다. 가설이 입증되어야 다음 Phase로 진행합니다.

Phase검증 가설핵심 구축물인프라 비용
Phase 1 KC 지식 체계가 실제 운용 가능한가? KC Admin MVP 프로덕션 + 카드 입력 ₩0 (기존 인프라)
Phase 1.5 KC 기반 AI 진단이 전환율을 높이는가? Clark/Lois 독립 웹뷰 + KC 연동 + FC 리드 전달 웹뷰 서버만
Phase 2 Agent 기반 자동 배정이 더 효율적인가? Agent 상태 + aGCR 스코어링 + Graph RAG 기존 DB 확장
Phase 3 시뮬레이션으로 시장을 예측할 수 있는가? 디지털 트윈 시뮬레이션 엔진 신규 인프라
핵심: Phase 1은 인프라 비용 ₩0으로 시작합니다. 기존 PostgreSQL과 pgvector를 그대로 활용하며, 새로운 인프라(Graph DB 등)는 Phase 2 이후에 데이터가 축적된 후 검토합니다. 단계마다 인프라 리스크를 최소화하면서 점진적으로 확장하는 전략입니다.

5. Phase 1/1.5 타임라인

출발점: KC Admin 프로토타입이 이미 존재

바이브 코딩으로 만든 프로토타입이 있으며 고객경험팀이 테스트 중입니다. 카드 CRUD, 관계 설정, 승인 워크플로우, 권한 관리가 포함되어 있어 프로덕션 재구축 시 스펙이 명확합니다.

Phase 1 + Phase 1.5 일정 (약 16~18주, 4~4.5개월)

Phase 1과 Phase 1.5는 후반부에 병렬 진행됩니다. Phase 1에서 KC Admin을 구축하는 동안 Clark/Lois UX 설계를 병행하고, KC Admin 완성 후 Clark/Lois 개발에 즉시 착수합니다.

8. 리스크

리스크 1 — CTO 공백
CTO 퇴사 예정, 후임 채용 면접 중, 대행 구조 미정. KC 프로젝트에서 edge table 설계, Clark/Lois 아키텍처 결정 등 CTO급 의사결정이 필요한 항목이 다수 있습니다. 후임 확정 전까지 CTO 대행 구조를 빠르게 확정해야 프로젝트 지연을 방지할 수 있습니다.
리스크 2 — 기존 제품 운영 유지
보닥 앱/플래너웹 업데이트는 중지하지만, 장애 대응과 긴급 버그 수정은 계속 발생합니다. 최소한의 운영 인력 확보가 필요합니다.
리스크 3 — AI 챕터 병목
AI 챕터 2명이 Clark/Lois 진단 엔진, KC 카드 기반 RAG 연동, 향후 aGCR 스코어링까지 담당해야 합니다. Phase 1.5~2에서 병목이 될 가능성이 있습니다.
