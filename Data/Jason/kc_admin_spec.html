Part 1
KC Admin이 무엇인가
개발 착수 전 반드시 공유해야 하는 핵심 개념입니다.

한 줄 정의

KC Admin = Clark AI가 고객에게 답변하기 위해 필요한 "보험 지식 카드"를 관리하는 어드민 시스템입니다.
담당자가 KC Admin에서 카드를 작성·승인하면, 그 카드가 자동으로 Vector DB에 반영되고, Clark이 고객 질문에 답변할 때 해당 카드를 참조합니다.

시스템 흐름 (전체 그림)

[KC Admin]
  담당자/검수자/준법감시인이
  카드를 작성·검수·승인
       │
       │ 카드 ACTIVE 전환 시 자동 트리거
       ▼
[Vector DB]
  카드 텍스트를 임베딩 벡터로 변환·저장
  (카드 수정 → 벡터 교체 / 카드 보관 → 벡터 삭제)
       │
       │ Clark이 고객 질문 수신 시 검색
       ▼
[Clark AI — 고객 대화]
  ① 벡터 검색으로 관련 카드 탐색
  ② 카드 간 참조 체이닝 (Concept → Risk-type → Rule → Evidence)
  ③ 고객 실제 보험 데이터와 Rule 조건 비교 (if-else 로직)
  ④ Policy 제약 확인 후 LLM 답변 생성
  ⑤ 필요 시 설계사 연결 (Playbook → FC API)

KC 카드가 일반 CMS와 다른 점

일반 CMS (예: 공지사항 게시판)KC Admin (온톨로지 기반)
글 1개 = 독립 콘텐츠카드 1개 = 지식 그래프의 노드 1개
카드 간 관계 없음카드끼리 FK 참조로 연결됨 (체이닝)
검색 = 키워드 매칭검색 = 벡터 유사도 검색
저장 = DB 저장저장 = DB 저장 + Vector DB 반영 (2곳)
승인 없음카드마다 사람이 검수·승인 (금융 규제 의무)
승인 절차가 있는 이유 금융 마이데이터 사업자 의무로, KC 카드 콘텐츠는 법적 검토 및 사람의 승인을 거쳐야 Clark이 참조할 수 있습니다. 승인되지 않은 카드(DRAFT, IN_REVIEW 상태)는 Clark이 절대 참조하지 않습니다.

Part 2
카드 7종 정의
각 카드 타입의 필드 목록, 참조 관계, 제약 조건을 정의합니다. * 필수 선택
①
Concept 카드
Clark의 검색 진입점 — 고객 질문과 가장 먼저 매칭

역할: 고객이 "암 보험이 뭐야?" 같은 질문을 하면 벡터 검색으로 가장 먼저 히트되는 카드. STANDALONE이면 바로 답변, LINKED이면 연결된 Risk-type으로 체이닝 시작.

필드 정의

필드명타입필수설명
idstring*자동 생성. 형식: CONCEPT.C01.V1
typeenum*고정값: CONCEPT
titlestring*카드 제목 (예: 암 치료비·생활비 이중 공백의 구조)
questionstring*이 카드가 답하는 핵심 질문 1줄
definitionstring*한 줄 정의 (25자 이내)
contentstring*본문 (왜 중요한가 3가지, 대표 오해 2가지 포함)
synonymsstring[]선택동의어 목록. 벡터 변환 시 본문과 합쳐 검색 정확도 향상
link_typeenum*STANDALONE (독립 개념) 또는 LINKED (Risk-type 연결)
linked_risk_typesstring[]LINKED 시 *연결된 Risk-type 카드 ID 목록. link_type이 LINKED일 때만 활성화
visibilityenum*PUBLIC (Clark 일반 사용) / INTERNAL (내부 전용)
statusenum자동DRAFT / IN_REVIEW / ACTIVE / ARCHIVED
assigneestring선택작성 담당자
created_atdatetime자동생성 시각
updated_atdatetime자동수정 시각

참조 관계

Concept (이 카드) → Risk-type (link_type=LINKED 시)

벡터 DB 임베딩 텍스트

// ACTIVE 상태 전환 시 아래 텍스트를 합쳐 임베딩
embedText = title + " " + definition + " " + synonyms.join(" ") + " " + content
초기 데이터 Phase 1 기준 27장 사전 정의 예정. 담당자가 KC Admin에서 직접 입력. 이후 운영 중 추가 생성 가능.
②
Risk-type 카드
보험 위험 유형 분류 레이블 — 현재 13종 정의됨

역할: 고객의 보험 상태에서 발생할 수 있는 위험 유형을 분류하는 마스터 데이터. Rule 카드와 Concept 카드가 이 카드를 참조함.

필드 정의

필드명타입필수설명

name_kostring*한국어 유형명 (예: 암투병생활비 부족형)
name_enstring*영문 코드명 (예: CANCER_GAP)

risk_levelenum*LOW / MODERATE / HIGH / CRITICAL
lead_score_weightnumber*설계사 배정 우선순위 가중치
statusenum자동ACTIVE / ARCHIVED

RT09비용 대비 보장 비효율형LOW종신보험 중심 구조로 보장 대비 보험료 비효율

개발 포인트 담당자가 KC Admin에서 코드 변경 없이 새 Risk-type을 추가할 수 있어야 합니다. Risk-type은 다른 카드(Rule, Concept)가 FK로 참조하므로 삭제 시 참조 정합성 검사 필요.
③
Rule 카드 KC의 핵심
조건 실행 엔진 — "어떤 조건이면 어떤 위험인가" 정의

역할: "암 진단금이 5천만 원 미만이면 암투병생활비 부족형에 해당한다"는 비즈니스 판단 규칙을 코드가 아닌 카드 데이터로 저장. 담당자가 기준값을 코드 수정 없이 변경 가능. Clark의 실시간 조건 판정이 이 카드의 데이터를 기반으로 실행됨.

필드명타입필수설명

④
Evidence 카드
Rule 판단 기준의 근거 데이터

역할: "왜 5천만 원이 기준인가"에 대한 근거를 담는 카드. Rule이 반드시 1개 이상 연결해야 함. Clark이 고객에게 근거를 설명할 때 사용.

Playbook 카드

⑦
Case 카드

Dry-run 실행 로직 (의사코드)

function evaluateRule(rule, customerData):

  // 1단계: 전제조건(base) — 전부 AND. 하나라도 실패 시 이 Rule 스킵
  for cond in rule.base_conditions:
    if not check(cond, customerData):
      return { matched: false, reason: "전제조건 미충족" }

  // 2단계: 판단조건(judge) — OR 결합. 하나라도 충족 시 해당
  results = []
  for cond in rule.judge_conditions:
    results.push({ cond, passed: check(cond, customerData) })

  if results.some(r => r.passed):
    return { matched: true, level: resolveLevel(results), details: results }
  else:
    return { matched: false, reason: "판단조건 미충족" }

function check(condition, customerData):
  value = getFieldValue(customerData, condition.source, condition.field)
  switch condition.operator:
    "IN":          return condition.value in value          // 담보코드 보유
    "NOT_IN":      return condition.value not in value      // 담보코드 미보유
    "LT":          return Number(value) < Number(condition.value)
    "LTE":         return Number(value) <= Number(condition.value)
    "GTE":         return Number(value) >= Number(condition.value)
    "EQ":          return String(value) == String(condition.value)
    "CONTAINS":    return value.includes(condition.value)
    "NOT_CONTAINS": return !value.includes(condition.value)
