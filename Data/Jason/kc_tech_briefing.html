Part 1
KC가 무엇인가 — 한 문장으로
KC(Knowledge Core)는 보험 지식을 7종의 카드로 관리하는 어드민 시스템이다.

워드프레스가 블로그 글을 만들고·수정하고·삭제하고·발행하듯, KC Admin은 보험 지식 카드를 만들고·수정하고·삭제하고·발행한다. 개발 관점에서 보면 CMS(Content Management System, 콘텐츠 관리 시스템)의 일종이다.

다만 일반 CMS와 다른 점이 세 가지 있다:

  1. 카드 간에 참조 관계가 있다. DB로 치면 FK(Foreign Key, 외래 키)이다. 이 관계가 Clark AI가 답변할 수 있는 범위를 제한한다.
  2. 카드마다 승인 절차가 있다. 작성 → 검수 → 승인 단계를 거쳐야만 Clark이 참조할 수 있다. 승인되지 않은 카드는 Clark이 볼 수 없다.
  3. 승인된 카드는 벡터 DB에 자동 반영된다. 기존에 해본 RAG(Retrieval-Augmented Generation, 검색 증강 생성)와 동일한 흐름인데, 차이점은 "문서를 기계적으로 자르는" 대신 "카드 한 장이 곧 하나의 검색 단위"가 된다는 점이다.

기존 RAG와 KC의 차이

기존에 해본 RAG는 아마 이런 구조였을 것이다:

긴 문서 → 글자 수 기준으로 자르기(chunking) → 벡터 변환 → 벡터 DB 저장

이 방식의 문제는, 자르는 위치에 따라 의미가 쪼개질 수 있다는 것이다. 예를 들어 "암 진단금"이라는 하나의 개념이 두 개의 조각에 나뉘면, AI가 불완전한 정보를 조합해서 엉뚱한 답을 만들어낸다.

KC는 이 문제를 다음과 같이 해결한다:

사람이 카드 단위로 지식을 작성 → 카드 1장 = 검색 단위 1개 → 벡터 변환 → 벡터 DB 저장

자르는 과정이 없다. 사람이 의미 단위로 직접 카드를 작성하기 때문에, 의미가 쪼개질 일이 없다.

Part 2
7종 카드 — 각 카드가 하는 일
각 카드를 클릭하면 상세 정의, 필드 구조, Clark에서의 동작 방식을 확인할 수 있다.
①
Concept 카드
"이 용어가 무슨 뜻이야?"

역할

보험 용어와 개념을 정의하는 카드이다. 고객이 "보험비가 얼마야?"라고 물으면, Clark은 "보험비 = 월납보험료"라는 것을 알아야 한다. 이 매칭 정보를 제공하는 것이 Concept 카드이다.

주요 필드

필드설명
term용어명 (예: "보험료")
definition한 줄 정의, 25자 이내
synonyms[]동의어 목록 (예: ["보험비", "월납보험료"]). 벡터 변환 시 본문과 합쳐서 변환하면 검색 정확도가 올라간다.
link_typeSTANDALONE 또는 LINKED
linked_risk_types[]연결된 Risk-type 카드 ID 목록 (LINKED일 때만)
visibilityPUBLIC / BASELINE / INTERNAL
statusDRAFT / IN_REVIEW / ACTIVE / ARCHIVED

두 가지 유형

  • Standalone (독립형): 다른 카드에 연결되지 않는 독립 개념이다. "보험이란 무엇인가?" 같은 일반 지식으로, Clark이 일상 대화에서 참조한다. 뒤에 나오는 "체이닝"에 참여하지 않는다.
  • Linked (연결형): Risk-type 카드에 연결된 개념이다. "암 보장"이라는 Concept은 "암투병생활비 부족형"이라는 Risk-type에 연결되어 있다. 고객이 "암 보험"에 대해 물으면, 이 연결을 따라가서 관련된 판단 기준과 근거를 모두 가져온다.

Clark에서의 동작

  1. 고객 질문 입력 → 벡터 검색으로 Concept 카드를 찾는다
  2. Standalone이면 → 카드 내용으로 바로 답변한다
  3. Linked이면 → 연결된 Risk-type으로 이동해서 진단을 시작한다
초기 데이터 초기 입력 대상으로 27장이 정의되어 있다. 담당자가 KC Admin에서 직접 입력하며, 이후 필요에 따라 추가 생성할 수 있다.
②
Risk-type 카드
"이 고객은 어떤 위험에 노출돼 있어?"

역할

고객의 보험 상태에서 발생할 수 있는 위험의 종류를 분류하는 카드이다. Clark이 고객의 보험 데이터를 분석한 후, "이 고객은 암 보장이 부족한 유형이다"라고 판정하는 분류 기준이 된다.

주요 필드

필드설명
risk_name위험 유형 이름 (예: "암투병생활비 부족형")
description설명
lead_score_weight리드 점수 가중치 (설계사 배정 우선순위에 사용)

초기 데이터 (8종)

위험 유형의미
무보장형기초적인 보장 자체가 없다
암투병생활비 부족형암 진단금이 부족하다
심혈관질환보장 협소형보장 범위가 좁은 질병코드만 보유하고 있다
뇌혈관질환보장 협소형위와 동일 구조, 뇌 영역이다
노후돌봄보장 공백형간병/치매 보장이 없다
납입여력부족 우려형갱신형 비율이 높아 장기 유지가 어렵다
비용 대비 보장 비효율형중복 가입 또는 과다 보장 상태이다
생애주기 적합도 저조형생애주기 대비 사망/노후 보장이 없다
개발 포인트 담당자가 KC Admin에서 코드 변경 없이 새로운 Risk-type을 추가할 수 있어야 한다.
③
Rule 카드 KC의 핵심
"어떤 조건이면 이 위험에 해당해?"

역할

"어떤 조건을 만족하면 해당 Risk-type으로 판정하는가"의 구체적인 기준을 정의하는 카드이다. 이 카드가 사실상 비즈니스 규칙 엔진이다.

"암 진단금이 5천만 원 미만이면 부족하다"는 판단을 코드가 아니라 카드에 저장해두고, 담당자가 KC Admin에서 코드 변경 없이 기준값을 직접 바꿀 수 있게 만드는 것이다. 기준이 바뀔 때마다 코드를 수정하고 배포할 필요가 없다.

주요 필드

필드설명
target_risk_type대상 Risk-type 카드 ID (FK)
conditions[]조건 목록. 각 조건: source(소스) + field(필드) + operator(연산자) + value(값). 조건 간 AND 결합.
warning_message조건 충족 시 Clark이 사용할 경고 메시지
linked_evidences[]연결된 Evidence 카드 ID 목록 (1개 이상 반드시 필요)
statusEvidence 미연결 시 DRAFT에서 벗어날 수 없다

조건에서 쓸 수 있는 필드 — 마이데이터(MYDATA) 7종

필드명설명타입
coverage_code담보코드 (ICIS 기준)코드
coverage_amt담보금액 (원)숫자
coverage_name담보명텍스트
renewal_status갱신 여부참/거짓
insu_type보험 종류코드
start_date보험 시작일날짜

조건에서 쓸 수 있는 필드 — Amplitude 유저 속성 13종

속성명설명

조건 예시 — "암투병생활비 부족형" 판정

조건 1: 소스=MYDATA, 필드=coverage_code, 연산자=NOT_IN, 값=['A4200']
  → "암진단(A4200) 담보를 아예 갖고 있지 않은 고객"

조건 2: 소스=MYDATA, 필드=coverage_amt, 연산자=LT, 값=50000000
  → "암 진단금이 5천만 원보다 적은 고객"

조건 3: 소스=AMPLITUDE, 필드=age, 연산자=GTE, 값=40
  → "만 40세 이상"

결합: (조건 1 또는 조건 2) 그리고 조건 3
핵심 개발 포인트
  • 조건 빌더 화면: 위 필드 목록을 드롭다운으로 제공하고, 연산자를 선택하고, 값을 입력하는 화면이다. 담당자가 코드 변경 없이 직접 조건을 설정할 수 있어야 한다.
  • Evidence 연결 강제: Evidence가 0개인 Rule은 검수 요청 버튼 자체가 비활성화된다.
  • 조건 대조 엔진: Clark이 실시간으로 고객 데이터를 Rule 조건과 비교하는 로직이다. AI가 아니라 일반 if-else 로직으로 구현한다.
④
Evidence 카드
"그 기준은 근거가 뭐야?"

역할

Rule 카드의 판단 기준이 "왜 그 숫자인가"를 뒷받침하는 근거 데이터를 담는 카드이다. Rule이 "암 진단금 5천만 원 미만이면 부족하다"고 판정할 때, 고객이 "왜 5천만 원이 기준이에요?"라고 물을 수 있다. Evidence 카드가 "40대 남성 평균 암 진단금은 7,200만 원이며..."라는 근거를 제공한다.

4가지 유형 — 데이터가 어디서 오는가

유형데이터 출처예시
마이데이터 기반보험 마이데이터 API에서 가져온 담보코드·금액·갱신 상태"A4200 담보 3천만 원 미만 고객 비율 42%"
보닥 통계 기반보닥 서버에 쌓인 사용자 통계 (연령별 평균 비교 등)"40대 남성 평균 암 진단금 7,200만 원"
질병코드 기반질병분류코드(KCD)와 담보코드(ICIS) 간 매핑"뇌출혈(I60-I62) vs 뇌졸중(I60-I64) 범위 차이"
프롬에이지 기반메디에이지 건강검진 API에서 가져온 질병 위험도"심혈관 질병위험도 동년배 상위 20%"
프롬에이지 관련 주의사항 프롬에이지 기반 Evidence는 건강검진 데이터를 다루는데, 이것은 법적으로 민감정보이다. 원본은 보닥 서버(국내)에만 저장하고, Amplitude(미국 서버)에는 "높음/보통/낮음" 같은 등급만 보낸다. 이 연동은 Phase 1.5에서 진행한다. Phase 1에서는 마이데이터/보닥통계/질병코드 기반 3종만 다루면 된다.
⑤
Playbook 카드
"이 고객을 설계사에게 연결해야 해?"

역할

Clark이 대화 중에 "지금 이 고객을 보험 설계사에게 넘겨야 하는가"를 판단하는 규칙을 정의하는 카드이다. Clark의 목적은 보험 진단 자체가 아니라, 진단 결과를 바탕으로 설계사와 고객을 연결하는 것(리드 전환)이다.

중요 Playbook은 다른 카드들의 연결 흐름(체이닝)과 별도로 동작한다. 체이닝 로직에 Playbook을 포함하지 않는다. 매 대화마다 별도로 키워드를 확인하는 독립 프로세스이다.

3개 단계

  • 전환 단계: "보험 추천해줘", "상담 받고 싶어" 같은 발화를 감지한다. 키워드 목록은 담당자가 KC Admin에서 코드 변경 없이 추가·수정할 수 있다.
  • 점수 산정 단계: Risk-type 점수 + 상담 희망 유형 점수를 합산한다.
  • 배정 단계: 점수에 따라 어떤 설계사에게 배정할지 결정하고, FC(First Connect) API로 리드를 전달한다.
⑥
Policy 카드
"이 답변, 법적으로 문제없어?"

역할

Clark의 최종 답변이 법률이나 규제를 위반하지 않는지 검증하는 안전장치이다. 보험은 금융 규제 산업이다. Clark이 "이 보험 상품을 추천합니다"라고 말하면 법적 문제가 된다. Policy 카드가 "Clark이 해서는 안 되는 것" 목록을 정의한다.

주요 필드

필드설명
regulation_name규제 문서명 (예: "금융소비자보호법")
checklist[]준수 확인 항목 (11가지)
clark_constraints[]Clark이 해서는 안 되는 것 목록
승인 절차가 다르다 다른 카드는 담당자 → 검수자 2단계인데, Policy는 담당자 → 검수자 → 준법감시인 3단계이다. 준법감시인이 승인하지 않으면 활성화되지 않는다.
⑦
Case 카드
"성공한 상담 사례를 모아두자"

역할

Clark과 고객의 대화에서 설계사 연결에 성공한 사례를 모아두는 카드이다. 성공 사례가 쌓일수록 Clark의 진단 품질이 좋아진다. 설계사 교육 자료로도 활용된다.

다른 카드와의 차이 — 생성 방식

  • 다른 6종: 담당자가 KC Admin에서 직접 작성한다.
  • Case: Clark이 리드 전환 성공 사례를 자동으로 태그한다 → 담당자가 검토한다 → 검수자가 최종 승인한다 (반자동).

Part 3
카드 간 연결 구조 — "체이닝"
7종 카드는 독립적으로 존재하지 않고, 서로 참조 관계로 연결되어 있다. 이 연결을 "체이닝"이라고 부른다. DB로 치면 FK(Foreign Key, 외래 키) 관계이다.

메인 체이닝 (한 방향으로만 흐른다)

Concept (Linked) → Risk-type → Rule → Evidence → Policy
  1. Concept → Risk-type: "암 보장"이라는 개념이 "암투병생활비 부족형"이라는 위험 유형으로 연결된다.
  2. Risk-type → Rule: "암투병생활비 부족형"에 해당하는 판단 조건들을 가져온다.
  3. Rule → Evidence: 그 판단 조건의 근거를 가져온다.
  4. Policy (전역): 최종 답변이 규제를 위반하지 않는지 검증한다.
체이닝의 실질적 의미 Clark이 "암 보장"에 대해 답변할 때, 이 체인에 속한 카드들만 참조한다. 체인 밖의 정보(예: 자동차 보험 관련 Rule)는 답변에 섞이지 않는다. 기존 RAG에서 관련 없는 문서 조각이 섞여서 엉뚱한 답변이 나오던 문제를 이 구조로 방지한다.

병렬 흐름 (Playbook)

메인 체이닝과 동시에, 별도로:
고객 발화 → Playbook (키워드 매칭) → 리드 점수 산정 → FC API로 설계사 배정

Playbook은 체이닝에 포함되지 않는다. 매 발화마다 별도로 키워드를 확인하는 독립 프로세스이다.

피드백 루프 (Case)

Clark 대화 결과 → Case 카드 자동 태그 → 검수 후 저장 → Clark 진단 품질 개선

DB 테이블 관점 요약

concepts 테이블
  ├── id, term, definition, synonyms[], link_type, visibility, status
  └── linked_risk_types[] (FK → risk_types)

risk_types 테이블
  ├── id, risk_name, description, lead_score_weight
  └── (concepts에서 참조됨, rules에서 참조됨)

rules 테이블
  ├── id, target_risk_type_id (FK → risk_types), conditions[], warning_message, status
  └── linked_evidences[] (FK → evidences, 1개 이상 필수)

evidences 테이블
  ├── id, evidence_type, data_source, usage_method
  └── (rules에서 참조됨)

policies 테이블
  ├── id, regulation_name, checklist[], clark_constraints[]
  └── (체이닝 마지막에서 전역 참조)

playbooks 테이블
  ├── id, conversion_keywords[], scoring_formula, allocation_rules
  └── (체이닝과 별도, 독립 동작)

cases 테이블
  ├── id, conversation_summary, detected_risk_types[], conversion_result
  └── (risk_types 참조)
Part 4
카드 한 장의 일생 — 만들어져서 Clark 답변에 쓰이기까지
"암투병생활비 부족형" Rule 카드 한 장이 태어나서 Clark의 답변에 반영되기까지의 전체 과정이다.
1
카드 생성
KC Admin에서 사람이 직접 작성한다

담당자가 KC Admin에 로그인한다. "Rule 카드" → "새 카드 만들기"를 클릭한다.

담당자가 입력하는 것들

  • 대상 위험 유형: 드롭다운에서 "암투병생활비 부족형" 선택
  • 조건 빌더에서 조건 추가:
    • 조건 1: 소스=마이데이터, 필드=담보코드, 연산자=포함하지 않음, 값=A4200
    • 조건 2: 소스=마이데이터, 필드=담보금액, 연산자=미만, 값=50000000
    • 조건 3: 소스=Amplitude, 필드=나이, 연산자=이상, 값=40
  • 경고 메시지: "현재 암 진단금이 5천만 원 미만으로, 암 발병 시 치료비 공백이 우려됩니다"
  • Evidence 연결: "40대 남성 평균 암 진단금" Evidence 카드를 검색해서 선택

저장하면 상태는 DRAFT(초안)이다.

이 단계에서 개발해야 하는 것
  • 카드 생성·수정·삭제·조회 API
  • 조건 빌더 화면 (마이데이터 7종 + Amplitude 13종 필드를 드롭다운으로 제공)
  • ICIS 담보코드 테이블 조회 기능 (조건에서 담보코드 선택 시 코드 테이블에서 검색)
  • Evidence 연결 화면 + 연결 필수 검증 (0개면 저장은 되지만 검수 요청은 불가)
2
드라이런 검증
"이 조건이 현실적인가" 테스트한다

담당자가 "드라이런" 버튼을 누른다. 미리 등록해둔 테스트용 고객 프로필을 선택한다.

테스트 프로필 예시

45세 남성
  - 담보코드: A4200 보유 (암진단 담보 있음)
  - 담보금액: 3,000만 원
  - 나이: 45세

드라이런 결과 화면

조건 1 (담보코드에 A4200 없음): ✕ 미충족 (보유하고 있으므로)
조건 2 (담보금액 < 5천만 원):   ✓ 충족 (3천만 < 5천만)
조건 3 (나이 >= 40):            ✓ 충족 (45 >= 40)
────────────────────────────
종합: (조건1 또는 조건2) 그리고 조건3 → ✓ 해당됨
→ "암투병생활비 부족형"으로 판정

담당자는 이 결과를 보고 조건이 너무 넓은지(너무 많은 고객이 걸리는지), 너무 좁은지(아무도 안 걸리는지) 확인한다.

이 단계에서 개발해야 하는 것
  • 테스트용 고객 프로필 관리 기능 (생성·수정·삭제, 초기 3~5명)
  • 조건 대조 엔진 — AI가 아니라 일반 if-else 로직이다. Rule의 조건과 고객 데이터를 비교하는 함수이다.
  • 결과 화면 (조건별 충족/미충족 개별 표시)
3
검수 절차
승인을 받는다

Rule 카드의 상태 변화

DRAFT(초안) → Evidence 연결 확인 → IN_REVIEW(검수 중) → ACTIVE(활성)
                                         ↓ 반려 시
                                    DRAFT로 되돌아간다
  1. 담당자가 "검수 요청"을 클릭한다 → 시스템이 Evidence 연결 여부를 자동 확인한다 → 미연결이면 요청을 차단한다
  2. 상태가 IN_REVIEW로 바뀐다 → 검수자에게 알림이 간다
  3. 검수자가 KC Admin에서 카드 내용을 확인한다: 용어가 정확한지, 기준값이 현실적인지, Evidence가 적합한지
  4. 승인하면 → ACTIVE / 반려하면 → DRAFT (반려 사유 필수 기재)

Policy 카드는 단계가 하나 더 있다

DRAFT → IN_REVIEW(검수자 검수) → LEGAL_REVIEW(준법감시인 법률 검토) → ACTIVE
이 단계에서 개발해야 하는 것
  • 카드 상태 관리 (카드 종류마다 상태 변화 규칙이 다르다)
  • 역할별 권한: 담당자(작성), 검수자(검수, 자회사 소속이므로 별도 권한 그룹), 준법감시인(법률 검토)
  • 검수 요청·승인·반려 API + 반려 사유 기록
  • 알림 (검수 요청이 들어왔을 때 검수자에게 알림)
4
벡터 DB 반영
자동 파이프라인이 동작한다

카드 상태가 ACTIVE로 바뀌는 순간, 벡터 DB에 자동으로 반영된다.

벡터로 변환하는 대상

  • 카드 본문 (정의, 경고 메시지 등)
  • 동의어 목록 (Concept 카드의 경우)
  • 연결된 Evidence의 텍스트 (Rule 카드의 경우)
  • 이것들을 합친 텍스트를 벡터로 변환해서 저장한다

기존 RAG와의 차이점

  • 기존: 긴 문서 → 500자씩 자르기 → 벡터 변환. 의미가 중간에 잘릴 수 있다.
  • KC: 카드 1장 = 벡터 1개. 의미 단위로 사람이 직접 작성했으므로 잘릴 일이 없다.

기술연구소가 자체 판단할 영역

  • 어떤 임베딩 모델을 쓸 것인가 (선택지 자유)
  • 벡터 DB 선택 (기존 PostgreSQL에 pgvector 확장을 쓰면 추가 비용 없이 가능)

기획에서 정해둔 것

  • 반영 시점: ACTIVE로 상태가 바뀔 때
  • 카드 수정 후 재승인 시: 기존 벡터 삭제 → 새 벡터 생성
  • 카드 보관 처리(ARCHIVED) 시: 벡터 삭제
이 단계에서 개발해야 하는 것
  • 상태 변경 감지 (상태가 ACTIVE로 바뀌면 자동으로 임베딩 실행)
  • 임베딩 파이프라인 (텍스트 합치기 → 벡터 변환 → 벡터 DB 저장)
  • 벡터 버전 관리 (카드가 수정되면 기존 벡터를 교체한다)

Clark이 고객에게 답변한다
모든 것이 합쳐지는 순간

고객이 Clark 웹뷰에서 "암 보험 괜찮은 거 있어?"라고 질문한다.

일어나는 일 — 순서대로

① 질문 수신
   "암 보험 괜찮은 거 있어?"

② 벡터 검색 — 관련 Concept 카드를 찾는다
   동의어 포함 벡터로 검색 → "암 보장" Concept 카드를 발견한다
   이 카드는 Linked 유형이다 → 연결된 Risk-type "암투병생활비 부족형"으로 이동한다

③ 체이닝 — 관련 카드를 모두 수집한다
   Risk-type: "암투병생활비 부족형"
   → 연결된 Rule 카드들을 수집한다
   → 각 Rule에 연결된 Evidence 카드들을 수집한다
   → Policy 카드를 수집한다 (모든 답변에 공통 적용)

④ 조건 대조 — 고객 데이터와 Rule을 비교한다 (if-else 로직)
   이 고객의 마이데이터와 Amplitude 속성을 Rule 조건과 비교한다
   - 담보금액 = 3,000만 원 < 5,000만 원 → 충족
   - 나이 = 45 >= 40 → 충족
   → "암투병생활비 부족형" 해당 확인

⑤ LLM에 보낼 자료를 조립한다
   [시스템 지시: Clark의 역할과 성격 + Policy 제약 사항]
   [참고 자료: 판정된 Risk-type + Rule 경고 메시지 + Evidence 근거]
   [고객 질문: "암 보험 괜찮은 거 있어?"]

⑥ LLM이 답변을 생성한다
   "현재 암 진단금이 3천만 원으로, 같은 연령대 평균(7,200만 원)에 비해
    부족한 편입니다. 암 치료 시 비용 공백이 생길 수 있어서,
    보장 구조를 한번 점검해보시는 게 좋겠습니다."

⑦ 안전 검증 — 4단계 방어
   1단계: 악의적 입력 차단 (프롬프트 인젝션 방지)
   2단계: 금지 표현 필터 (특정 상품 추천 등)
   3단계: 답변이 체이닝 범위 안에 있는지 확인한다
   4단계: 판단 불가능한 경우 "전문 상담사 연결"을 안내한다

⑧ 답변을 고객에게 전달한다

⑨ 동시에 별도로: Playbook을 확인한다
   "괜찮은 거 있어?" → 상담 의향 신호를 감지한다 → 아직 명확한 상담 요청은 아니다 → 대기
   (나중에 "상담 받고 싶어"라고 하면 설계사 연결이 시작된다)
이 단계에서 개발해야 하는 것
  • 벡터 검색 + 체이닝 로직 (Concept → Risk-type → Rule → Evidence 순서로 수집)
  • 조건 대조 엔진 (마이데이터 + Amplitude 데이터와 Rule 조건을 비교하는 if-else 로직)
  • LLM 자료 조립기 (수집된 카드 내용을 LLM에 보낼 형태로 변환)
  • 4단계 안전 검증 로직
  • Playbook 키워드 매칭 (체이닝과 별도 프로세스)
  • FC API 연동 (설계사 배정 시)
