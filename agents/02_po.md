# Agent 02 — PO (Product Owner)

## 역할 한 줄 정의

프로젝트 전체 범위를 관리하고 에이전트 간 작업을 조율한다. 각 산출물이 기획 의도에 부합하는지 최종 판단하며, 사용자에게 의사결정을 요청하는 창구 역할을 한다.

---

## decisions.md 관리 책임

PO는 `context/decisions.md`의 **관리 주체**다. 세션 중 설계 결정이 확정되는 즉시 업데이트한다.

### 확정 트리거

- 사용자가 명시적 승인("확정", "그렇게 하자", "OK")을 준 시점
- PO가 결정을 선언하고 사용자가 반박 없이 수용한 시점

### 업데이트 절차

1. `decisions.md`에 새 섹션 추가 — 날짜 명시 (예: `2026-06-10 확정`)
2. 기존 내용과 충돌 시 구 항목 삭제 또는 교체 (레거시 방치 금지)
3. 미확정 항목은 `## 미결 항목` 테이블에 먼저 등록 → 확정 시 본문 섹션으로 이동, 폐기 시 행 삭제
4. 영향 받는 에이전트 파일이 있으면 변경 범위를 사용자에게 고지 후 업데이트 요청

### 레거시 방지 원칙

- decisions.md에 없는 설계 결정은 "아직 확정되지 않은 것"으로 간주한다
- 에이전트 파일 본문에서 CLAUDE.md·기획서(`Data/`)를 직접 인용하는 대신 decisions.md 섹션명을 참조한다
- 분기(기획 변경·설계 번복)가 생기면 구 결정 삭제 + 신 결정 등록을 반드시 같이 처리한다

### Phase 1 포함

**카드 5종:**

**Rule 조건 데이터 소스 3종 (2026-06-22 재확정 — 필수/선택 구분):**
- `MYDATA` (마이데이터) — 고객 보험 계약 데이터 (ICIS 마이데이터). 담보코드 지정 후 담보금액·갱신여부 등 조건. **기본: 필수(required=true)**
- `PROMAGE` (프롬에이지) — 메디에이지 프롬에이지 건강분석 데이터. 암위험도·질병위험도·생체나이·의료비예측 카테고리. 등급 조건(위험/주의/양호) 또는 수치 조건. **기본: 선택(required=false) — 미연동 사용자 대응**
- `PROFILE` — 사용자 기본정보. 나이·성별·결혼·자녀·직업위험도·출산예정·거주지. 동일 담보 기준값을 연령·성별별로 세분화할 때 사용. **기본: 필수(required=true)**

**Evidence 단일 유형 (2026-06-17 재확정):**
- 공인 외부 통계 기반 — HIRA(심평원)·NHIS(건보공단)·KOSTAT(통계청)·FSS(금감원) 등 공인 기관 통계를 운영자가 직접 입력
- 마이데이터·프롬에이지는 상담 시 자동 전달 → Evidence 카드 불필요

## 2026-06-22 확정 사항 (누적)

### [1] Concept 카드 — Risk-type 연결 필수
- Concept → Risk-type 연결: **필수(required)**. 연결 없으면 KC 체인 미진입.
- Concept에 Standalone 기능 없음 — 완전 제거됨 (구 기획 흔적 삭제 대상).
- Concept 미매칭 시: Case 3(RAG→Fallback) 또는 Case 4(Playbook 감지) 분기.

### [2] Case 1~4 — KC매칭 × Playbook감지 매트릭스
- Case 1: KC Concept 매칭O + Playbook 미감지 → KC 구조화 답변
- Case 2: KC Concept 매칭O + Playbook 감지O → KC 구조화 + CTA 버튼
- Case 3: KC 미매칭 + Playbook 미감지 → RAG(약관→FAQ) → Fallback 생성형
- Case 4: KC 미매칭 + Playbook 감지O → Standalone 가이드 주입 + CTA 버튼
- Playbook이 Case 4로 모든 미매칭 시나리오를 커버 → Concept Standalone 제거 근거.

### [3] Risk-type 우선순위 — 가중치 완전 제거
- 1순위: 중요도 서열 (높음 > 보통 > 낮음)
- 2순위: 선택 조건 충족 개수 (시스템 자동)
- 3순위: 카드 코드 오름차순
- **가중치(×3/×2/×1) 완전 제거** — 코드/문서에서 삭제. 높음/보통/낮음 서열만 사용.

### [4] Rule 조건 빌더 — 필수/선택 구분
- 소스 3종: MYDATA / PROMAGE / PROFILE (Amplitude 없음)
- MYDATA 기본: required=true (필수)
- PROMAGE 기본: required=false (선택 — 미연동 사용자 대응)
- PROFILE 기본: required=true (필수)
- 각 조건 행(ConditionRow)에 required boolean 속성 있음
- **약관 DB 연동(useContractDb):** Rule 카드의 선택 옵션. 활성 시 담보코드 기준 약관 자동 조회 → 면책조항 자동 포함. Policy 카드와 다른 개념.

### [5] Policy 카드 — 필드 2개만
- **확정 필드:** Policy 이름(name) + Clark 앱 표시 문구(appDisplayText)
- **삭제 필드(코드/문서에서 완전 제거):** 규제 문서명 / 적용 범위 / 핵심 조항 요약 / 출력 대상 화면 / 출력 제한 설정 / 준수 체크리스트
- Clark 앱 표시 문구 비어있으면 검수 요청 불가
- 승인 2단계: 도메인 검수자 → 준법감시인
- 목적: 운영자가 Clark 앱 표시 면책 고지 문구를 직접 작성

### [6] Playbook — Standalone 선택사항
- Standalone 답변 가이드: 선택사항. 비워두면 Clark 기본 안내 문구 사용.
- "최소 20자 필수" 표현 완전 삭제.
- 최소 키워드: 3개 (필수)
- CTA 기본 버튼(consult) 필수
- approved 상태에서 캔버스 연결 없이 직접 "라이브 전환" 가능

### [7] RAG 아키텍처
- 약관 RAG: 자동 파이프라인 (크롤러 자동, 운영자 관리 불가)
- Clark 서비스 FAQ RAG: 운영자 직접 등록 → 즉시 인덱스 등록 (19_faq-rag.html, 승인 프로세스 없음)
- Fallback: 18_system-settings.html 설정 적용 (경쟁사/외부 서비스 언급 금지 등)
- KC 체인 매칭 실패 시 순서: RAG → Fallback

### [8] 파일 참조 기준

- Playbook Standalone: 14_answer-logic-guide.html

## UI 품질 기준: 템플릿 우선 원칙

**데이터 항목이 달라도 같은 목적의 기능은 반드시 같은 UI 구조로 구현한다.**

복수의 화면에 걸쳐 같은 역할을 하는 UI가 필요할 때, PO는 구현 시작 전 UI 디자이너(03)에게 **템플릿 정의를 먼저 완료**하도록 지시한다. 개별 화면 담당 코더가 각자 다른 방식으로 구현하면 운영자는 화면마다 다른 위치에서 같은 동작을 찾아야 한다.

**판단 기준:**
- 2개 이상의 화면에서 같은 목적의 UI가 필요하다 → 템플릿 정의 후 구현
- 신규 화면 요청 시: "기존 템플릿으로 해결되는가?"를 먼저 확인한 후 에이전트에 전달
- 기존 구현체가 템플릿과 다를 경우 → UI 검수자(08) 체크리스트에 포함시켜 수정 요청

## Clark AI 답변 케이스 (Phase 1 핵심 개념)

KC 메인 체인과 Playbook 체인은 **동시에 독립 분석**한다. 두 결과의 조합으로 4가지 답변 케이스가 결정된다.

| 케이스 | KC Concept 매칭 | Playbook 키워드 감지 | Clark 출력 |
|---|---|---|---|
| **Case 1** | ✓ 매칭 | — 미감지 | KC 구조화 답변 (Concept→Risk-type→Rule→Evidence→Policy 체인) |
| **Case 2** | ✓ 매칭 | ✓ 감지 | KC 구조화 답변 + CTA 버튼 |
| **Case 3** | — 미매칭 | — 미감지 | RAG(약관→FAQ) → Fallback 생성형 답변 (18_system-settings 설정 제한 규칙 LLM 주입) |
| **Case 4** | — 미매칭 | ✓ 감지 | Standalone 가이드 주입 + LLM 생성형 답변 + CTA 버튼 |

### KC 메인 체인 (Case 1·2 발동 시)

```
사용자 질문 입력
      ↓
① 의미 매칭
   Concept 카드 synonyms와 질문 키워드 비교
      ↓
   KC 매칭됨? (동시에 Playbook 체인도 독립 분석)
   ├── YES → ② Risk-type 판정
   │          Rule 카드 조건 평가
   │          (MYDATA + Promage + 프로파일 3-source 조합)
   │              ↓
   │          Rule 조건 충족?
   │          ├── YES + Playbook 감지 → Case 2: KC 기반 답변 + CTA 버튼
   │          ├── YES               → Case 1: KC 기반 답변
   │          └── NO                → Case 3 또는 Case 4
   └── NO  → Playbook 감지 여부에 따라 Case 3 또는 Case 4
```

### Playbook 병렬 체인 (Case 2·4 발동 시)

```
사용자 발화 → 발화 키워드 감지 (KC 메인 체인과 독립 동작)
              ↓
         Playbook 매칭됨?
         ├── YES + KC 매칭  → Case 2: KC 답변 + CTA 버튼
         └── YES + KC 미매칭 → Case 4: Standalone 가이드 → LLM 답변 + CTA 버튼
```

**Case 4 구현 포인트:** Playbook의 "Standalone 답변 가이드" 필드를 LLM 프롬프트에 주입 → 자연어 답변 생성 후 CTA 버튼 노출. 버튼만 단독 노출 시 CTA 클릭률이 낮으므로 필수. Standalone 답변 가이드는 선택사항 — 비워두면 Clark 기본 안내 문구 사용 (최소 글자 수 제한 없음).

**KC 기반 답변 조건 (Case 1·2):**
- 질문 키워드가 Concept synonyms에 매칭
- Concept이 Risk-type에 연결되어 있어야 KC 체인 진입 가능 (필수)
- 해당 Risk-type에 연결된 라이브 Rule 존재
- Rule 조건 평가 가능 (MYDATA / Promage / 프로파일 중 하나 이상)

**상세 시나리오:** `13_answer-logic.html` (개발자 스펙) · `14_answer-logic-guide.html` (기획자 시나리오 — Case 2 + Case 4) 참조

**v2 워크플로우: 카드 내용은 편집기, 연결은 캔버스에서 분리 처리**

```
[카드 등록 — 내용 입력]
각 카드 유형별로 (순서 무관, 독립 등록 가능):
  → 카드 편집기에서 내용 입력

[카드 연결 — 캔버스에서 처리]
  → 카드 캔버스(00_canvas-main.html) 접근
  → 피커 패널(상단 5컬럼)에서 포컬 카드 클릭
  → 그리드(하단) "연결 가능" 섹션에서 "연결 추가" 버튼 클릭 → 확인 모달

  CONNECT_RULES 연결 방향 (최종):
  concept  → risk-type  (필수)
  risk     → rule       (필수, 최소 1개)
  rule     → evidence   (필수, 최소 1개)
  rule     → policy     (선택)
  evidence → (단말)
  policy   → (단말)
  playbook → (단말, 독립 체인)

  캔버스 표시 범위: active + approved + review 카드 (draft 제외)
  연결 가능 범위: active + approved만 (review = 승인요청은 연결 불가)
  approved 카드 간 연결 → 준비 연결(pending edge, 점선 파랑), KC Engine 미반영
  active 카드 간 연결 → active edge(실선 초록), KC Engine 즉시 반영
  편집기: 연결 상태 읽기 전용 배지 + "캔버스에서 변경" 버튼만 표시

[검증]
  → 사전 테스트 (드라이런: 11_dry-run.html)
  → 카드 캔버스 연결 확인 (연결됨·연결가능·해당없음 그리드)

[라이브 전환]
  → "배포 요청": pending edge → active edge, KC Engine 즉시 반영
  → 라이브 전환은 사전 테스트(④) 통과 후 캔버스에서 처리 (검수·승인 화면에서 처리하지 않음)
```

## 카드 유형별 역할 (등록 순서 자유 — 캔버스에서 연결)

v2부터 카드 등록 순서 의존성 없음. 각 카드를 독립 등록·승인 후 캔버스에서 연결.

| 카드 | AI 답변 체인 위치 | 비고 |
|---|---|---|
| Concept | 체인 입구 — 동의어 매칭 | Risk-type 연결 필수. 연결 없으면 KC 체인 미진입. Standalone 기능 없음 |
| Risk-type | Concept → **Risk** → Rule | 공개범위: 고객 공개 |
| Rule | Risk → **Rule** → Evidence·Policy | 3-source 조건 (MYDATA·Promage·프로파일) |
| Evidence | Rule → **Evidence** | 단일 유형: 공인 외부 통계 (직접 입력) |
| Policy | Rule → **Policy** | 준법감시인 2단계 승인 필수. 필드 2개만: name + appDisplayText |
| Playbook | 병렬 독립 체인 | 발화 키워드 감지 → CTA. Standalone 답변 가이드는 선택사항 |

### 카피라이팅 원칙 (운영자 눈높이)

모든 UI 텍스트는 비기술 운영자가 바로 이해할 수 있는 언어로 작성한다. 금지/대체 표현 전체 기준은 `guides/copywriting.md` 참조.

**핵심 금지 표현:** 런타임에 / 파라미터 / 라우팅 / 매핑 / 비활성(→잠김/잠금 처리/연결 불가) / 활성화됩니다 / 임계값(→기준값/판단 기준) / 시뮬레이션(→사전 테스트) / 상/중/하 / 3-source (화면·문서 노출 금지)
**화면 표기:** MYDATA→마이데이터 / Promage→프롬에이지 (API 코드는 각각 MYDATA·PROMAGE 유지)
**공개범위:** 한국어 전용 — 고객 공개 / 공통 기준 / 내부 전용 (영문 Public/Baseline/Internal 금지). **편집기에는 미표시** — 카드 유형으로 자동 고정, 목록 화면에서만 읽기 전용 배지 표시 (select 사용 금지)
