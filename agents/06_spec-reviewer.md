## 검증 관점 및 체크리스트

### A. 카드 편집기 — 필드 정합성

각 카드 편집기의 입력 필드가 **`context/card-types.md`**(단일 진실원 — 기획서와 충돌 시 이 파일이 우선한다, `context/decisions.md` 헤더 원칙)와 일치하는지 확인한다(2026-07-07 정리 — 이전엔 필드 목록 전체가 여기 사본으로 있어 card-types.md가 갱신될 때마다 드리프트 위험이 있었음). 카드 유형: Concept·Risk-type·Rule·Evidence·Policy·Playbook 6종 + Case(Phase 1.5 이후, 목업 범위 외).

### A-1. Case 0~4 매트릭스 — 조회형 게이트 + KC매칭 × Playbook감지 (2026-07-06 Case 0 추가)

**Case 0(데이터 조회형)이 Concept 매칭보다 먼저 실행되는 선행 게이트다.** 감지되면 아래 4케이스 매트릭스 자체가 실행되지 않고 마이데이터 직접 조회로 종결된다(콘텐츠 매칭 없음). **Playbook 매칭은 Case 0 여부와 무관하게 항상 별도 병행 실행되며, 감지되면 조회 응답에도 CTA가 가산된다** — 검수 시 "Case 0인데 CTA가 없다"고 오판하지 않도록 주의. 상세 정의·1차 지원 조회 유형은 `context/answer-logic.md` "Case 0 — 데이터 조회형" 참조.

Case 0 미감지 시에만, KC 체인 매칭 여부와 Playbook 감지 여부의 조합으로 4가지 케이스가 결정된다. Case 1~4는 마이데이터 연동 상태(③④) 전용. 구현된 화면(특히 13_answer-logic.html)이 이 매트릭스를 정확히 반영하는지 확인한다.

| 케이스 | KC Concept 매칭 | Playbook 감지 | 처리 방식 |
|---|---|---|---|
| Case 1 | O | X | KC 구조화 답변 |
| Case 2 | O | O | KC 구조화 + CTA 버튼 |
| Case 3 | X | X | RAG(약관→FAQ) → Fallback 생성형 |
| Case 4 | X | O | Standalone 가이드 주입 + CTA 버튼 |

**핵심 제약:**
- Concept에 Standalone 기능 없음 — 구 기획 흔적(Concept Standalone 표현) 있으면 오류
- Playbook이 Case 4로 KC 미매칭 시나리오 전체를 커버
- Concept 미매칭 시 반드시 Case 3 또는 Case 4로 분기
- Case 0은 Rule 카드로 구현하지 않는다 — MYDATA Rule 조건은 담보금액(coverage_amt) 1종만 허용하므로 갱신·만료·상품유형 조회는 Rule 체계 밖의 별도 로직이어야 한다
