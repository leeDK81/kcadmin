## 사용자 데이터 상태 — 선행 분기 (2026-06-26 확정)

> **Clark과 KC Engine은 별개 서비스·별개 법인이다.**
> KC Engine은 Clark의 회원 DB에 접근할 수 없다. Clark은 가용한 데이터만 전달하고, KC Engine은 null 여부로 상태를 판단한다. 별도 `user_state` 파라미터 불필요.

### Clark → KC Engine API 호출 구조

```json
{
  "query": "암진단금 얼마야",
  "profile": { ... } or null,   // null = 미가입 또는 미로그인
  "mydata":   { ... } or null,  // null = 마이데이터 미연동
  "promage":  { ... } or null   // null = 건강정보 미연동
}
```

KC Engine은 각 필드의 null 여부로 두 가지를 결정한다:
1. **Rule 평가 범위** — null인 데이터 소스의 조건은 평가 생략
2. **Playbook 감지 조건 대조** — 카드에 설정된 데이터 미보유 조건과 비교

### Clark → KC Engine 전달 데이터 (상태별)

| 사용자 상태 | PROFILE | MYDATA | PROMAGE | 약관 RAG | 답변 우선순위 | Playbook |
|---|---|---|---|---|---|---|
| ① 미가입/미로그인 | null | null | null | 불가 | FAQ RAG → LLM | PROFILE 없음 → 회원가입 CTA |
| ② 회원 + 마이데이터·프롬에이지 미연동 | 있음 | null | null | 불가 | KC(PROFILE만) → FAQ RAG → LLM | 마이데이터없음 → 마이데이터 연동 CTA |
| ③ 마이데이터 연동 + 프롬에이지 미이용 | 있음 | 있음 | null | 가능 | KC(PROMAGE 스킵) → 약관 RAG → FAQ RAG → LLM | 프롬에이지없음 → 프롬에이지 연동 CTA |
| ④ 메인 서비스 이용 (전체 데이터) | 있음 | 있음 | 있음 | 가능 | KC(전체 평가) → 약관 RAG → FAQ RAG → LLM | 일반 키워드 → 리드 전환 CTA |

**약관 RAG 가능 조건**: 마이데이터에서 가입 상품 목록을 알아야 어떤 약관을 검색할지 특정 가능. ①② 상태에서는 불가. ③④ 가능.
**답변 우선순위 공통 원칙**: 앞 단계 성공 시 뒤 단계 호출 없음(분기). Playbook은 어느 단계든 키워드 감지 시 CTA 가산.

*주1 — 상태 ② Rule 평가 방식:* KC Engine은 모든 Rule을 대상으로 평가를 시도한다. 단, 각 Rule의 조건별로:
- MYDATA·PROMAGE 조건이 **필수**인 경우 → 데이터 null = 조건 불충족 → Rule 미통과
- MYDATA·PROMAGE 조건이 **선택**인 경우 → 해당 조건 스킵, 나머지 조건으로만 평가
- 결과적으로 PROFILE 조건(나이/성별/결혼여부/자녀유무/운전여부)만으로 구성되거나, MYDATA·PROMAGE를 선택으로만 쓴 Rule만 통과 가능.

### 상태별 답변 흐름

```
[상태 ① — profile: null]
KC 체인 평가 전체 불가 (전달 데이터 없음)
  └─ FAQ RAG → 결과 없으면 → Fallback LLM
     + Playbook 병렬: "PROFILE 없음" 조건 Playbook 감지 → 해당 CTA

[상태 ② — mydata: null]
KC 체인 → PROFILE 조건 Rule만 평가
  ├─ PROFILE Rule 통과 → KC 구조화 답변 + 마이데이터 연동 유도 문구
  └─ Rule 미통과 → FAQ RAG → 결과 없으면 → Fallback LLM
     (모든 ② 답변에 마이데이터 연동 유도 문구 공통 포함)
  + Playbook 병렬: "마이데이터없음" 조건 Playbook 감지 → 해당 CTA

[상태 ③ — promage: null, mydata 있음]
KC 체인 → PROFILE + MYDATA 조건 평가 (PROMAGE 선택 조건은 스킵)
RAG 순서: 약관 RAG → FAQ RAG → Fallback LLM
→ Case 1~4 매트릭스 적용 (하단 참조)
+ Playbook 병렬: "프롬에이지없음" 조건 Playbook or 일반 감지 → 해당 CTA

[상태 ④ — 전체 데이터 있음]
KC 체인 전체 평가 (PROFILE + MYDATA + PROMAGE)
RAG 순서: 약관 RAG → FAQ RAG → Fallback LLM
→ Case 1~4 매트릭스 적용 (하단 참조)
+ Playbook 병렬: 일반 키워드 감지 → Playbook 카드 설정 CTA (리드 전환)
```

### Playbook 데이터 미보유 감지 조건

Playbook 카드 편집기에서 아래 조건 중 하나를 선택적으로 설정할 수 있다(라디오 버튼 — 카드당 최대 1개, 2026-07-20 체크박스→라디오 전환). 선택된 조건에 해당하는 사용자에게만 이 Playbook이 감지된다. 아무 조건도 선택하지 않으면 데이터 보유 여부에 관계없이 키워드 발화만으로 감지된다.

| 선택 조건 | 감지 대상(필드 null 여부) | 활용 예 |
|---|---|---|
| PROFILE 없음 | profile == null | 회원가입 유도 CTA |
| 마이데이터없음 | mydata == null | 마이데이터 연결 유도 CTA |
| 프롬에이지없음 | promage == null | 프롬에이지 연동 유도 CTA |
| (없음) | 상관없음 | 일반 리드 전환 CTA |

동일 키워드에 대해 데이터 상태별로 별도 Playbook 카드를 등록하면 상태에 맞는 CTA를 각각 노출할 수 있다.

**우선순위 규칙(2026-07-20 확정) — 조건 간 상호배타 처리:** 사용자 상태 4종은 깔때기 구조라 하위 필드가 null이면 상위 필드도 함께 null이다(예: 상태① PROFILE null인 사용자는 마이데이터·프롬에이지도 당연히 null). 그래서 위 세 조건은 필드 자체만 보면 서로 겹친다 — 조건 정의를 바꾸는 대신, **여러 데이터-미보유 조건 Playbook이 동시에 매칭되면 PROFILE없음 > 마이데이터없음 > 프롬에이지없음 순으로 가장 앞단(더 먼저 해결해야 하는) 조건 카드 하나만 CTA로 남기고 나머지는 억제한다.** `context/matching-policy.md`의 "유사도 최고점 1개만 노출" 규칙보다 이 우선순위가 먼저 적용된다 — 유사도 비교는 이 우선순위로 걸러진 후보군 안에서만 수행된다.

| 사용자 상태 | 동시에 조건이 true인 Playbook | 실제 노출 |
|---|---|---|
| ① PROFILE null | PROFILE없음 + 마이데이터없음 + 프롬에이지없음 | PROFILE없음만 |
| ② 마이데이터 null (PROFILE 있음) | 마이데이터없음 + 프롬에이지없음 | 마이데이터없음만 |
| ③ 프롬에이지 null (PROFILE·마이데이터 있음) | 프롬에이지없음 | 프롬에이지없음 |
| ④ 전체 있음 | 없음(조건 미설정 카드만 대상) | 일반 리드 전환 |

## Case 0 — 데이터 조회형 (신규, 2026-07-06 확정, 2026-07-06 Playbook 병행 원칙 정정)

> **Concept 매칭보다 먼저 실행되는 선행 게이트.** "위험을 진단해달라"가 아니라 "내가 가진 데이터를 그대로 보여달라"는 요청(예: "내가 가진 보험 리스트 알려줘", "나 운전자보험 가입되어 있어?", "나 갱신되는 보험 있어?", "만료된 보험 몇 개야?")은 Concept(위험 진단)·FAQ RAG(일반 판단 노하우)·LLM Fallback(개념 설명) 어디에도 해당하지 않는다 — 셋 다 **저작된 콘텐츠를 매칭**하는 구조인데, 이 요청은 콘텐츠 없이 **마이데이터 원본 값을 그대로 조회·집계**하면 끝나기 때문이다.

> **⚠️ Playbook은 이 게이트의 적용 대상이 아니다(2026-07-06 정정).** `context/matching-policy.md`의 원래 원칙대로 Playbook은 Concept·Case 0 판별 결과와 **완전히 무관하게 항상 독립적으로 병행 매칭**된다. Case 0으로 판별돼 마이데이터를 직접 조회해 답하는 중에도, 그 발화가 어떤 Playbook의 발화 키워드와 겹치면(예: "운전자보험 가입되어 있어?" ↔ PB08 "운전자보험 필요한가요" 계열) CTA가 정상적으로 가산 노출된다. 조회 응답이라고 CTA를 막을 이유가 없다 — 오히려 "지금 이 얘기를 하고 있다"는 신호이므로 리드 전환 기회다.

### 왜 Concept·FAQ RAG·LLM Fallback 세 경로로는 처리할 수 없는가

- **Concept/KC 체인**: Risk-type Rule의 MYDATA 조건은 **담보금액(coverage_amt) 1종만 허용**하도록 의도적으로 제한돼 있다(`context/card-types.md` "MYDATA Rule 조건" 참조). 갱신여부·상품유형·시작일·종료일은 애초에 조건 필드로 선택조차 안 된다 — 위험 갭 진단용으로 좁게 설계된 것이라, "갱신 예정이 있는지" 같은 조회는 Rule 카드로 구현이 불가능하다.
- **FAQ RAG**: 계약과 무관한 일반 판단 노하우·서비스 이용 안내용이다. "내 계약이 어떻게 돼 있는지"는 노하우가 아니라 사실 조회다.
- **LLM Fallback**: `18_system-settings.html` 허용 범위가 "일반적인 보험 개념 설명(상품 추천 제외)"으로 한정돼 있다 — 사용자 본인의 실제 마이데이터 조회 결과를 반환하는 건 이 범위 밖이다.

### 실행 순서

```
[사용자 발화 입력]
        │
        ├──────────────────────────────┐
        ▼                              ▼
[Case 0 의도 분류: 조회형 발화 감지]      [Playbook 벡터 검색] ← Case 0과 무관하게 항상 병행
        │                              (동시 병렬 실행, 감지 시 CTA 후보 확보)
   ┌────┴────┐                          │
  감지O        감지X                     │
   │            │                       │
   ▼            ▼                       │
[데이터 상태     [Concept 매칭            │
 확인 후 처리]    → Case 1~4]            │
   │            │                       │
   └──────┬─────┘                       │
          ▼                             │
   [핵심 답변(조회 결과 또는 진단/RAG/Fallback)] ──── CTA 있으면 가산 ────┘
```

**감지 방식**: Concept·Playbook과 동일한 임베딩 유사도 엔진을 재사용하되, "조회형 예시 발화" 전용 인덱스를 **Concept 인덱스와 완전히 분리**해 구성하고 Concept 매칭보다 먼저 시도한다(2026-07-07 명확화 — "암보험 가입되어 있어?"(조회) vs "암보험 괜찮나요?"(진단)처럼 표현이 겹치는 문장이 많아 같은 인덱스를 쓰면 서로 오염되기 때문). "~있어?", "~돼 있어?", "몇 개야?", "~알려줘" 같은 어미 + 보유·가입·갱신·만료 등 조회성 키워드 조합이 특징. Playbook 벡터 검색은 이 판별과 별개로 항상 동시 실행된다.

**기준값·우선순위 (2026-07-07 신규 확정):**
- Case 0 기준값은 Concept 기준값보다 **보수적으로(더 높게)** 설정한다. 오탐 손실이 비대칭적이기 때문이다 — 진단 의도인데 Case 0으로 잘못 분류되면 사용자가 KC 위험 진단(핵심 서비스 가치)을 못 받고 단순 조회 답만 받아 손해가 크다. 반대로 Case 0을 놓쳐 Concept으로 흘러가도 최악의 경우 "공백 없음" 안심 메시지 정도로 끝나 손해가 작다.
- Case 0과 Concept이 둘 다 기준값을 넘겼을 때: **Case 0 유사도가 Concept 유사도보다 일정 마진 이상 높을 때만 Case 0을 채택**하고, 마진 이내면 Concept(진단)을 우선한다. 즉 애매하면 항상 진단 쪽으로 보낸다. 마진 실제 수치는 실사용 로그 확보 후 개발팀이 튜닝한다.

**예시 발화·상품유형 유사어 등록 방식 (2026-07-07 확정, 2026-07-07 화면 분리):** "조회형 예시 발화"(유형 4종)와 "상품유형 유사어 연결표"(예: 실비보험=실손의료보험)는 **둘 다 BE 하드코딩이 아니라 KC Admin에서 운영자가 직접 등록·수정**한다. 실사용 로그를 보며 계속 튜닝해야 하는 대상이라 개발 배포 없이 바로 반영돼야 하기 때문이다. 단, Concept처럼 승인 워크플로우가 있는 정식 카드 타입으로 만들 필요는 없다 — FAQ RAG의 "Q 필드에 발화 예시 여러 개 등록"과 동일한 수준의 단순한 리스트/표 구조로 충분하다. 두 데이터는 화면(사이드바 메뉴)도 별개다 — `20_query-messages.html`(조회형 메시지 관리, 카드 라이브러리 상단) / `21_synonym-management.html`(유사어 관리, 시스템 설정 하위).
- **조회형 예시 발화**: 유형 4종(전체목록/특정상품/갱신예정/만료건수)별로 예시 문장 리스트를 운영자가 등록·수정.
- **상품유형 유사어 연결표**: 사용자 표현(유사어) → 마이데이터 표준 상품유형코드 1:1 연결. **유사도 매칭이 아니라 결정론적 연결**로 처리한다 — 이는 사실 조회이므로 정확해야 하며, 위험 진단(Concept)처럼 "비슷하면 다 잡는" 방식을 쓰면 엉뚱한 상품 정보를 사실인 것처럼 알려주는 사고 위험이 있다. **등록 대상은 완전히 같은 상품의 다른 이름일 때만이며, 이름이 비슷해도 실제로 다른 상품(예: 자동차보험≠운전자보험)은 등록 금지** — 운영자가 자유롭게 등록하는 화면이므로 이 원칙을 화면 안내문에 명시한다(승인 절차는 두지 않기로 확정 — Concept 수준 검수 게이트는 과함).
- (2026-07-07 확인) 마이데이터 API는 갱신일·만료일·상품유형·가입일·상품명 필드를 모두 제공한다 — 이 전제 위에서 아래 조회 유형 4종을 설계한다.

### 데이터 상태별 처리 (위 "사용자 데이터 상태 — 선행 분기" ①~④와 연동)

| 상태 | 처리 |
|---|---|
| ① profile: null | 조회 불가 — "로그인/회원가입 후 확인할 수 있어요" 안내 (PROFILE 없음 Playbook과 동일 톤 재사용) + **로그인/회원가입 버튼 항상 함께 노출(2026-07-07, 아래 "CTA 결합" 3번 참조)** |

## 매칭 시 답변 처리 — 카드 우선 UI로 통일 (2026-07-05 확정, 2026-07-05 프레이밍 정정, 2026-07-05 N=1도 카드로 통일)

> Case 1·2("KC 구조화 답변")는 원래 매칭된 Risk-type이 **1개**일 때의 포맷이었다. `card-policy.md`의 Concept↔Risk-type **N:N** 카디널리티는 **2026-06-21부터 이미 확정된 아키텍처**다("하나의 Concept이 여러 Risk-type을 동시에 트리거 가능") — 새로 설계해야 할 능력이 아니다. 다만 지금까지는 "실제로 N개가 매칭되면 답변을 어떻게 보여줄지"에 대한 정의가 없었고, 아래가 그 정의다.

**중요 — 실제 컨텐츠 현황 확인 (2026-07-05):** `contents/07_connections/chain-map.json`을 코드로 확인한 결과, 현재 등록된 Concept 24개는 **전부 Risk-type 1개에만 연결**돼 있다(N:N 능력이 아직 한 번도 실사용되지 않음). 즉 이 섹션이 다루는 "N개 매칭"은 **새 기능을 제안하는 게 아니라, 이미 있는 아키텍처를 컨텐츠가 아직 충분히 활용하지 못하고 있는 컨텐츠 갭**을 메우는 작업이다.

### N=1이어도 항상 카드 우선 (2026-07-05 확정 — UI 일관성 + 개발 단순성)

처음에는 "N≥2일 때만 요약 카드, N=1이면 바로 심층 답변"으로 설계했었다. **정정:** N에 따라 응답 형태가 갈리면 개발 쪽에서 분기 로직이 늘어나 복잡해진다. 그래서 **N과 무관하게 항상 결과 카드 방식으로 통일**한다 — Concept이 매칭되면 몇 개의 Risk-type이 걸리든(1개든 4개든 23개든) 동일하게 "말풍선 + 결과 카드(1장 이상) + 카드 탭 → 심층 답변" 흐름을 따른다. N=1일 때 카드 1장을 굳이 거치는 게 사용자 입장에서는 탭 한 번이 더 필요한 사소한 비용이지만, 개발 입장에서는 "N개수에 따른 조건 분기"가 아예 없어져 구현이 훨씬 단순해진다 — 이 트레이드오프를 의도적으로 선택한다.

### 발생 경로 — 전부 같은 처리, N 규모만 다름

| 경로 | 예시 | 매칭 개수(N) 규모 |
|---|---|---|
| **단일 매칭** | "실비랑 사망보장 둘 다 없나요" 류 단독 주제 Concept | 1개 |
| **토픽형 다건 매칭** | "암보험 괜찮나요?" → 암보험 Concept이 T02·T18·T25·T29 전부에 연결 | 보통 2~5개 |
| **종합진단 트리거** | "내 보험 전체 진단해줘" → risk_type_refs에 활성 23개 전체를 연결한 Concept 카드 1장 | 최대 23개(활성 Risk-type 전체) |

이 셋은 사용자 의도는 다르지만 **답변 처리 메커니즘은 완전히 동일**하다 — "매칭된 Risk-type이 N개일 때 카드로 보여주고 탭하면 심층 답변"이라는 하나의 로직이며, 차이는 N의 규모뿐이다. 별도 엔진이나 새 카드 타입을 만들지 않는다. "종합진단"도 예외적인 신규 카드 타입이 아니라, 동일한 N:N 메커니즘을 규모만 크게(23개) 적용한 것뿐이다.

- **2026-07-05 실제 저작 완료:** `contents/04_concept/concepts.md`·`contents/07_connections/chain-map.json`에 5개 그룹 Concept 등록 완료 — `CN-T02A`(암보험, 기존 확장, T02·T18·T25·T29), `CN-HOSPALL`(입원비보험, 신규, T19·T24·T26·T30·T32), `CN-INJURYALL`(상해보험 종합, 신규, T20·T22·T23), `CN-CAREALL`(간병보험 종합, 신규, T05·T27), `CN-ALL`(종합진단, 신규, risk_type_refs=활성 23개 전체). Concept 총 24→28개. T코드 변경 시 이 5개 카드도 함께 갱신 필요(`contents/agents/04_content-po.md` 체인 정합성 검토 항목에 추가됨).

### UI 구현 방식 — 채팅 버블 안 텍스트 나열이 아니라 탭 가능한 결과 카드

Clark은 모바일 웹 채팅 앱이므로, N개 항목을 말풍선 하나에 텍스트로 욱여넣는 것은 실제 구현 가능한 UI가 아니다. 실제 채팅 UI에서 흔히 쓰이는 **"메시지 + 리스트 템플릿(첨부 카드)" 패턴**을 N=1 포함 모든 경우에 동일하게 따른다:

1. **말풍선**: 한 줄 총평(N=1이면 "고객님은 [위험유형명]에 해당하는 것 같아요" 정도로 축약 가능) + "아래 항목을 눌러보시면 자세히 안내해드려요" 짧은 안내만. Evidence 인용·상세 근거는 넣지 않는다.
2. **결과 카드(말풍선 밖, 별도 UI 블록)**: 항목별로 탭 가능한 카드 1장씩 — 위험유형명 + 매우 짧은 공백 요약 1줄 + 화살표(탭 유도). N=1이어도 카드 1장으로 동일하게 렌더링. Rule의 clarkMessage 요약 수준이며 진단 선언·근거 문장은 넣지 않는다.
3. **도메인 그룹핑 없음 (2026-07-05 확정):** 건강보험/생명보험/상해/특종 그룹 타이틀을 넣지 않는다. N이 아무리 커도(종합진단 최대 23개) 카드를 평평하게(flat) 나열한다 — UI 일관성을 위해 규모에 따른 표시 방식 분기를 만들지 않는다.
4. **정렬 기준**: 기존 "다중 Risk-type 감지 우선순위"(1순위 중요도, 2순위 선택조건 충족 개수, 3순위 최근 배포순, `card-types.md` 참조)를 그대로 재사용한다 — 용도는 "카드를 어떤 순서로 나열할지". 내부 "중요도" 라벨 텍스트 자체는 카드에 노출하지 않는다
5. **PROMAGE 선택조건 매칭 표시**: "Rule 선택 조건(PROMAGE) — 매칭 시 답변 반영 원칙"의 전체 문장을 카드마다 넣지 않고, 해당 카드에 짧은 배지(예: "⚠️건강정보 매칭")만 표시. 전체 설명은 카드를 탭했을 때(2단계 상세)에서 노출
6. **CTA**: 카드 리스트 아래에 기존과 동일한 CTA 버튼

실제 UI 예시는 `contents/html/08_ai-preview.html` 참조 — 2026-07-05 페이지 전체를 이 카드 우선 패턴으로 재구성 완료. "N:N 다건 매칭 Concept" 섹션(CN-T02A·CN-HOSPALL·CN-INJURYALL·CN-CAREALL·CN-ALL 5개)과 "단독 매칭 Concept" 섹션(Risk-type 1:1, 23개, N=1도 카드 우선) 모두 `.app-result-cards`/`.app-result-card` 컴포넌트로 통일.

### 상세형 포맷 — 카드 탭 다음 턴

사용자가 결과 카드 하나를 탭하면(또는 "암진단금 부족형 자세히 알려줘"처럼 직접 지목하면), 그 Risk-type 하나에 대해 **기존 Case 1·2 심층 포맷을 그대로 사용**한다(진단 선언 + 이유 + Evidence 출처·연도 인용 + PROMAGE 선택조건 매칭 시 경각심 문장 + 면책 + CTA). 새 포맷을 만들지 않는다 — `contents/html/08_ai-preview.html`의 "단독 매칭 Concept" 23개 체인 답변이 이 상세형의 실제 예시다.

**같은 채팅 스레드 유지 — 새 페이지 열지 않음 (2026-07-05 확정):** Clark은 모바일 웹 "채팅" 앱이다. 카드 탭은 **사용자가 그 항목명을 타이핑한 것과 동일하게 처리**되며, 새 페이지·새 화면으로 이동하지 않고 **같은 채팅창 안에 새 말풍선으로 이어진다**(요약 카드 화면은 위로 스크롤되며 그대로 남아있음). 카카오톡 플러스친구·은행 챗봇 등에서 카드/퀵리플라이를 탭했을 때 같은 대화창에 새 메시지가 추가되는 것과 동일한 문법이다. 페이지 네비게이션을 쓰면 "AI 챗봇과 대화한다"는 느낌이 깨진다. 실제 예시는 `contents/html/08_ai-preview.html`의 모든 28개 항목 — 카드 탭 이후 화면이 같은 phone-screen 안에서 `.app-tap-divider` 구분선과 함께 이어지는 것으로 표현(N=1도 동일).

**CTA는 턴마다 독립적으로 결정 (Playbook 감지와 무관하게 additive, 2026-07-05 확정):** 요약 카드 턴과 카드 탭 이후 심층 답변 턴은 **서로 다른 발화(턴)**이므로, 각 턴마다 `matching-policy.md`의 Concept·Playbook 매칭이 독립적으로 실행된다. 즉 요약 카드 턴에서 Playbook이 감지돼 CTA가 붙었다고 해서 그 CTA가 다음 턴에도 자동으로 이어지는 게 아니며, 카드 탭 이후 턴에서 Playbook이 (같거나 다르게) 감지되면 그 턴만의 CTA가 별도로 붙는다. 한 대화 안에 서로 다른 CTA가 두 번 나오는 것은 중복이 아니라 "각 턴에 맞는 CTA가 붙는다"는 기존 Case 1~4 로직이 그대로 적용된 결과다.

**나머지 카드 재노출 (N≥2일 때, 2026-07-05 확정):** N=1일 때는 해당 없음. N≥2일 때, 심층 답변 + CTA 다음에 "다른 보장 공백도 확인해보시겠어요?" 같은 짧은 문장과 함께 **아직 탭하지 않은 나머지 카드를 다시 노출**한다. 카드 리스트는 채팅 기록에 계속 남아있어 위로 스크롤하면 재확인할 수 있지만, 대화가 길어질수록(특히 종합진단처럼 카드가 많을 때) 스크롤 없이 이어서 다음 항목을 확인할 수 있게 하기 위함이다. 탭한 카드는 재노출 목록에서 제외한다. 새 컴포넌트가 아니라 기존 결과 카드 UI를 답변 끝에 한 번 더 재사용하는 것뿐이다. 실제 예시는 `contents/html/08_ai-preview.html`의 5개 그룹 Concept 항목(CN-T02A·CN-HOSPALL·CN-INJURYALL·CN-CAREALL·CN-ALL) 참조.

### N = 0 처리 (전체 진단 결과 공백 없음)

경로 B에서 23개 Rule을 전부 평가했는데 하나도 충족하지 않으면(공백이 없으면), 결과 카드 없이 "특별한 보장 공백이 확인되지 않아요. 잘 준비하고 계세요!" 류의 안심 메시지로 응답한다 — 빈 카드 리스트를 보여주지 않는다.

## KC 메인 체인 vs Playbook 병렬 체인

KC 메인 체인과 Playbook 체인은 **동시에 독립 분석**한다. 두 결과의 조합으로 Case 1~4가 결정됨.

```
사용자 질문
  ├─ [KC 메인 체인] Concept 동의어 매칭
  │    ├─ 매칭O → Risk-type 판정 → Rule 평가
  │    └─ 미매칭
  └─ [Playbook 체인] 발화 키워드 감지 (독립 동작)
       ├─ 감지O
       └─ 미감지

결과 조합:
  매칭O + 미감지 → Case 1: KC 구조화 답변
  매칭O + 감지O  → Case 2: KC 구조화 답변 + CTA 버튼
  미매칭 + 미감지 → Case 3: RAG(약관→FAQ) → Fallback 생성형
  미매칭 + 감지O  → Case 4: Standalone 가이드 주입 + CTA 버튼
```

## RAG 아키텍처 (2026-06-21 확정)

```
사용자 질문
├─ KC 카드 매칭 → 구조화 답변 (Case 1/2)
└─ KC 미매칭 → RAG 레이어
   ├─ 약관 RAG: 가입 상품 약관 자동 파싱·인덱스 (크롤러 자동, KC Admin 직접 관리 불가)
   └─ Clark 서비스 FAQ RAG: 운영자 직접 등록 → 즉시 인덱스 등록 (19_faq-rag.html, 승인 없음)
```

|---|---|---|
| 약관 RAG | 가입 상품 약관 원문 검색·인용 | 크롤러 자동 파이프라인 |

**(2026-07-04 정교화) FAQ RAG 등록 기준 — "계약 특정적" vs "일반 판단 노하우" 구분:**

| 구분 | 판단 기준 | 처리 |
|---|---|---|
| 계약 특정적 질문 | 특정 개인의 실제 계약 내용을 알아야 정확히 답할 수 있는 질문 (예: "제 보험에 이 담보 있어요?", "제 진단금 얼마예요?") — 사람마다 정답이 다름 | **FAQ RAG 등록 금지.** 약관 RAG(개인 계약 조회) 또는 KC 체인(구조화 진단)이 처리 |
| 일반 판단 노하우 질문 | 계약과 무관하게 업계·회사 판단 기준으로 보편적으로 답할 수 있는 질문 (예: "용종제거가 질병수술에 해당하나요?", "유사암과 일반암 차이가 뭐예요?", "실비랑 정액보험 차이", "갱신형 vs 비갱신형 뭐가 나아요?") | **FAQ RAG 등록 대상.** 약관 원문 검색으로는 안 나오고(약관에 모든 시술명·비교 판단이 나열돼 있지 않음) 일반 LLM 지식만으로도 부정확할 수 있어, GA·보험사의 축적된 청구·상담 노하우를 사전 등록해 답변 품질을 높인다 |

## FAQ RAG 매칭 방식 (2026-07-05 확정)

> `context/matching-policy.md`의 Concept·Playbook 매칭 엔진(임베딩 벡터 유사도)을 FAQ 인덱스에도 동일하게 재사용한다. 별도 엔진을 새로 만들지 않는다.

**호출 시점:** 위 "상태별 답변 흐름"·"Case 1~4" 순서에 따라 KC 체인 미매칭(또는 약관 RAG 불가·결과없음) 상태에서 진입한다.

**인덱싱 대상:** 등록된 FAQ 각 항목의 **"질문 표현(Q)" 필드 전체**(운영자가 입력한 여러 발화 예시 전부) — FAQ 순번(카드 ID)과 함께 인덱싱해 매칭 시 어느 FAQ인지 식별한다. Concept의 "동의어·예문 전체 인덱싱" 방식과 동일하다.

**FAQ 필드 구조 (2026-07-06 확정 — Q/A 2필드에서 4필드로 확장):**

| 필드 | 필수여부 | 용도 | LLM 재구성 여부 |
|---|---|---|---|
| 질문(Q) | 필수 | 임베딩 인덱싱 대상(발화 예시 여러 개) | 해당 없음(인덱싱 전용) |
| 답변(A) | 필수 | 매칭 시 최종 답변의 근거 텍스트 | **재구성됨** — LLM이 자연스러운 문장으로 재작성 |
| 출처 | 선택 | 공인 기관·보고서 등 근거 출처 표기(운영자 화면·검수용) | 재구성 대상 아님 — 고객 화면에 원문 노출 안 함(`guides/customer-messaging.md` 워싱 원칙에 따라 A 재구성 시 자연스럽게 녹이는 재료로만 참조) |
| ⚠️ 유의사항 | 선택 | 법적·보상 관련 유의사항·면책 안내 | **재구성 대상 아님 — 원문 그대로 답변 끝에 고정 부착.** KC 체인의 Policy(면책 문구)가 "RAG·LLM 별도 호출 없음"으로 원문 그대로 나가는 것과 동일한 원칙 — 유의사항은 의미가 조금이라도 달라지면 안 되는 내용이라 LLM이 자유롭게 재구성하지 않는다 |

이전엔 편집기(`mockups_v2/19_faq-rag.html`)가 Q/A 2필드만 지원해 실제 콘텐츠(`contents/08_faq/faq-rag.md`, Q/A/출처/⚠️ 4필드)와 구조가 안 맞았다 — 편집기에 출처·⚠️ 유의사항 필드를 추가해 맞췄다.

**매칭 절차:**
```
[사용자 질의] (KC 미매칭 + 약관 RAG 불가/결과없음 상태에서 진입)
      │
      ▼
[FAQ 벡터 검색] — 등록된 전체 FAQ의 Q 필드와 임베딩 유사도 비교 (동시 실행)
      │
      ▼
 유사도 최고점 1개 선택
      │
      ▼
 기준값 이상인가?
  ├─ Yes → 매칭된 FAQ의 A(답변) 원문을 컨텍스트로 LLM에 전달
  │         → LLM이 자연스러운 문장으로 재구성해 최종 답변 생성
  │         → ⚠️ 유의사항이 등록돼 있으면 재구성 없이 원문 그대로 답변 끝에 덧붙임
  └─ No  → FAQ RAG 결과 없음 → Fallback LLM(18_system-settings 설정 적용)
```

**LLM 답변 생성 원칙:**
- 매칭된 FAQ의 A(답변) 원문 범위 안에서만 재구성한다 — 등록되지 않은 사실을 추가·추론하지 않는다(할루시네이션 방지).
- **⚠️ 유의사항은 LLM 재구성 대상이 아니다** — 원문 그대로 고정 부착한다(위 필드 구조 표 참조).
- 여러 FAQ가 동시에 기준값을 넘겨도 **유사도 최고점 1개만** 사용한다(Concept 매칭과 동일 원칙). 카테고리 A(판단 노하우)·B(서비스 안내)가 동시 매칭돼도 카테고리 구분 없이 유사도 순으로만 판단한다.

**Concept·Playbook과의 차이:** Concept·Playbook은 매칭 성공 시 "카드 구조"(Risk-type→Rule→Evidence→Policy 체인, 또는 CTA)로 이어지지만, FAQ RAG는 매칭 성공 시 **등록된 A(답변) 텍스트 자체가 최종 답변의 근거**가 된다 — 후속 체인이 없는 종단(leaf) 구조다.

**미결(개발팀 결정 필요 — `matching-policy.md` 미결 항목과 동일 성격):**

| 항목 | 설명 |
|---|---|
| 기준값 수치 | Concept·Playbook과 같은 기준값을 공유할지, FAQ 전용 별도 기준값을 둘지 |
| Top-K | 최고점 1개만 쓸지, Evidence처럼 상위 여러 개를 검색해 LLM이 맥락에 맞게 선택하게 할지 |
| 카테고리 A/B 우선순위 | 판단노하우(A)와 서비스안내(B)가 동시 매칭 시 유사도만으로 정할지, 카테고리별 가중치를 둘지 |

## 약관 RAG 버전 매핑 방법 (2026-06-24 확정)

같은 보험사·상품이라도 **계약 시작일에 따라 약관 세부 내용이 다름**. 상품 고유코드(상품코드)는 마이데이터에 미포함.

### 매핑 키 (Phase 1)

마이데이터 가입 정보에서 추출 가능한 값:

```
보험사명 + 상품명 + 계약시작일(연도 단위)
예) "삼성생명" × "무배당 암보험" × "2019년 가입분" → 약관 v2019
```

크롤러 파이프라인이 약관 파일을 저장할 때 아래 메타데이터를 함께 인덱싱:

| 메타데이터 | 예시 |
|---|---|
| insurer | 삼성생명 |
| product_name | 무배당 암보험 |
| effective_start | 2018-04-01 |
| effective_end | 2021-03-31 |

조회 쿼리: `insurer = X AND product_name 포함 AND effective_start ≤ 계약시작일 ≤ effective_end`

### 한계 및 Phase 2 개선 방향

| 구분 | 내용 |
|---|---|
| Phase 1 한계 | 같은 연도 내 중간 개정이 있을 경우 버전 불일치 가능성 있음 |
| 답변 면책 | "가입 시점에 따라 세부 약관 내용이 다를 수 있습니다" 문구 자동 포함 |
| Phase 2 목표 | 보험사 API 직접 연동 → 계약번호로 정확한 약관 버전 조회 |

## Evidence · Policy 활용 방식 (2026-06-26 확정)

### Rule 선택 조건(PROMAGE) — 매칭 시 답변 반영 원칙 (2026-07-05 확정)

`context/card-types.md`의 Rule 필수/선택 조건 정의 중, PROMAGE(프롬에이지 건강위험도) 선택 조건은 **우선순위 계산(2순위 활용)뿐 아니라 답변 문구 자체에도 반영**한다.

| 상황 | 답변 처리 |
|---|---|
| PROMAGE 선택 조건 매칭 안 됨(미충족 또는 미연동) | 기존과 동일 — MYDATA 필수 조건 기반 진단 문구만 표시 |
| PROMAGE 선택 조건 매칭됨(해당 위험도 항목이 Rule 조건 충족 — EQ 정확히 일치 또는 GTE/LTE 범위 포함) | 진단 문구에 "프롬에이지로 연동하신 건강정보를 보니 [항목] 위험도가 '[등급]' 등급으로 확인돼요" 문장을 추가해 경각심을 높인다. 근거(Evidence) 설명 뒤에는 "위험/경고 등급으로 확인된 만큼 더 서둘러 점검해보시는 게 좋다"는 취지의 마무리 문장을 덧붙인다 |

**표기 규칙:**
- 위험도는 **"고위험/위험/경고/주의/양호" 5단계 등급명만** 노출한다(2026-07-06 3단계→5단계 확장, 심각도 순: 고위험>위험>경고>주의>양호). PROMAGE 원본 점수·수치는 노출하지 않는다(등급명 정책은 `guides/copywriting.md` 기존 규정과 동일). Rule 조건이 GTE/LTE로 범위를 지정했더라도, 사용자에게 보여주는 건 **실제 매칭된 그 사용자의 등급 하나**이지 조건식 자체가 아니다(예: "위험 이상" 조건에 실제 등급이 "고위험"이면 "고위험 등급으로 확인돼요"라고 표시).
- 면책 문구(Policy)에 "프롬에이지 위험도는 의학적 진단이 아닌 참고 지표"라는 문장을 추가로 포함한다.
- Rule에 PROMAGE 선택 조건이 아예 등록돼 있지 않은 Risk-type은 이 처리를 적용하지 않는다(임의로 위험도 언급을 만들어내지 않음).

실제 모바일 앱 UI 적용 예시(①기본 케이스 ②선택조건 매칭 시 비교)는 `contents/html/08_ai-preview.html`(T02·03·04·05·10·17·18·19·20·22·23·24·25·26·29·30·32 17개 체인) 참조.

### Evidence — LLM 맥락 선택

| 항목 | 정책 |
|---|---|
| 선택 방식 | LLM이 사용자 질문 맥락에 맞는 Evidence를 **"활용 방법" 필드** 기준으로 선택·인용. 연결된 전부를 나열하지 않음 |
| 권장 연결 수 | Rule 1개당 Evidence **3개 이하** 권장 (하드 제한 없음. 초과 시 LLM 선택 품질 저하 가능) |
| "활용 방법" 필드 역할 | LLM의 선택 기준. 어떤 Rule 조건의 근거인지, 왜 이 수치가 기준인지 명확히 작성할수록 맥락 인용 품질 향상 |
| **인용 표기 원칙(2026-07-05 확정)** | Evidence 수치를 답변에 인용할 때는 **출처 기관 + 기준 연도**를 문장 안에 자연스럽게 포함한다(예: "금융감독원 2025년 통계 기준 실손보험 미가입자는..."). 사용자가 근거의 신뢰도를 스스로 판단할 수 있어야 하기 때문이다. 단, 보고서 정식 명칭·출처 URL까지는 채팅 문장에 넣지 않는다(딱딱해짐) — 상세 출처가 필요하면 별도 액션(예: "출처 더보기")으로 분리 검토 |

### Policy — 연결된 전부 적용

| 항목 | 정책 |
|---|---|
