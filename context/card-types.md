## 필드별 AI 참조 여부 총정리 (2026-07-06 신규 확정)

> 개발자가 프롬프트·템플릿 조합 로직을 짤 때 "이 입력 필드가 실제 사용자 화면에 나가는지, 나간다면 원문 그대로인지 재구성되는지, 아니면 아예 매칭/내부 판단에만 쓰이고 사용자에게는 안 나가는지"를 한눈에 보기 위한 표. 카드별 필드 상세는 아래 각 섹션 참조, 개발용 API 키·타입까지 포함한 표는 `policy/06_field-data.html`.

**4가지 분류:**

| 분류 | 의미 |
|---|---|
| 🗣️ **발화(원문)** | 사용자 화면에 **그대로** 나간다. LLM이 재구성·재작성하지 않는다(Policy 면책 문구가 대표 사례 — 의미가 조금이라도 바뀌면 안 되는 내용) |
| 🗣️ **발화(재구성)** | 이 필드의 내용을 **근거로** LLM(또는 KC 엔진의 문장 조합 로직)이 자연스러운 문장으로 재작성해서 사용자에게 보여준다. 원문 그대로는 안 나간다(기관명·수치 표기 등은 `guides/customer-messaging.md` 워싱 원칙 적용) |
| 🔍 **매칭 전용** | 사용자에게 **절대 발화되지 않는다.** 사용자 질문과 이 카드를 연결(임베딩 매칭·라우팅)하는 데만 쓰인다 |
| 🔒 **내부 전용** | 운영자 화면·검수·통계용 메타데이터다. AI 답변 생성 파이프라인에 아예 전달되지 않는다(카드 코드·상태·공개범위·연결 그래프 정보 등) |

> **혼동 주의(2026-07-06):** 위 "🔒 내부 전용"은 **AI 참조 여부 분류 라벨**이지, 카드의 `공개범위(scope)` 필드값(고객 공개/공통 기준/내부 전용)과는 **다른 축**이다. 예를 들어 Concept 카드는 `공개범위=고객 공개`(AIO 색인 대상, `mockups_v2/15_aio-guide.html` 참조)이면서도 개념명·설명 필드 자체는 "🔒 내부 전용"(AI 답변에 직접 발화되지 않고 매칭에만 쓰임)으로 분류된다 — 두 표기가 같은 단어를 써서 모순처럼 보일 수 있으나 서로 다른 질문에 답하는 것이니 혼동하지 말 것.

**핵심 원칙 — KC 체인(Risk-type→Rule→Evidence→Policy) vs FAQ RAG:**
- **KC 체인 매칭 성공(Case 1·2)**: `context/answer-logic.md` "RAG·LLM 별도 호출 없음" 원칙에 따라, Risk-type 유형명·설명 + Rule의 `clarkMessage` + Evidence 지표·기준값(자연어 재구성) + Policy `appDisplayText`를 **템플릿으로 조합**한다. 별도의 생성형 LLM 호출이 없다 — 즉 발화되는 문장은 전부 운영자가 미리 써둔(또는 자연어 워싱해둔) 콘텐츠의 조합이다.
- **FAQ RAG 매칭 성공(Case 3)**: 등록된 답변(A)을 LLM이 **런타임에 자연스러운 문장으로 재구성**한다. ⚠️ 유의사항만 예외적으로 원문 그대로 고정 부착된다(Policy와 동일 원칙).

**카드 타입별 요약표:**

| 카드 | 필드 | AI 참조 여부 |
|---|---|---|
| Risk-type | 명칭(name) | 🗣️ 발화(원문) — "고객님은 [명칭]에 해당하는 상태예요" 진단 선언에 사용 |
| Risk-type | 설명(desc) | 🗣️ 발화(원문) |
| Risk-type | 중요도(priority) | 🔒 내부 전용 — 다중 감지 시 노출 순서 계산에만 사용 |
| Risk-type | 카드 코드(T코드)·상태·공개범위 | 🔒 내부 전용 — T코드는 운영자 전용 식별자, 고객 화면 절대 비노출 |
| Concept | 동의어 목록(synonyms) | 🔍 매칭 전용 — 임베딩 벡터 매칭 대상. FAQ의 Q필드와 동일한 역할 |
| Concept | 개념명(name)·설명(desc) | 🔒 내부 전용 — 매칭 성공 후 답변은 Risk-type부터 시작, Concept 자체 내용은 발화 안 됨 |
| Concept | 연결 Risk-type | 🔒 내부 전용 — 라우팅 정보 |

| Evidence | 보고서명(reportName)·출처 URL | 🔒 내부 전용 — 보고서 정식 명칭·URL은 채팅 문장에 넣지 않음(딱딱해짐) |

| Evidence | 제목(title)·카드 코드 | 🔒 내부 전용 |
| Rule | Clark 경고 메시지(clarkMessage) | 🗣️ 발화(원문) — KC 구조화 답변의 핵심 문장. LLM 재구성 없음 |

| Policy | Clark 앱 표시 문구(appDisplayText) | 🗣️ 발화(원문) — 면책조항 섹션에 그대로 삽입, LLM 재구성 없음 |

## Risk-type 카드

**역할:** Clark AI가 보험 리스크를 분류하는 기준 카드. Concept·Rule 카드가 이 카드를 참조해 KC 체인을 구성.

**중요도 (2026-06-22 확정):**

| 중요도 | 의미 |
|---|---|
| 높음 | 핵심 진단 유형 — 다른 유형과 동시 감지 시 선노출 |
| 보통 | 일반 진단 유형 — 기본값 (미설정 시 자동 적용) |
| 낮음 | 보조 진단 유형 — 상위 유형 노출 후 후순위 안내 |

**다중 감지 우선순위 (2026-06-22 재확정):**

| 순위 | 기준 | 의미 |
|---|---|---|
| 1순위 | 중요도 서열 (높음 > 보통 > 낮음) | 운영자 판단 — 서비스 전략 |
| 2순위 | 선택 조건 충족 개수 | 시스템 자동 계산 — 개인화 |
| 3순위 | 최근 배포순 | 타이브레이크 (가장 최근 라이브 전환일 우선) |

**가중치 완전 제거됨:** 중요도 ×3/×2/×1 가중치 개념 삭제. 높음/보통/낮음 서열만 사용.

## Concept 카드

**역할:** 사용자 질문의 핵심 단어를 동의어로 매칭해 Risk-type 체인으로 연결하는 입구 카드.

- Concept → Risk-type 연결 **필수** (최소 1개). 연결 없으면 KC 체인 미진입.
- Standalone 기능 없음 — 완전 제거됨 (구 기획 흔적 코드 삭제 대상)
- 미매칭 시 Case 3 (RAG→Fallback) 또는 Case 4 (Playbook 감지) 분기

## Evidence 카드

**역할:** Rule 판단 기준값의 근거를 제공하는 공인 통계 데이터 카드. Clark 상담 시 추천 이유 설명에 활용.

**단일 유형: 공인 외부 통계 기반** (code: `external`) — 유형 select 없음.

**편집기 필드 (2026-06-17 확정):**

| 필드 | 필수 | 설명 |
|---|---|---|
| 제목 | ✅ | 핵심 내용 한 줄 요약 |
| 출처 기관 | ✅ | HIRA / NHIS / KOSTAT / FSS / 기타 |
| 보고서명 | ✅ | 인용한 공식 보고서명 |
| 기준 연도 | ✅ | 통계 발행 연도 |
| 지표명 | ✅ | 활용할 통계 지표 명칭 |
| 기준값 | ✅ | 단위 포함 입력 (예: 3,800만원, 68%) |
| 출처 URL | 선택 | 원본 자료 링크 |
| 활용 방법 | ✅ | 어떤 Rule 기준값의 근거인지, 왜 이 수치가 기준인지. Clark AI가 이 필드를 기준으로 Rule에 연결된 여러 Evidence 중 답변에 인용할 것을 선택함(`context/answer-logic.md` "Evidence — LLM 맥락 선택") — 명확하게 작성할수록 인용 품질 향상 |

**폐지 항목:** 보닥통계(stat)·프롬에이지(fromage) 유형 — 완전 제거됨.

## Rule 카드

**역할:** 마이데이터·프롬에이지·프로파일 3개 소스 기반으로 Risk-type 판정 조건을 정의하는 카드. Evidence와 연결해 판단 근거를 갖춤.

### Rule 조건 빌더 3-source (2026-06-08 확정 / 2026-06-22 재확정)

**MYDATA + PROMAGE + 프로파일** — Amplitude 완전 제거.

| source | 화면 표기 | 배경색 클래스 | 기본 required |
|---|---|---|---|
| MYDATA | 마이데이터 | `.cond-row-mydata` (기본 회색) | `true` (필수) |
| PROMAGE | 프롬에이지 | `.cond-row-promage` (보라, #FAF8FD) | `false` (선택) |
| PROFILE | 프로파일 | `.cond-row-profile` (초록, #F4FBF8) | `true` (필수) |

**조건 구분:**

| 구분 | 동작 | 역할 |
|---|---|---|
| 필수 조건 | 모두 충족해야 Rule 발동 (AND) | 리스크 해당 여부 판단 기준선 |
| 선택 조건 | 미충족 시에도 Rule 발동 | ① 충족 개수가 Risk-type 우선순위 2순위에 활용됨 ② (2026-07-05 추가) PROMAGE 선택 조건이 실제로 매칭되면 Clark 답변 문구 자체에 반영해 경각심을 유도한다 — 상세 답변 생성 규칙은 `context/answer-logic.md` 참조 |

**PROMAGE 미연동 대응:** 프롬에이지 미연동 사용자는 PROMAGE 조건 평가 불가. 따라서 PROMAGE 조건은 선택 조건으로 설정 권장. 운영자가 필수로 변경하는 것은 허용 (의도적 설계).

**조건 빌더 컬럼 구조 (2026-06-26 6열 통일):**

| 열 | 헤더 | 소스별 내용 |
|---|---|---|
| 1 | 소스 | 마이데이터 / 프롬에이지 / 프로파일 select |
| 2 | 담보코드·카테고리·항목 | MYDATA: 담보코드 select / PROMAGE: 위험도 항목 select / PROFILE: 나이·성별·결혼여부·자녀유무·운전여부 select |
| 3 | 조건 | MYDATA: op select(LTE/GTE 등) / PROMAGE: op select(EQ/GTE/LTE) / PROFILE 나이: 사이(∈) static / PROFILE 나머지: 같음(=) static |
| 4 | 기준값 | MYDATA: 금액 input / PROMAGE: 고위험·위험·경고·주의·양호 select / PROFILE 나이: 시작세~종료세 dual input / PROFILE 나머지: 값 select |
| 5 | 필수여부 | 필수 / 선택 radio |
| 6 | (삭제) | 삭제 버튼 |

**PROMAGE 조건 구조 (2026-06-25 단순화 / 2026-06-26 optgroup 적용 / 2026-07-06 5단계·GTE·LTE 확장):**
- **카테고리 단일화:** 암위험도 + 질병위험도 → **위험도** (50개 항목 통합). 생체나이·의료비예측 삭제.
- **등급 체계:** 프롬에이지가 실제로 내려주는 값은 **고위험 > 위험 > 경고 > 주의 > 양호** 5단계 서수(ordinal) 등급이다.
- **op:** EQ / GTE / LTE — 5단계 등급 select. 심각도 방향은 고위험이 가장 심각(값이 큰 쪽), 양호가 가장 약함(값이 작은 쪽)으로 취급 — "위험 이상"(GTE 위험 = 고위험·위험 매칭), "주의 이하"(LTE 주의 = 주의·양호 매칭) 같은 조건을 만들 수 있다.
- **편집기 Row:** source select → 항목 select (optgroup 2개) → op select(EQ/GTE/LTE) → 등급 select
- **optgroup 구분:** `<optgroup label="암위험도">` 23개(canr_ca~smbl_ca) + `<optgroup label="질병위험도">` 27개(고지혈증~췌장염)

**MYDATA Rule 조건:** 담보코드 select + 담보금액(coverage_amt) LTE/GTE/LT/GT/EQ. 필드 드롭다운 없음.

**USER_PROFILE_FIELDS for Rule 조건 (2026-06-26 업데이트):**

| 필드코드 | 레이블 | op | 기준값 | 비고 |
