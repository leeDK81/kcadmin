# KC Admin — Concept·Playbook 감지 정책

> **진실원.** 사용자 질의 → Concept·Playbook 매칭 방식과 결과 처리 규칙.
> 변경 시 PO(02)가 이 파일을 즉시 업데이트한다.
> 구현 세부(기준값·모델 선택)는 개발팀이 결정 후 이 파일에 반영한다.

---

## 개요

사용자 질의가 들어오면 **Concept 매칭**과 **Playbook 매칭**을 동시에 실행한다.
둘은 역할이 다르므로 독립적으로 결과를 낸다.

| 카드 | 역할 | 결과 |
|---|---|---|
| Concept | KC 답변 체인 진입 트리거 | KC 구조화 답변 생성 |
| Playbook | CTA 노출 트리거 | 버튼·안내 레이어 추가 |

---

## 매칭 방식 — 임베딩 의미 유사도

두 카드 모두 **임베딩 벡터 유사도**로 매칭한다.
키워드 완전 일치 방식을 쓰지 않는다.

**이유:**
- 운영자가 등록하지 않은 표현(동의어·구어체·오탈자)도 의미가 같으면 잡아야 한다.
- "암"만 등록해도 "악성종양", "종양 진단", "cancer" 같은 표현이 매칭되어야 한다.
- Concept 동의어·예문과 Playbook 예상질의 모두 문장·단어 혼용이 가능하므로 동일 엔진으로 처리한다.

---

## 인덱싱 대상

### Concept 카드
- **동의어·예문 필드 전체** (운영자가 입력한 단어·구·문장 전부)
- 카드 ID와 함께 인덱싱 → 매칭 시 어느 Concept인지 식별

### Playbook 카드
- **예상 질의 필드 전체** (운영자가 입력한 문장들)
- 카드 ID와 함께 인덱싱 → 매칭 시 어느 Playbook인지 식별

> Concept의 "용어명(대표명)"은 인덱싱 필수. 동의어 미등록 카드도 대표명으로는 매칭되어야 함.

## 실행 구조

> **2026-07-06 갱신 — Case 0(데이터 조회형) 선행 게이트 추가, 2026-07-06 Playbook 병행 원칙 정정.** Concept 매칭을 실행하기 전에, "위험 진단"이 아니라 "내가 가진 데이터를 그대로 보여달라"는 조회형 발화인지 먼저 판별한다. 감지되면 Concept 매칭(KC 체인)을 건너뛰고 마이데이터를 직접 조회해 응답한다. **단, Playbook 벡터 검색은 이 판별과 무관하게 항상 독립적으로 병행 실행된다** — Case 0으로 조회 응답을 하는 중에도 발화가 어떤 Playbook 키워드와 겹치면 CTA가 정상적으로 가산된다. 상세는 `context/answer-logic.md` "Case 0 — 데이터 조회형" 참조.

```
[사용자 질의 입력]
        │
        ├───────────────────────────────┬──────────────────────────────┐
        ▼                               ▼                              ▼
[Case 0 의도 분류: 조회형 감지]      [Concept 벡터 검색]          [Playbook 벡터 검색]
        │                        (Case 0 감지 시 스킵)              (항상 실행)
   ┌────┴────┐                          │                              │
  감지O        감지X                     ▼                              ▼
   │            │              유사도 최고점 1개 선택        유사도 최고점 1개 선택
   ▼            ▼              기준값 이상이면 매칭           기준값 이상이면 매칭
[마이데이터        (Concept 결과로               │                              │
 직접 조회 응답]    Case 1~4 진행)                ▼                              ▼
   │            │                       KC 체인 진입 or RAG/Fallback      CTA 레이어 적용
   │            │                      (Case 0 감지 시 이 경로 자체가 없음)   (핵심 답변에 CTA 추가 노출)
   └──────┬─────┴──────────────────────────────┘                              │
          ▼                                                                   │
   [핵심 답변(조회 결과 / KC 구조화 / RAG / Fallback)] ◄─── CTA 있으면 가산 ─────┘
```

## 결과 처리

### Case 0(데이터 조회형) 매칭 — Concept 전용 최우선 게이트
- 감지되면 **Concept 매칭만** 건너뛰고 마이데이터 직접 조회로 종결한다.
- **Playbook 매칭은 이 게이트와 무관하게 항상 별도로 실행된다** — Case 0으로 판별돼도 Playbook 키워드가 겹치면 CTA가 정상적으로 가산된다.
- 상세 정의·1차 지원 조회 유형·데이터 상태별 처리는 `context/answer-logic.md` "Case 0 — 데이터 조회형" 참조.

### Concept 매칭
- 기준값 이상인 Concept 중 **유사도 최고점 1개만** KC 체인에 진입시킨다.
- **동점(또는 근접 동점)이면 가장 최근에 배포(라이브 전환)한 카드가 선택된다** (2026-07-07 변경 — 기존 "카드 ID 오름차순"은 CN-T02A·CN-T02B가 "암진단금" 동의어를 공유해 항상 알파벳상 앞선 CN-T02A만 이기는 편향을 실제로 일으켰다(2026-07-05 발견, 컨텐츠 중복 제거로 임시 해결). Risk-type 3순위 타이브레이크(2026-06-26, 카드코드 오름차순→최근 배포순)와 동일 기준으로 통일한다).
- 기준값 미만이면 Concept 미매칭 → KC 체인 없이 약관 RAG → FAQ RAG → LLM Fallback.
- **사전 방지책(2026-07-07 신규):** 신규 Concept 등록·수정 시 기존 전체 Concept과 동의어·예문 중복도를 검사해 겹침이 크면 저작 단계에서 미리 정리한다(`contents/agents/04_content-po.md` "동의어·예상질의 중복 검사" 참조). 위 동점 처리는 이 사전 검사를 통과한 뒤에도 남는 드문 경우의 안전망이다.

### Playbook 매칭
- 카드 감지는 **키워드(임베딩 유사도) 매칭만으로** 결정된다 — 사용자의 데이터 보유 상태는 카드가 매칭 후보인지 여부에 전혀 관여하지 않는다(2026-07-20 최종 — 이전엔 "데이터 미보유 감지 조건"으로 카드 자체를 게이트했으나, 상담신청이 마이데이터·회원가입을 이미 전제하는 구조라 카드 단위 사전 분기가 불필요하다고 판단해 폐지. 경과는 `context/decisions.md` 2026-07-20 항목 참조).
- 기준값 이상인 Playbook 중 **유사도 최고점 1개만** 매칭되어 그 카드의 버튼 세트가 노출된다.
- **동점(또는 근접 동점)이면 가장 최근에 배포한 카드가 선택된다** (2026-07-07 신규 — Concept과 동일 기준으로 통일. 기존엔 Playbook 동점 규칙이 정의돼 있지 않았음). 사전 방지책은 Concept과 동일하게 `contents/agents/04_content-po.md` 참조.
- 기준값 미만이면 CTA 미노출 (KC 답변만 단독 제공).
- **매칭된 카드 안에서 어떤 버튼이 뜨는지는 별도 로직**: 기본 버튼은 항상 노출, 추가 버튼은 각자의 노출 조건(상시/PROFILE없음/마이데이터없음/프롬에이지없음)이 사용자의 실시간 null 상태와 일치할 때만 노출된다. 여러 추가 버튼이 동시에 조건을 만족할 수 있고(예: 상시 노출 버튼 + PROFILE없음 버튼이 상태①에서 함께 뜸), 서로 다른 필드를 보므로 우선순위 조정이 필요 없다. 상세는 `context/answer-logic.md` "Playbook 데이터 미보유 감지 조건" 절 참조.

### 둘 다 매칭될 때
- Concept 매칭 → KC 답변 생성
- Playbook 매칭 → KC 답변 하단에 CTA 추가
- 두 결과가 충돌하지 않으므로 **항상 병렬 노출(additive)**.

### 둘 다 미매칭일 때
- 약관 RAG → FAQ RAG → LLM Fallback 순으로 진행 (answer-logic.md 참조).
- Playbook CTA 없음.
- **FAQ RAG도 이 문서의 임베딩 유사도 엔진을 그대로 재사용한다** (2026-07-05 확정). FAQ 인덱싱 대상·매칭 절차·미결 항목 상세는 `answer-logic.md` "FAQ RAG 매칭 방식" 섹션 참조.
