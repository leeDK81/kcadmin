## 표준 답변 구조 (2026-07-05 확정 — `mockups_v2/13_answer-logic.html` "Clark 실제 앱 출력 형식 원칙", `contents/html/08_ai-preview.html` 카드 우선 UI 기준)

> **주의:** 아래는 이전(~2026-07-04) "T코드·영문 Risk-type을 답변에 노출하는 5단계 구조"를 대체한 현재 구조다. T코드·영문 "Risk-type"·담보코드·연산자(GTE/LTE) 등 운영자·개발자 전용 식별자는 **실제 사용자 화면에 절대 노출하지 않는다** — 기획·개발 문서(`13_answer-logic.html` 등)에 T02·RU001 같은 코드가 보이는 건 어디까지나 설명 목적이다.

1. **진단 선언 우선:** "고객님은 **[위험유형명(한글)]**에 해당하는 상태예요"처럼 Risk-type의 한글 명칭(카드 필드에 이미 있음)으로 먼저 결론을 선언한 뒤 이유·근거를 설명한다. T코드·영문 표기 없음.
2. **상황 설명 + 근거:** 왜 위험한지 + Evidence 통계를 자연스러운 문장으로 녹여 설명(수치 각주 나열 아님). 출처 기관·기준 연도는 "국민건강보험공단 2024년 기준"처럼 자연스러운 한글 문장에 포함하되, 영문 약어(HIRA·NHIS·FSS 등)는 노출하지 않는다.
3. **(선택) 프롬에이지 매칭 시 경각심 반영:** Rule의 프롬에이지 선택 조건이 실제로 매칭되면 "프롬에이지로 연동하신 건강정보를 보니 [항목] 위험도가 '[등급]' 등급으로 확인돼요" 문장을 추가한다. 등급명(고위험/위험/경고/주의/양호, 2026-07-06 3단계→5단계 확장)만 노출하고 원본 수치·조건식(GTE/LTE 등)은 노출하지 않는다.
4. **면책 고지:** Policy 카드 문구, 상황 설명과 분리된 블록(원문 그대로, LLM 재구성 없음).
5. **CTA 버튼:** Playbook 감지 시 "보닥 플래너와 상담하기" 계열 버튼 추가(KC 체인 성패와 무관한 가산 요소).

다건 매칭(N:N Concept, 2026-07-05 확정)일 때는 진단 선언 문장 대신 **말풍선(짧은 총평) + 결과 카드 리스트**로 먼저 보여주고, 카드를 탭하면 그 항목의 1~4단계 답변이 같은 채팅 스레드에 이어진다. 실제 적용 예시는 `contents/html/08_ai-preview.html` 참조.

## 워싱 원칙

- **출처 각주를 그대로 붙이지 않는다.** `source-corpus.md` 원자료는 문장 안에 자연스럽게 녹인다.
  - 원자료: "암(악성신생물) 환자 1인당 급여 진료비 520만 5,326원 (HIRA, 2023년도)"
  - 워싱 결과: "건강보험 급여 기준 암환자 1인당 연평균 직접 의료비는 약 520만 원이지만, 비급여·간병·생활비까지 포함하면 업계 추정 2,000만~3,500만 원 수준입니다."
- **숫자는 고객 체감 단위로.** 만원/억원 단위로 반올림, 불필요한 소수점·천단위 세부 자리 제거.
- **출처 기관명을 채팅 문구에 직접 노출하지 않는다.** "HIRA 발표", "NHIS 통계" 같은 기관 약어 대신 "건강보험 급여 기준", "통계상" 등으로 순화한다. 출처 상세는 운영자 화면(Evidence 편집기)에서만 관리한다.
- **담보 현황(마이데이터 실값)과 통계(공인 출처 자연어)를 섞지 않는다.** 표(2번)는 실측값 그대로, 설명(3번)은 자연어로 — 각 블록의 정보 성격을 유지한다.
- **면책 고지는 상황 설명과 분리된 블록으로 유지한다.** 통계 문장 안에 법적 고지 문구를 섞지 않는다.
- **금지어는 `guides/copywriting.md` 목록을 그대로 준용한다** (기술 용어 노출 금지 원칙 동일 적용 — 예: "파라미터", "런타임에" 등).
- **정보 제공 톤 유지, 공포 마케팅 금지.** "반드시", "무조건", "당장 가입하세요" 같은 단정적 압박 표현을 쓰지 않는다. 위험을 알리되 판단은 고객 몫으로 남긴다.
