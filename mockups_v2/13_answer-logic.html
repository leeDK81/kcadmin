로직 스펙은 개발자·기획자를 위한 구현 기준 문서입니다. 사용자 데이터 상태(미가입·마이데이터 미연결·연결)별 선행 분기, KC 매칭 × Playbook 감지 조합에 따른 4가지 답변 케이스, 카드 선택 우선순위, 출력 필드 연결 범위를 정의합니다.
예시로 보기는 기획자·운영자를 위한 시나리오 설명입니다. "암보험 얼마나 있어야 해?"(Case 2), "설계사 연결해줘"(Case 4) 시나리오로 카드 탐색 흐름을 보여줍니다.

답변 케이스 결정 — KC 매칭 × Playbook 감지 상태 ③(마이데이터 연결) 전용

사용자 발화는 KC 메인 체인과 Playbook 체인이 동시에 독립 분석합니다. 두 결과의 조합에 따라 4가지 답변 케이스 중 하나가 결정됩니다.

케이스KC Concept 매칭Playbook 키워드 감지Clark 출력

Case 3 RAG 처리 흐름: KC 미매칭 + Playbook 미감지 시 RAG 레이어가 먼저 실행됩니다. 약관 RAG → Clark 서비스 FAQ RAG 순서로 검색하며, 유사도가 기준값(기본 0.75) 이상이면 RAG 기반 답변을 반환합니다. 0.5~0.75 구간이면 "상담사 연결" 유도 메시지를 출력하고, 0.5 미만이면 Fallback 제한 생성형 답변이 발동됩니다. 기준값은 시스템 설정에서 관리자가 조정 가능합니다.
Case 4 동작 방식: Concept 카드 미인식 + Playbook 키워드 감지 시, Playbook 편집기의 Standalone 답변 가이드 내용을 AI 답변 생성에 활용합니다. 버튼만 단독 노출 시 클릭률이 낮으므로 가이드 기반 자연어 답변을 먼저 표시합니다. Playbook이 감지된 경우에는 RAG 레이어를 거치지 않고 Case 4로 바로 처리됩니다.

Step 2 — Rule 카드 평가 기준

선택된 Concept → 연결된 Risk-type에 1:1로 전속된 라이브 Rule(1개)의 조건을 평가합니다. Risk-type과 Rule은 1:1 관계이므로 복수 Rule 평가·선택 과정이 없습니다.

Rule 조건 평가
항목기준
필수 조건AND — 모두 충족되어야 Rule 발동. 하나라도 미충족 시 KC 미매칭 처리
선택 조건미충족 시에도 Rule 발동. 충족 개수는 복수 Risk-type 동시 감지 시 노출 우선순위 2순위에 활용됨
Rule 미발동필수 조건 미충족 → RAG 레이어(약관 RAG → Clark 서비스 FAQ RAG) 우선 시도 → RAG 결과 없으면 Fallback 제한 생성형으로 전환
데이터 소스별 조건 평가 방법 — 각 카드 데이터 연결 구조 →
소스평가 방법
마이데이터고객 마이데이터에서 지정 담보코드(coverage_code)의 확인 항목(field)이 기준값(value)을 조건 연산자(op)로 충족하는지 평가
프로파일사용자 프로파일 데이터(나이·성별·결혼여부·자녀유무 4개 항목)에서 지정 필드가 조건을 충족하는지 평가. 나이: BETWEEN (시작세~종료세 범위), 나머지 enum 타입: EQ 조건
프롬에이지상담 시 수집된 건강검진 데이터(위험도 — 암·질병 50개 항목)를 3단계 등급(위험/주의/양호)으로 변환 후 조건 평가. 수치 직접 비교 금지 [§3-8].
프롬에이지 미연동프롬에이지 데이터 없는 사용자 — 해당 조건 "미충족" 처리. 필수 조건이면 Rule 미발동. 선택 조건이면 Rule은 발동하되 충족 개수에서 제외됨 → 선택 조건 설정 권장
기타 소스 없음해당 소스 데이터 취득 불가 시, 해당 조건은 "미충족"으로 처리
Step 2-B — 다중 Risk-type 감지 시 노출 우선순위

하나의 사용자 질의로 복수의 Risk-type이 동시에 감지될 때, 최종 노출 순서를 결정하는 기준입니다.

설계 원칙: Rule은 "이 사용자가 해당 리스크인가" 진단만 담당합니다. 우선순위는 서비스 전략(중요도)과 사용자 적합도(선택 조건 충족 개수)로만 결정되며, Rule 충족 개수·조건 수 합계는 우선순위에 사용하지 않습니다.
우선순위 결정 — 3단계 계층 구조
순위기준설명
1순위 중요도 운영자가 설정한 서비스 전략 판단 — 높음 > 보통 > 낮음. 동일 중요도면 2순위로 이동
2순위 선택 조건 충족 개수 발동된 Rule의 선택 조건 중 실제 충족된 개수 합계. 이 사용자에게 더 강하게 해당하는 Risk-type 우선
3순위 카드 코드 오름차순 최종 타이브레이크. T01 → T02 → … 순서
운영자 중요도 (Risk-type 편집기에서 설정)
중요도활용 기준
높음핵심 진단 유형 — 다른 유형과 동시 감지 시 반드시 선노출
보통일반 진단 유형 — 기본값 (미설정 시 자동 적용)
낮음보조 진단 유형 — 상위 유형 노출 후 후순위 안내
적용 예시
T02(중요도: 높음) Rule 발동, 선택 조건 2개 충족 → 1순위: 높음 / 2순위: 2개
T05(중요도: 높음) Rule 발동, 선택 조건 3개 충족 → 1순위: 높음 / 2순위: 3개
→ 1순위 동점(둘 다 높음) → T05 우선 (2순위 선택 조건 3 > 2)
2·3순위는 시스템 자동 계산. 운영자는 중요도 3단계 설정만 합니다.
중요도 설정 → Risk-type 카드 편집기
Step 3 — Evidence 참조 방식

충족된 Rule에 연결된 Evidence 카드들이 답변의 근거 데이터로 활용됩니다.

항목기준
참조 방식LLM이 사용자 질문 맥락에 맞는 Evidence를 선택·인용. 연결된 Evidence의 활용 방법 필드가 선택 기준 — 전부 나열하지 않음
운영 권장: Rule당 Evidence 3개 이하 (초과 시 LLM 선택 품질 저하 가능)
Evidence 단일 유형 공인 외부 통계 (external) — HIRA·NHIS·KOSTAT·FSS 등 공인 기관 통계. 운영자가 Evidence 카드에 직접 등록한 고정값 (기준 연도 명시 필수)

T02에 Rule이 여러 개 연결되어 있으면 하나씩 조건을 검사합니다. 조건에 맞는 Rule이 있으면 그 중 가장 조건이 많이 맞는 Rule을 선택합니다.
의사가 여러 진단 기준 중에서 이 환자에게 해당하는 기준을 찾는 것과 같습니다.
Evidence 복수 Evidence → 전부 참조
Rule에 Evidence가 여러 개 연결되어 있으면 선택 없이 모두 참조합니다. 근거가 많을수록 답변이 더 풍부해집니다.
법원에서 주장을 뒷받침하는 증거 자료를 여러 개 제출하는 것과 같습니다.
Policy 복수 Policy → 모두 적용
Rule에 Policy가 여러 개 연결되어 있으면 모두 동시에 적용합니다. 여러 Policy가 충돌하면 더 엄격한 규칙을 따릅니다.
회사 규정이 여러 개일 때 가장 엄격한 규정을 기준으로 따르는 것과 같습니다.
카드가 없거나 조건이 안 맞으면 어떻게 되나요?
1
동의어가 등록된 Concept 카드가 없다
예: 사용자가 "IRP 어때요?"라고 물었는데 "IRP"를 동의어로 가진 카드가 없는 경우. Clark AI가 이 질문의 의도를 KC 카드 체계로 분류할 수 없습니다.
→ 일반 AI 답변. KC 카드 미사용.
2
Concept 카드는 있지만 이 고객의 조건에 맞는 Rule이 없다
예: "암보험" 단어는 인식했고 T02 유형도 확인했지만, 이 고객의 암진단금이 이미 충분히 높아 모든 Rule 조건을 통과하지 못한 경우.
→ 일반 AI 답변. "해당 없음"으로 판단.
3
Concept 카드가 Risk-type과 연결되지 않았다
예: "암진단" Concept 카드는 등록되어 있지만 아직 어떤 Risk-type과도 연결하지 않은 상태.
→ 일반 AI 답변. 카드 연결 작업 필요 안내.
최종 답변에서 각 카드의 어떤 내용이 쓰이나요?
카드 답변에 사용되는 내용 어디에 쓰이나 데이터 출처 답변에 쓰이지 않는 내용
① Concept 없음 사용자 질문 인식에만 사용. 답변 본문에는 포함되지 않습니다. 없음 — 질의 키워드 비교 전용 용어명, 동의어 목록, 한 줄 정의 — 모두 내부 매칭용
② Risk-type 위험 유형명 · 설명 위험 유형명 → 답변 첫 문장 제목
예: "고객님은 암투병생활비 부족형으로 진단됩니다."
설명 → 진단 이유 본문		 없음 — 카드 연결 구조 리드 스코어 — Phase 1.5+ 전용
③ Rule Clark 경고 메시지 경고 메시지 → 핵심 안내 문구로 출력
예: "암진단금이 5,000만원 미만입니다. 보완을 검토해보세요."		 마이데이터 담보 금액

④ Evidence 활용 방법 · 데이터 출처 활용 방법 → 근거 설명 섹션
