KC Admin 목업

01 용어 정의

KC Admin 시스템에서 사용하는 핵심 용어의 공식 정의

시스템 핵심
KC카드 Knowledge Card
Clark AI가 보험 상담 시 사용하는 판단 기준 단위. 6종(Risk-type·Evidence·Concept·Rule·Policy·Playbook)이 있으며 각각 역할이 다르다. 운영자가 KC Admin에서 등록·관리한다.
Clark AI
보닥의 AI 보험 상담 엔진. KC 카드 체인을 참조하여 사용자 질문에 맞춤형 답변을 생성한다. KC Admin에서 등록한 카드가 승인·연결·라이브 전환을 거쳐야 Clark AI에 반영된다.
KC Admin
KC 카드를 등록·검수·연결·배포하는 내부 운영 시스템. 이 정책서의 구현 대상. 운영자·검수자·관리자가 사용한다.
AI 답변 체인
Answer Chain
Clark AI가 답변을 생성할 때 참조하는 카드 연결 경로. 체인 순서: Concept → Risk-type → Rule → Evidence / Policy. Playbook은 이 체인과 독립적으로 병렬 동작한다.
카드 캔버스
Card Canvas
승인완료된 카드들을 시각적으로 연결하는 KC Admin의 전용 화면(00_canvas-main.html). 카드 간 연결은 반드시 이 화면에서만 설정·변경한다. 편집기에서는 연결 상태를 읽기 전용으로만 확인할 수 있다.
연결·엣지
엣지 Edge
카드와 카드 사이의 연결선. 두 종류가 있다: 준비 연결(pending)과 라이브 연결(active).
준비 연결 Pending Edge
approved(승인완료) 카드가 포함된 연결. 캔버스에 점선 파란색(#1A4A9A, stroke-dasharray:6,4)으로 표시. Clark AI에 미반영 상태이며, 모든 연결 카드가 라이브 전환될 때 active 엣지로 전환된다.
라이브 연결 Active Edge
연결된 모든 카드가 active(라이브) 상태인 연결. 실선 초록색(#0F6E56)으로 표시. Clark AI에 즉시 반영된다.
CONNECT_RULES
카드 유형 간 연결 방향을 정의하는 규칙. concept→risk, risk→rule, rule→evidence·policy. 이 방향 외의 연결은 불가하다. 자세한 내용은 03 연결 정책 참조.
카드 유형 6종
Risk-type 리스크 유형
사용자의 보험 보장 공백 유형을 분류하는 카드. Clark AI 답변 체인의 핵심 분류 기준. 코드 형식: T01~T23 (T06·T15·T16 미사용, 총 20개). 공개범위: 고객 공개.
Evidence 근거 자료
공인 외부 통계를 직접 입력하는 카드. 출처: HIRA·NHIS·통계청·FSS. Rule 카드가 사용하는 근거 데이터. 공개범위: 내부 전용.
Concept 개념
AI 답변 체인의 진입점. 사용자 질의 키워드와 동의어 매칭을 담당. Risk-type 연결이 필수이며, 연결 없으면 KC 체인 미진입(Case 3 또는 Case 4로 분기). 하나의 Concept은 여러 Risk-type과 연결 가능(1:N). Standalone 기능 없음. 공개범위: 고객 공개.
Rule 판단 규칙
고객의 보험 보장 공백 여부를 판단하는 조건 집합. 마이데이터·프롬에이지·사용자 프로파일 3개 데이터 소스 기반으로 조건 빌더에서 설정. 공개범위: 공통 기준.
Policy 정책
Clark 앱에 표시할 면책 고지 문구를 운영자가 직접 작성하는 카드. 구성: Policy 이름 + Clark 앱 표시 문구 2개 필드만으로 단순 구성. 준법감시인 포함 2단계 승인 필요. 공개범위: 내부 전용.
Playbook 플레이북
발화 키워드 감지 → CTA(Call-to-Action) 전환을 담당하는 카드. 메인 AI 체인(Concept→Risk→Rule→…)과 독립적으로 병렬 동작. 공개범위: 내부 전용 고정.
데이터 소스
마이데이터 MyData
고객의 보험 가입 현황 데이터. Rule 조건 빌더에서 담보코드 + 담보금액(coverage_amt) 조건만 허용. 상담 시 Clark AI에 자동 전달. Evidence 카드 등록 불필요.
화면 표기: "마이데이터" (MYDATA 영문 표기 금지)
프롬에이지 Promage
개인 건강 위험도 분석 데이터. 암위험도(23개) + 질병위험도(27개) = 50개 항목 (flat 단일 select). 생체나이·의료비예측 항목은 Rule 조건 빌더에서 사용 불가. 상담 시 자동 전달. Clark 앱 출력 시 위험/주의/양호 3단계 등급명만 표시(수치·percentile 노출 금지).
사용자 프로파일 User Profile
나이·성별·결혼여부·자녀유무·운전여부 5개 필드. 나이는 BETWEEN(시작세~종료세 범위), 나머지 4개는 EQ 조건. Rule 조건 빌더에서 직접 참조. 직업위험도·출산예정·거주지는 삭제됨. drives(운전여부 — 무관/유/무) 항목 추가됨.
역할·승인
운영자
KC 카드를 등록·수정·검수 요청하는 실무자. 카드를 직접 승인할 수 없음.
검수자
운영자가 요청한 카드를 검토·승인·반려하는 역할. Policy 카드 외 1단계 검수 담당. FAQ RAG Q&A(Clark 서비스 전용) 검토 후 인덱스 등록(승인) 권한도 보유.
준법감시인 Compliance Officer
Policy 카드의 2단계 최종 승인자. 도메인 검수자 승인 후 준법감시인이 추가 승인해야 Policy가 active 전환 가능.
AI 답변 케이스 Case 1~4는 마이데이터 연동 상태(③④) 전용
Case 1 — KC 기반 정밀 답변
KC Concept 매칭O + Playbook 미감지. KC 카드 체인(Concept→Risk-type→Rule→Evidence/Policy)을 완전 통과하여 Clark AI가 맞춤형 구조화 답변 생성.
Case 2 — KC 기반 + CTA
KC Concept 매칭O + Playbook 감지O. KC 구조화 답변에 CTA 버튼이 추가로 출력된다.
Case 3 — RAG 우선 시도 → Fallback 생성형
Concept 미매칭 + Playbook 미감지. 먼저 약관 RAG → Clark 서비스 FAQ RAG 순서로 검색을 시도한다. RAG 결과가 있으면(유사도 ≥ 기준값) RAG 기반 답변 출력. RAG 결과도 없을 때에만 시스템 설정(18_system-settings.html)의 LLM 가이드라인이 적용된 제한 생성형 답변 출력.
Case 4 — Playbook Standalone
KC 미매칭 + Playbook 감지O. Playbook의 Standalone 답변 가이드 기반으로 답변 + CTA 버튼 출력. Standalone은 Playbook이 전담하며 Concept 카드에 Standalone 기능 없음.
워크플로우·시스템
wf-tracker
카드 편집기 상단에 표시되는 4단계 진행 표시바. ① 내용 입력 → ② 검수 요청 → ③ 승인완료 → ④ 라이브. 캔버스 연결·사전 테스트·라이브 전환은 00_canvas-main.html에서 처리.
캔버스 연결·사전 테스트는 00_canvas-main.html에서만 수행 (사이드바에 별도 메뉴 없음)
AIO AI Optimization
사용자 질의에 AI가 KC 카드를 참조할 수 있도록 최적화하는 구조적 마크업 및 llms.txt 관리 방식. 고객 공개 카드(Concept·Risk-type)만 AIO 색인 대상. Evidence·Rule·Policy·Playbook은 AIO 색인 제외.
카디널리티 Cardinality
카드 간 연결의 개수 관계. Concept:Risk-type=N:N (양방향 다대다), Risk-type:Rule=1:1 (전속 — Risk-type 1개에 Rule 1개, 공유 불가), Rule:Evidence=N:N, Rule:Policy=N:N. 자세한 카드 유형별 카디널리티는 03 연결 정책 참조.
상태·기타
등급명
Clark AI 출력 시 사용하는 3단계 표현: 위험 / 주의 / 양호. 상/중/하·고/중/저 표현은 사용 금지.
공개범위
카드가 노출되는 대상 범위. 카드 유형에 따라 자동 고정되며 운영자가 변경 불가. 고객 공개(Concept·Risk-type), 공통 기준(Rule), 내부 전용(Evidence·Policy·Playbook).
사전 테스트 Dry-run
실제 고객 프로필 데이터로 Rule 조건을 테스트하여 AI 답변 경로를 미리 확인하는 기능. 라이브 전환 전 필수 절차.
화면 표기: "사전 테스트" ("시뮬레이션" 사용 금지)
라이브 전환
사전 테스트 통과 후 배포 요청을 통해 카드를 active 상태로 전환하는 행위. pending edge → active edge로 변경. Clark AI에 즉시 반영.
Fallback Policy
KC Concept 카드 미매칭 + RAG 검색 결과도 없을 때 LLM이 제한 없이 생성형 답변을 출력하지 않도록 제어하는 시스템 수준 설정. 18_system-settings.html에서 관리. 카드 등록·승인 흐름 없이 저장 즉시 반영. 허용 범위: 일반 보험 개념 설명·절차 안내. 금지: 구체적 보장 금액 판단, 특정 상품 추천, 청구 가능 여부 판단, 경쟁사 언급.
RAG Retrieval-Augmented Generation
KC 미매칭 시 내부 문서를 벡터 검색해 답변 근거를 보강하는 레이어. KC 체인보다 낮은 우선순위로 동작하며, 유사도 기준값 이상 결과가 있을 때만 RAG 기반 답변을 출력한다. 소스는 약관 RAG(가입 상품 약관 자동 파싱)와 FAQ RAG(Clark 서비스 전용 Q&A) 2종. 모두 보닥 내부 데이터이며 외부 공개 정보를 소스로 사용하지 않는다.
약관 RAG
보험 약관을 자동 파싱하여 인덱스를 구성하는 RAG 소스. 마이데이터 연동 상태(③④)에서만 가능 — 미연동 시 건너뜀. 크롤러 감지 → 재파싱 → 인덱스 갱신 파이프라인으로 운영자 개입 없이 자동화. 설정은 관리자가 시스템 설정에서 관리.
FAQ RAG
Clark 서비스 전용 Q&A(앱 사용 안내, 보닥 플래너 연결, 서비스 정책 등)를 인덱스로 사용하는 RAG 소스. 약관 RAG가 처리하지 못하는 서비스 고유 질의를 커버한다. 약관·보장 관련 질의는 약관 RAG가 담당하므로 FAQ RAG에 등록하지 않는다. 운영자가 Q&A를 직접 등록 → 검수자 승인 → 인덱스 등록. 등록 권한: 운영자. 인덱스 등록(승인) 권한: 검수자.
유사도 기준값 Similarity Threshold
RAG 검색 결과를 답변에 사용할지 결정하는 기준값. 18_system-settings.html에서 관리자가 설정. 기본값 권장: 0.75 이상 = RAG 기반 답변 / 0.5~0.75 = "상담사 연결" 유도 / 0.5 미만 = Fallback 생성형.
