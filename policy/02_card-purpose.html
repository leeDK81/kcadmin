답변 케이스 결정 — KC 매칭 × Playbook 감지 마이데이터 연동 상태(③④) 전용

사용자 발화는 KC 메인 체인과 Playbook 체인이 동시에 독립 분석됩니다. 두 결과의 조합에 따라 4가지 답변 케이스 중 하나가 결정됩니다. Case 1~4는 마이데이터 연동(상태 ③④)에서만 적용되며, 미연동 상태(①②)는 별도 분기 규칙을 따릅니다.

케이스 KC Concept 매칭 Playbook 키워드 감지 Clark 출력
Case 1 ✓ 매칭 — 미감지 KC 구조화 답변 (Risk-type 유형명·설명 + Rule 경고 메시지 + Evidence 인용 + Policy 문구) — RAG·LLM 호출 없음
Case 2 ✓ 매칭 ✓ 감지 KC 구조화 답변 + CTA 버튼 — Evidence·Policy는 KC 답변에 포함. CTA만 Playbook이 가산 추가
Case 3 — 미매칭 — 미감지 RAG 우선 시도 (약관 RAG → Clark 서비스 FAQ RAG 순) → RAG 결과 있으면 RAG 기반 답변 / RAG 결과도 없으면 Fallback 제한 생성형 답변 (경쟁사·외부 서비스 언급 금지 등 제한 규칙 적용)
Case 4 — 미매칭 ✓ 감지 LLM 생성형 답변 (Playbook Standalone 가이드 컨텍스트 주입) + CTA 버튼
KC 기반 답변 불가 조건 (Fallback 진입) — 아래 3가지 중 하나라도 해당 시:
① Concept 동의어 매칭 없음  |  ② 연결된 Rule이 라이브 상태가 아님  |  ③ 라이브 Rule이 있으나 Rule 조건 미충족
KC 미매칭 시 RAG 레이어(약관 RAG → Clark 서비스 FAQ RAG) 우선 시도 → RAG 결과 없을 때 최종적으로 Fallback 생성형 발동.
RAG 판단 기준·Fallback 제한 규칙 설정 → 시스템 설정(18_system-settings.html)
주요 선택 규칙 (개발자 참조):
· Concept 매칭: 완전 일치 > 포함 일치. 복수 매칭 시 타이브레이킹 — ①매칭 방식 → ②synonyms 일치 수 → ③Risk-type 연결 여부 → ④최근 라이브 전환일. 최종 Concept 1개만 선택.
· Rule 평가: Risk-type에 연결된 Rule(1개, 1:1 전속) 조건 평가 — 필수 조건은 AND(모두 충족해야 Rule 발동), 선택 조건은 미충족 시에도 Rule 발동(충족 개수는 Risk-type 노출 우선순위 2순위에 활용).
· Policy 복수 적용: 연결된 모든 라이브 Policy 동시 적용(AND). 충돌 시 더 제한적인 Policy 우선. 전체 출력 차단 시 Clark 기본 안내 문구로 대체, 운영팀 알림 발생.

Risk-type — 리스크 유형

역할: 사용자의 보험 보장 공백 유형을 분류하는 카드. Clark AI 답변 체인의 핵심 분류 기준이며, 사용자가 어떤 보장 공백 상황에 있는지를 정의한다.

체인 위치: Concept → Risk-type → Rule. Concept이 질문의 주제를 잡고, Risk-type이 해당 주제에서 어떤 위험 유형인지를 분류한다.

다중 감지: 하나의 사용자 진단에서 복수의 Risk-type이 동시에 감지될 수 있다. 노출 우선순위는 ①중요도(높음>보통>낮음) → ②선택 조건 충족 개수(시스템 자동) → ③최근 배포순으로 결정된다.

공개범위: 고객 공개 코드 형식: T01 ~ T10 현행 운영 코드: T01~T09 active, T10 paused 중요도: 높음 / 보통(기본값) / 낮음

Evidence — 근거 자료

역할: Rule 카드가 사용하는 공인 외부 통계를 등록하는 카드. Clark AI가 직접 활용할 수 없는 공인 기관 통계 수치를 운영자가 직접 입력한다.

단일 유형: 공인 외부 통계 기반(external) 단일 유형. 마이데이터·프롬에이지는 상담 시 자동 전달되므로 Evidence 카드 불필요.

허용 출처: HIRA(건강보험심사평가원), NHIS(국민건강보험공단), 통계청(KOSTAT), FSS(금융감독원), 기타 공인 기관.

공개범위: 내부 전용 출처 유형: 공인 외부 통계만 허용

Concept — 개념

역할: AI 답변 체인의 진입점. 사용자 질의 키워드와 동의어 매칭을 담당. Concept이 매칭되어야 KC 기반 정밀 답변(Case 1/2)이 시작된다.

체인 위치: Concept → Risk-type. 하나의 Concept은 복수의 Risk-type과 연결 가능(1:N 관계). Risk-type 연결은 필수(chip-req) — 미연결 시 KC 체인에 진입하지 못한다.

구성 요소: 개념명, 설명, 동의어 목록(사용자 질의 키워드 집합). 동의어가 많을수록 매칭 정확도가 높아진다.

공개범위: 고객 공개 Risk-type 연결: 1:N (다중 선택, 캔버스에서 설정)

Rule — 판단 규칙

역할: 고객의 보험 보장 공백 여부를 실제로 판단하는 조건 집합. 마이데이터·프롬에이지·사용자 프로파일 3개 데이터 소스를 조건 빌더로 설정.

조건 빌더 데이터 소스 3종:
· 마이데이터(MyData): 담보코드 select + 담보금액(coverage_amt) + 조건(LTE/GTE) + 기준값. 존재여부·갱신여부·보험유형·날짜 조건 불가.
· 프롬에이지(Promage): 위험도 항목 select(50개 flat, 암위험도 23개 + 질병위험도 27개 구성) + 같음(EQ 고정) + 등급(위험/주의/양호). 생체나이·의료비예측 사용 불가.
· 사용자 프로파일: 나이(BETWEEN, 시작세~종료세) / 성별(무관/남/여) / 결혼여부(무관/기혼/미혼) / 자녀유무(무관/있음/없음) / 운전여부(무관/유/무) 5개 항목. 직업위험도·출산예정·거주지 삭제됨.

약관 DB: 약관 DB 확보 시 Rule 조건 충족 → 담보코드 기준으로 자동 조회 → Clark 답변에 면책조항 섹션 자동 포함. (별도 플래그 없이 엔진이 자동 처리)

공개범위: 공통 기준 출력 등급: 위험 / 주의 / 양호 (상/중/하 금지)

Policy — 정책

역할: Clark 앱에 표시할 면책 고지 문구를 운영자가 직접 작성하는 카드. Clark 앱에 표시되는 면책 고지 문구를 이 카드의 "Clark 앱 표시 문구" 필드로 관리. 앱 코드 하드코딩 금지.

승인 특이사항: 도메인 검수자 승인 후 준법감시인이 추가 최종 승인하는 2단계 승인 필요. Policy 카드는 준법감시인 승인 없이 active 전환 불가.

체인 연결: Rule → Policy (선택사항). 연결 없이도 Policy 자체는 등록·승인 가능.

공개범위: 내부 전용 승인 단계: 2단계 (도메인 검수 + 준법감시인)

Playbook — 플레이북

역할: 사용자 발화에서 특정 키워드 감지 시 CTA(Call-to-Action) 버튼을 추가하거나 Standalone 답변을 제공하는 카드. 메인 AI 체인(Concept→Risk→Rule→…)과 완전히 독립적으로 병렬 동작한다.

동작 방식:
· KC 체인 매칭 + Playbook 발화 키워드 감지 → KC 답변 + CTA 버튼 추가 (Case 2)
· KC 체인 미매칭 + Playbook 발화 키워드 감지 → Standalone 가이드 기반 답변 + CTA (Case 4)

CTA 버튼 유형: 상담 연결(consult) / 앱 내 이동(deeplink) / 외부 링크(external). "상담 연결"이 기본 버튼(필수).

Standalone 답변 가이드: Playbook 편집기 Card ② 내 "Standalone 답변 가이드" 텍스트 필드(선택 — 비워두면 Clark 기본 안내 문구 사용). Case 4(KC 미매칭 + Playbook 키워드 감지) 발생 시 LLM 시스템 프롬프트에 주입되어 답변 방향을 가이드.

라이브 전환 경로: Playbook은 캔버스 연결 없이 approved 상태에서 직접 "라이브 전환" 가능. 사전 테스트 불필요.

병렬 체인 4단계 (P1~P4):
· P1 — 발화 키워드 감지: 사용자 발화에서 Playbook 등록 키워드 부분 매칭. 미감지 시 전환 없음
· P2 — 리드 스코어 산출: 메인 체인 Risk-type 판정 결과 + 상담 유형 가중치 합산 (Phase 1.5 구현 예정)
· P3 — 고객 DB 전달: 리드 스코어 포함 고객 정보를 원수사·GA에 전달 (Phase 1.5)
· P4 — 배정 최적화: 원수사·GA 내부 처리 — KC Admin 관할 외 (Phase 2)
MVP 범위: P1(발화 감지) + P2(스코어 산출) + P3(DB 전달). P4는 원수사·GA API 연동 후 Phase 2 예정.

공개범위: 내부 전용 고정 (변경 불가) 최소 키워드: 3개 체인 독립: CONNECT_RULES 적용 안 함 Standalone 가이드: 선택 — 비워두면 Clark 기본 안내 문구 사용 (Case 4 LLM 주입)
카드별 답변 출력 필드 — 개발자 참조

KC 기반 답변(Case 1·2) 생성 시 각 카드의 어떤 필드 데이터가 사용자 답변에 포함되는지 정의합니다.

카드 유형 필드 포함 여부 사용자 답변 내 활용 방식
① Concept synonyms · name · definition 미포함 질의 매칭 전용. 답변 본문에 직접 노출 안 함 (AIO·내부 교육 전용)
전체 필드 미포함 Concept은 질의 인식 경로 역할만 담당. 답변에 포함되지 않음
② Risk-type name (위험 유형명) 포함 답변 첫 문장 제목. 예: "고객님은 암투병생활비 부족형으로 진단됩니다."
description (설명) 포함 진단 이유 본문 섹션에 참조
③ Rule clarkMessage 포함 핵심 출력 메시지. 예: "암진단금이 5,000만원 미만입니다. 치료비 보완을 검토해보세요."
conditions (판단 조건) 미포함 내부 판단 로직 전용. 필드명·연산자·기준값은 사용자에게 노출하지 않음
④ Evidence usageNote · title 조건부 근거 섹션 있을 때 포함. "이 진단의 근거: ..." 형식. Evidence가 없으면 근거 섹션 생략, clarkMessage는 그대로 출력
원본 수치 (rawData) 금지 프롬에이지 원본 수치 고객 공개 금지 [§3-8]. 반드시 3단계 등급(위험/주의/양호)으로 변환 후 포함
⑤ Policy appDisplayText (Clark 앱 표시 문구) 조건부 이 필드만 사용자 답변에 포함. Policy 자체 내용(조건·배경·원문)은 미노출 (내부 전용 고정)
프롬에이지 민감정보 처리 원칙 [§3-8]: 건강검진 원본 수치·위험값은 보닥 서버(국내)에만 저장, 외부 서버·LLM 전송 금지. 답변 시 반드시 3단계 등급으로 변환. 허용: "건강위험도가 높은 편입니다" / 금지: "암위험도 +19.6%"
