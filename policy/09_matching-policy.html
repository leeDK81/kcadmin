09 감지·매칭 정책

사용자 질의가 들어왔을 때 Concept·Playbook 카드를 찾는 방식과 결과 처리 규칙 (확정: 2026-06-29)

개요 — 두 카드가 하는 일
CONCEPT PLAYBOOK
역할 KC 답변 체인의 진입 트리거 CTA 노출의 트리거
운영자 등록 항목 동의어·예문 (단어·구·문장 혼용 가능) 예상 질의 (사용자가 할 법한 문장)
매칭 결과 Concept 식별 → Risk-type → Rule → … 체인 진입 Playbook 식별 → 해당 CTA 버튼·안내 노출
KC 체인과의 관계 KC 체인 그 자체 KC 체인과 독립 — 결과에 CTA만 추가
핵심 원칙: 두 카드는 동시에 병렬로 검색한다. 같은 질문에 Concept과 Playbook이 모두 매칭되어도 충돌하지 않는다 — KC 답변을 생성하면서 CTA를 함께 노출한다.
매칭 방식 — 임베딩 의미 유사도
키워드 완전 일치 방식을 쓰지 않는다
운영자가 "암"을 등록해도 사용자는 "종양 판정", "악성신생물", "cancer 진단" 같은 다양한 표현을 쓴다. 단어가 정확히 일치해야만 잡는 방식으로는 실제 상담에서 대부분의 질문을 놓친다.
임베딩 벡터 유사도로 매칭한다
운영자가 등록한 텍스트(동의어·예문·예상질의)를 벡터로 변환해 인덱싱한다. 사용자 질의도 동일 방식으로 벡터화한 뒤 가장 의미가 가까운 카드를 찾는다. 표현이 달라도 의미가 같으면 매칭된다.
예: "암"만 등록 → "악성종양", "종양 진단", "cancer" 모두 매칭 가능
예: "해지하고 싶어요" 등록 → "계약 취소할게요", "더 이상 보험 유지 안 할래요" 매칭 가능
Concept과 Playbook 모두 동일한 엔진을 사용한다
Concept의 동의어·예문과 Playbook의 예상질의 모두 단어·문장이 혼용될 수 있다. 별도 엔진을 쓸 이유가 없으므로 동일한 임베딩 모델로 처리한다. 다만 인덱스는 분리하여 관리한다.
인덱싱 대상
CONCEPT 카드 인덱스
인덱싱 필드
✅ 용어명 (대표명) — 필수. 미등록 카드도 대표명으로는 매칭되어야 함
✅ 동의어·예문 전체 — 운영자가 입력한 단어·구·문장 모두
❌ 한 줄 정의 — 매칭 대상 아님 (설명 텍스트)
인덱스 키
concept_id — 매칭 결과로 어느 Concept인지 식별
PLAYBOOK 카드 인덱스
인덱싱 필드
✅ 예상 질의 전체 — 운영자가 입력한 문장 모두
❌ Playbook 명칭 — 매칭 대상 아님 (운영자용 식별명)
❌ CTA 설명 — 매칭 대상 아님 (노출 내용)
인덱스 키
playbook_id — 매칭 결과로 어느 Playbook인지 식별
인덱스 분리: Concept 인덱스와 Playbook 인덱스는 별도로 관리한다. 검색 시 각각 독립적으로 쿼리하고 결과를 병합한다.

결과 처리 규칙
1
Concept 단독 매칭
KC 체인(Risk-type → Rule → Evidence → Policy)을 통해 구조화 답변을 생성한다.
CTA 없음.
2
Playbook 단독 매칭
KC 체인 없이 CTA만 노출한다.
KC 답변이 없으므로 약관 RAG → FAQ RAG → LLM Fallback 순으로 답변을 생성하고 CTA를 추가한다.
3
Concept + Playbook 동시 매칭
KC 체인으로 구조화 답변을 생성하고, 답변 하단에 Playbook CTA를 추가한다 (additive).
두 결과가 충돌하지 않으므로 항상 병렬 노출한다.
예: "암 진단 보장 알려주세요" → KC 답변(암 체인) + "플래너 상담 연결" CTA
4
둘 다 미매칭
약관 RAG → FAQ RAG → LLM Fallback 순으로 진행한다 (answer-logic.md 참조).
CTA 없음.
복수 매칭 처리: 기준값 이상인 결과가 여러 개일 경우 유사도 최고점 1개만 선택한다. 동점이면 카드 ID 오름차순으로 선택한다. Concept·Playbook 각각 동일한 규칙을 적용한다.
